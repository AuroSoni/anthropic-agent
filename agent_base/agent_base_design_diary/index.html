01 Genesis

The Big Picture — Why agent_base?

The anthropic_agent library started as a tightly coupled implementation for building AI agents on Anthropic's Claude API. It worked — streaming, tool execution, state persistence, multimodal support — but it was locked to a single provider.

The refactoring goal: extract the orchestration logic — the agentic loop, tool dispatch, context management, media handling — into a provider-agnostic base that any LLM provider can plug into. The provider-specific bits (wire format, SDK calls, streaming event parsing) move to thin adapter layers.

Key Insight This isn't just "rename the package." We're introducing three entirely new subsystems — media backends, tool sandboxing, and tool result enveloping — that reshape how data flows through the agent. Each one addresses a concrete limitation of the original design.

What Changed, At a Glance

Aspectanthropic_agentagent_base
Provider coupling Anthropic-only, SDK calls scattered throughout Provider-agnostic core + pluggable provider adapters
File handling file_backends/ — storage only, no resolution media_backend/ — combined storage + resolution + streaming
Tool results Flat strings or legacy ToolResult ToolResultEnvelope — dual projection (LLM vs UI)
Tool I/O isolation Tools call os, subprocess directly All I/O through Sandbox interface (swappable isolation)
Agent construction Monolithic constructor Composition of independent, swappable components
02 Philosophy

Design Principles

Six principles govern every architectural decision in agent_base. They are the litmus tests for "does this belong here?"

01
Separate Identity from Location from Representation
A media file's media_id is stable. Where its bytes live is a backend concern. How those bytes are projected (base64, URL, reference) depends on the consumer. Three concerns, three layers, zero coupling.
02
Tools Never Touch the OS Directly
All file I/O and command execution flows through a Sandbox interface. Swap LocalSandbox for DockerSandbox by changing config, not code. The litmus test: grep -r "import os" common_tools/ returns zero.
03
The Tool Owns Its Result Shape
Each tool defines a ToolResultEnvelope subclass with two projections: one for the LLM context (minimal), one for the conversation log (rich). The agent loop is polymorphic — no if/elif chains.
04
Provider-Agnostic Core, Provider-Specific Edges
ContentBlock, Message, AgentConfig live in core/. No provider SDK imports. Translation to wire format lives in providers/<name>/.
05
Composition Over Inheritance
The agent is assembled from independently swappable components: Provider, Sandbox, MediaBackend, Compactor, MemoryStore, storage adapters. All wired at construction time.
06
Lazy Resolution at the Boundary
Content blocks store abstract references (media_id, FILE_ID) internally. Resolution to base64/URL/bytes happens at the consumer boundary — never at creation time.

04 Foundation

Core Domain Model

Everything in core/ is provider-agnostic. No provider SDK imports. No provider-specific logic. This is the canonical vocabulary the entire framework speaks.

The ContentBlock Hierarchy

The polymorphic content model follows a "best of all" philosophy: neither the LCD of all LLMs nor the superset. It represents the richest semantically meaningful content types the framework understands, with kwargs as the escape hatch for provider extensions.

Serialization Contract

core/types.py — ContentBlock base Contract 
@dataclass
class ContentBlock(ABC):
    """Base for all content types. raw is transient, kwargs are persisted."""
    content_block_type: ContentBlockType = field(init=False)
    kwargs: Dict[str, Any] = field(default_factory=dict)
    raw: Any = field(default=None, repr=False)  # ← never serialized

    @abstractmethod
    def to_dict(self) -> Dict[str, Any]:
        """JSON-native primitives only. Enums emit .value strings."""
        ...

    @classmethod
    def from_dict(cls, data: Dict) -> "ContentBlock":
        """Match-based polymorphic dispatcher."""
        block_type = data.get("content_block_type")
        match block_type:
            case ContentBlockType.TEXT.value:
                return _build_dataclass_block(TextContent, data)
            # ... other cases ...
Design Decision The data field on media blocks is transient — never serialized. Actual bytes/base64 are resolved on demand via media_id from the media backend. This keeps persisted messages lightweight and makes storage backend swaps trivial.

Message & Usage

Message is a single turn in a conversation — one or more ContentBlocks, a role, optional usage metrics, and provider/model identity. The MessageFormatter ABC translates between canonical messages and provider wire format.

core/messages.py Dataclass 
@dataclass
class Message:
    id: str = field(default_factory=lambda: str(uuid.uuid4()))
    role: Role = Role.USER
    content: List[ContentBlock] = field(default_factory=list)
    stop_reason: Optional[str] = None
    usage: Optional[Usage] = None
    provider: str = ""
    model: str = ""

class MessageFormatter(ABC):
    @abstractmethod
    def format_messages(self, messages, params) -> Dict: ...
    @abstractmethod
    def parse_response(self, raw_response) -> Message: ...

Agent & Provider ABCs

The Agent ABC defines the external contract: run() and resume_run_with_tool_results(). All orchestration logic lives in concrete subclasses under providers/.

The Provider ABC abstracts the raw LLM API call — authentication, retry, streaming. The Agent calls self.provider.generate(); it never imports a provider SDK directly.

core/agent.py + core/provider.py ABC 
class Agent(ABC):
    @abstractmethod
    async def run(self, prompt: str | Message) -> AgentResult: ...
    @abstractmethod
    async def resume_run_with_tool_results(self, results) -> AgentResult: ...

class Provider(ABC):
    @abstractmethod
    async def generate(self, system_prompt, messages, tool_schemas,
                       llm_config, model, max_retries, ...) -> Message: ...
    @abstractmethod
    async def generate_stream(self, ..., queue, stream_formatter) -> Message: ...
Separation Agent owns orchestration (step counting, tool dispatch, compaction). Provider owns transport (HTTP, retry, streaming, response parsing). MessageFormatter owns translation (canonical ↔ wire format). Three concerns, three classes.
05 Tools

The Tool System — Envelopes & Registry

The tool system splits cleanly into two modules: tools/ is infrastructure (registry, decorator, envelope ABC) and common_tools/ is application code (concrete tool implementations).

The ToolResultEnvelope Pattern

The central insight: tool result shapes vary by tool type (code exec returns stdout+files, sub-agent returns a nested conversation, web search returns citations). This is a fundamentally different axis of variation from content blocks, which vary by LLM provider.

The ToolResultEnvelope isolates the tool-type axis. Its two projection methods isolate the consumer-type axis.

tools/types.py ABC 
@dataclass
class ToolResultEnvelope(ABC):
    tool_name: str = ""
    tool_id: str = ""
    is_error: bool = False
    error_message: str | None = None
    duration_ms: float | None = None

    @abstractmethod
    def for_context_window(self) -> list[ContentBlock]:
        """Minimal projection for the LLM's next turn."""
        ...

    @abstractmethod
    def for_conversation_log(self) -> dict[str, Any]:
        """Rich projection for UI display / conversation history."""
        ...

How It Wires Into the Loop

The agent loop is polymorphic over envelope types — no if/elif chains. Adding a new tool means defining one envelope subclass. Zero changes outside the tool's file.

Agent loop — tool handling Loop 
# In the agent's tool handling path:
envelope: ToolResultEnvelope = await self.tool_registry.execute(tool_name, tool_input)

# Projection 1 → LLM context window
context_blocks = envelope.for_context_window()
result_block = ToolResultContent(
    tool_id=envelope.tool_id,
    tool_name=envelope.tool_name,
    tool_result=context_blocks,
    is_error=envelope.is_error,
)
self.agent_config.messages.append(Message.user([result_block]))

# Projection 2 → Conversation log / UI
log_entry = envelope.for_conversation_log()
self.conversation.append_tool_result(log_entry)
Auto-Wrapping Tools that return plain str or legacy ToolResult objects are automatically wrapped in GenericTextEnvelope / LegacyMediaEnvelope. No existing tool breaks during migration.

ToolRegistry

The registry handles registration, dispatch, schema export, and sandbox injection:

  • register() / register_tools() — single or batch registration
  • execute() → ToolResultEnvelope — async dispatch with auto-wrapping
  • execute_tools() — bounded parallel via asyncio.Semaphore
  • get_schemas(format="anthropic"|"openai") — provider-specific schema export
  • attach_sandbox(sandbox) — inject sandbox into all registered tools
  • check_for_relay() — identify frontend/relay tool calls
06 Isolation

Sandbox Isolation

The Sandbox abstracts all tool interactions with "the outside world" into four verbs: read a file, write a file, list a directory, run a command.

sandbox/types.py ABC 
class Sandbox(ABC):
    # Lifecycle — async context manager
    async def setup(self) -> None: ...
    async def teardown(self) -> None: ...

    # Filesystem (all paths are relative)
    async def read_file(self, path: str) -> str: ...
    async def write_file(self, path: str, content: str) -> None: ...
    async def read_file_bytes(self, path: str) -> bytes: ...
    async def list_dir(self, path: str = ".") -> list[str]: ...
    async def file_exists(self, path: str) -> bool: ...
    async def delete(self, path: str) -> None: ...

    # Execution
    async def exec(self, command: str, timeout, cwd, env) -> ExecResult: ...
    async def exec_stream(self, command: str, ...) -> AsyncIterator[str]: ...

Three Isolation Levels

LevelImplementationSetupFilesystemExecutionBoundary
1 LocalSandbox ~1ms Path-restricted host dir subprocess Path containment
2 DockerSandbox 2–5s Bind-mounted host dir docker exec Process + FS isolation
3 E2BSandbox 5–15s Remote VM filesystem Remote execution Full VM isolation
Bind-Mount Asymmetry (Level 2) In DockerSandbox, file operations go through a bind-mount (local disk speed), but command execution goes through docker exec (isolated). Best of both worlds: fast file I/O and isolated execution.

The SandboxFileBridge

The bridge connects two worlds — working files (transient, inside the sandbox) and persisted files (durable, in the media backend):

sandbox/bridge.py Bridge 
class SandboxFileBridge:
    def __init__(self, media_backend: MediaBackend, cache: LocalFileCache | None): ...

    async def materialize_into_sandbox(self, file_id, sandbox, sandbox_path):
        """Download from media backend → write into sandbox workspace."""

    async def persist_from_sandbox(self, sandbox, sandbox_path, agent_uuid) -> MediaMetadata:
        """Read from sandbox workspace → store in media backend."""
Key Design Decision Relative paths only. The LLM generates paths like src/main.py. The sandbox resolves them internally. No consumer ever sees the real filesystem location. This is what makes sandbox levels swappable without touching tool code.
07 Media

Media Backend — Storage Meets Resolution

The media backend is the combined storage + resolution system, replacing the old file_backends/ module. It handles storing bytes and projecting media_id references into the format each consumer needs.

media_backend/base.py ABC 
class MediaBackend(ABC):
    # Storage
    async def store(self, content: bytes | AsyncByteStream,
                    filename, mime_type, agent_uuid) -> MediaMetadata: ...
    async def retrieve(self, media_id, agent_uuid) -> bytes | None: ...
    async def retrieve_stream(self, media_id, agent_uuid) -> AsyncIterator[bytes]: ...
    async def delete(self, media_id, agent_uuid) -> bool: ...

    # Resolution (projections for different consumers)
    async def to_base64(self, media_id, agent_uuid) -> dict: ...  # → LLM
    async def to_url(self, media_id, agent_uuid) -> str: ...     # → Frontend
    async def to_reference(self, media_id, agent_uuid) -> dict: ... # → Log

    # Sandbox integration
    def attach_sandbox(self, sandbox: Sandbox) -> None: ...

Lazy Resolution in Content Blocks

Media content blocks store media_id references, not inline data. Resolution happens at the consumer boundary:

  • LLM call: MessageFormatter.format_messages() calls media_backend.to_base64()
  • Frontend SSE: Streaming formatter calls media_backend.to_url()
  • Persistence: to_dict() serializes just the media_id
Why Combined? to_url() lives on the backend because URL shape is tied to storage type: local returns /api/files/{id}, S3 returns a presigned URL, GCS returns a signed URL. Splitting storage and resolution would force them to share internal state.

MediaMetadata

media_backend/types.py Dataclass 
@dataclass
class MediaMetadata:
    media_id: str                    # Stable identity ("media_{uuid4().hex[:16]}")
    media_mime_type: str             # "image/png", "application/pdf"
    media_filename: str              # Original filename
    media_extension: str             # "png", "pdf"
    media_size: int                  # Size in bytes
    storage_type: str               # "local", "s3", "memory"
    storage_location: str            # Physical location (path, S3 key)
    extras: dict[str, Any] = field(default_factory=dict)

LocalFileCache

A read-through + write-back cache between tools and the media backend. Not a storage backend itself — a performance layer. Features double-checked locking to prevent duplicate downloads and LRU eviction for space management.

13 File Lifecycle

File Lifecycle Management — Zones Meet Manifest

Files flow through the agent system in fundamentally different patterns depending on their origin, purpose, and required durability. The sandbox is an ephemeral workspace; the MediaBackend is durable storage. Files cross that boundary in five distinct patterns, each with different materialization, persistence, and cleanup rules.

The Problem: Five Categories, One Sandbox

#CategoryOriginDirectionDurabilitySandbox?
1 User uploads Frontend / API MediaBackend → Sandbox Permanent Materialized on demand
2 Provider server files Provider API External → MediaBackend (ref) Permanent (metadata) Not materialized
3 Exportable artifacts Tools in sandbox Sandbox → MediaBackend Permanent Created in sandbox, cloned out
4 Session workspace Tools in sandbox N/A Session-scoped Full residence
5 Ephemeral output Tools in sandbox None Transient Sandbox-only
Core Tension The sandbox and MediaBackend have different lifetimes. User uploads must outlive the sandbox. Exportable artifacts must be cloned out before teardown. Provider files are never in the sandbox at all. Ephemeral output should never escape the sandbox. And the workspace sits in between — session-scoped but optionally archivable. Who classifies files, who decides what persists, and what happens at lifecycle boundaries?

The Design: Hybrid Zones + Manifest

The solution combines filesystem zone conventions for the common cases with an explicit FileManifest for tracking and override. Zones provide sensible defaults; the manifest provides control and auditability.

The FileLifecycle Enum

file_lifecycle.py Enum 
class FileLifecycle(Enum):
    DURABLE     = "durable"      # Permanent in MediaBackend. Survives sandbox teardown.
    EXPORTABLE  = "exportable"   # Created in sandbox. Cloned to MediaBackend on flush.
    SESSION     = "session"      # Lives in sandbox. Dies with it. Optionally archivable.
    EPHEMERAL   = "ephemeral"    # Sandbox scratch. Never persisted. Never archived.
    REFERENCE   = "reference"    # External provider file. Metadata only, bytes external.

Sandbox Zone Layout

The sandbox filesystem has predefined zone directories. A file's location determines its default lifecycle — no explicit tagging required for the common path.

sandbox_root/ Zone Layout 
sandbox_root/
├── .imported/              # Zone 1 — DURABLE (materialized from MediaBackend)
│   └── {media_id}/         #   Lazily populated on first tool access.
│       └── filename.ext    #   Read-through. Write-back if dirty.
│
├── workspace/              # Zone 2 — SESSION (LLM's working directory, default cwd)
│   └── src/...             #   Tools read/write freely.
│                           #   Archive-or-drop on teardown (configurable).
│
├── .exports/               # Zone 3 — EXPORTABLE (durable output artifacts)
│   └── report.pdf          #   Auto-registered in manifest as EXPORTABLE.
│                           #   Flushed to MediaBackend on sync/teardown.
│
├── .ephemeral/             # Zone 4 — EPHEMERAL (transient scratch space)
│   ├── logs/               #   NEVER persisted. NEVER archived.
│   └── code_output/        #   Destroyed on teardown.
│
└── .refs/                  # Zone 5 — REFERENCE (provider-file manifest)
    └── manifest.json       #   media_id → provider_file_id mapping.
                            #   No bytes materialized.
Design Note .-prefixed zones (.imported/, .exports/, .ephemeral/, .refs/) are system-managed — hidden from the LLM's natural working space. The LLM works in workspace/ (its default cwd) and doesn't reason about plumbing. The only zone tools explicitly write to is .exports/ — a simple convention: "write here if the user should keep it."

Zone-to-Lifecycle Defaults

ZoneDefault LifecycleOverride?Notes
.imported/ DURABLE No Always durable — these came from MediaBackend
workspace/ SESSION Yes → EXPORTABLE Tool or agent can promote via manifest
.exports/ EXPORTABLE No Writing here is an explicit durability signal
.ephemeral/ EPHEMERAL No Never persisted, never archived
.refs/ REFERENCE No Metadata only, bytes managed externally

Conflict resolution: If a file's zone convention and manifest disagree, the manifest wins. The manifest is authoritative; zones provide defaults for files not yet tracked.

The FileManifest

The manifest is the single source of truth for all tracked files. It records each file's lifecycle disposition, current location (sandbox path and/or MediaBackend metadata), origin, and sync state.

file_manifest.py Dataclass 
@dataclass
class FileRecord:
    media_id: str
    lifecycle: FileLifecycle
    sandbox_path: str | None = None
    media_metadata: MediaMetadata | None = None
    provider_ref: str | None = None        # For REFERENCE files
    source: str = ""                       # "user_upload", "tool:{name}", "provider:{id}"
    dirty: bool = False                   # Modified since last sync
    created_at: str = ""
    last_synced_at: str | None = None

class FileManifest:
    records: dict[str, FileRecord]       # Keyed by media_id

    def by_lifecycle(self, lifecycle) -> list[FileRecord]: ...
    def dirty_files(self) -> list[FileRecord]: ...
    def exportable_files(self) -> list[FileRecord]: ...
    def surfaceable_files(self) -> list[FileRecord]:
        """DURABLE + EXPORTABLE + REFERENCE — files visible to the user."""

    def to_dict(self) -> dict: ...
    @classmethod
    def from_dict(cls, data) -> "FileManifest": ...
Key Insight Only DURABLE, EXPORTABLE, and REFERENCE records are persisted in the manifest. SESSION and EPHEMERAL files are not tracked — they die with the sandbox and don't need recovery. This keeps the manifest small and focused on what matters.

How Each Category Flows

Category 1 — User Uploads (DURABLE)

User upload → API layer
  → media_backend.store(bytes, filename, mime_type)
  → MediaMetadata created, FileRecord(lifecycle=DURABLE) in manifest
  → File NOT yet in sandbox
  ... later, tool needs the file ...
  → SandboxFileBridge.materialize(media_id, ".imported/{media_id}/filename")
  → manifest.record.sandbox_path updated
  → Tool reads/modifies via sandbox.read_file(...)
  → If modified: manifest.record.dirty = True
Inbound flow: MediaBackend → Sandbox (lazy materialization)

Category 2 — Provider Server Files (REFERENCE)

Provider response contains file_id (e.g., Anthropic container file)
  → media_backend.register_reference(provider_file_id, metadata)
  → FileRecord(lifecycle=REFERENCE, provider_ref=file_id) in manifest
  → No sandbox materialization
  → Resolution: to_base64() calls provider API to fetch bytes
  → Resolution: to_url() returns provider-hosted or proxied URL
External reference: never enters the sandbox

Category 3 — Exportable Artifacts (EXPORTABLE)

Tool generates user-facing output
  → sandbox.write_file(".exports/report.csv", data)
  → Zone convention → FileRecord(lifecycle=EXPORTABLE) auto-created
  → At sync point: SandboxFileBridge.persist_from_sandbox(...)
  → MediaMetadata created, stored in MediaBackend
  → manifest record updated with media_metadata + last_synced_at
Outbound flow: Sandbox → MediaBackend

Category 4 — Session Workspace (SESSION)

Tool clones a repo, creates working files
  → sandbox.write_file("workspace/src/main.py", code)
  → Zone convention → SESSION lifecycle (not tracked in manifest)
  → Files live and die with the sandbox
  → If archiving: workspace/ tarred to MediaBackend on teardown
Session-scoped: lives in sandbox, optionally archived

Category 5 — Ephemeral Output (EPHEMERAL)

Code execution writes stdout, bash logs
  → sandbox.write_file(".ephemeral/logs/exec_001.log", output)
  → Zone convention → EPHEMERAL (never tracked in manifest)
  → Destroyed on teardown. No recovery. No archive.
Transient: never escapes the sandbox

Teardown Protocol

on_teardown() Lifecycle 
async def on_teardown(self, agent_uuid, archive_workspace=False):

    # 1. Flush EXPORTABLE files to MediaBackend
    for record in self.manifest.exportable_files():
        if record.sandbox_path and (not record.media_metadata or record.dirty):
            content = await self.sandbox.read_file_bytes(record.sandbox_path)
            meta = await self.media_backend.store(content, ...)
            record.media_metadata = meta
            record.dirty = False

    # 2. Write-back dirty DURABLE files (modified user uploads)
    for record in self.manifest.dirty_files():
        if record.lifecycle == FileLifecycle.DURABLE:
            content = await self.sandbox.read_file_bytes(record.sandbox_path)
            await self.media_backend.update(record.media_id, content, agent_uuid)
            record.dirty = False

    # 3. Optionally archive workspace/ as a tarball
    if archive_workspace:
        archive = await self.sandbox.exec("tar czf - workspace/")
        await self.media_backend.store(archive, "workspace.tar.gz", ...)

    # 4. Persist manifest to AgentConfig
    self.agent_config.file_manifest = self.manifest.to_dict()
    await self.config_adapter.save(self.agent_config)

    # 5. Destroy sandbox filesystem
    await self.sandbox.teardown()

Reload Protocol

on_reload() Lifecycle 
async def on_reload(self, agent_uuid):

    # 1. Create fresh sandbox with zone directories
    await self.sandbox.setup()
    await self.sandbox.exec("mkdir -p .imported .exports .ephemeral .refs workspace")

    # 2. Load manifest from AgentConfig
    self.manifest = FileManifest.from_dict(self.agent_config.file_manifest)

    # 3. Restore workspace from archive if available
    archive = self.manifest.find_archive()
    if archive:
        bytes = await self.media_backend.retrieve(archive.media_id, agent_uuid)
        await self.sandbox.write_file_bytes(".archive.tar.gz", bytes)
        await self.sandbox.exec("tar xzf .archive.tar.gz && rm .archive.tar.gz")

    # 4. Rebuild .refs/manifest.json from REFERENCE records
    refs = {r.media_id: r.provider_ref
            for r in self.manifest.by_lifecycle(FileLifecycle.REFERENCE)}
    await self.sandbox.write_file(".refs/manifest.json", json.dumps(refs))

    # 5. .imported/ stays empty — lazy materialization on first access
    # 6. .exports/ stays empty — already in MediaBackend
    # 7. .ephemeral/ starts empty — transient by definition
Crash Recovery Relying only on teardown for persistence is fragile — a crash loses EXPORTABLE files. The agent loop should sync incrementally after each tool step that produces EXPORTABLE files. Cost is low (only unsynced files in .exports/) and ensures at most one tool step's work is lost.

Manifest Persistence

The FileManifest is serialized into AgentConfig.file_manifest and persisted via the existing AgentConfigAdapter. This means:

  • Survives sandbox teardown — it's in durable storage.
  • Available on reload — the agent knows what to re-materialize.
  • Available across machines — for horizontal scaling.

What the User Sees

LifecycleSurfaced?How
DURABLE Yes User uploaded them; shown in file list
EXPORTABLE Yes Generated artifacts; downloadable
REFERENCE Yes Provider-generated; shown with attribution
SESSION No Internal workspace; not user-facing
EPHEMERAL No Scratch space; never visible

FileManifest.surfaceable_files() returns DURABLE + EXPORTABLE + REFERENCE. This populates generated_files on AgentResult and streams to the frontend via MetaDelta(type="meta_files").

Open Questions

Under Consideration These decisions depend on deployment context and are intentionally left configurable:
  • Sandbox lifetime policy: When should sandboxes be torn down? Options: session end, idle timeout, resource pressure, admin action.
  • Workspace archiving: Should workspace/ be archived on teardown? Trade-off: enables perfect resume but costs storage. Configurable via SandboxConfig.archive_workspace_on_teardown.
  • Reload frequency: Single-server (rare) vs. serverless (every request). The reload protocol is idempotent and fast (lazy materialization).
  • SESSION promotion: Can workspace files be promoted to EXPORTABLE post-creation? Current design: yes, via manifest.set_lifecycle(). Risk: tools forget. Mitigation: agent loop inspects envelopes for generated_files hints.
  • Provider file fallback: What if the provider API is unavailable for REFERENCE resolution? Options: fail fast, cached version, placeholder.
  • Cross-sandbox sharing (subagents): Subagent writes to .exports/, parent reads via media_id through shared MediaBackend.

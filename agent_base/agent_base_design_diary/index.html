The agent_base Framework

Developer Diary · Architecture & Design

A provider-agnostic, async-first foundation for production AI agents — from anthropic_agent to a composable, extensible framework.

Genesis

The Big Picture — Why agent_base?

The anthropic_agent library started as a tightly coupled implementation for building AI agents on Anthropic's Claude API. It worked — streaming, tool execution, state persistence, multimodal support — but it was locked to a single provider.

The refactoring goal: extract the orchestration logic into a provider-agnostic base that any LLM provider can plug into. Three entirely new subsystems — media backends, tool sandboxing, and tool result enveloping — reshape how data flows through the agent.

Aspectanthropic_agentagent_base
Provider coupling Anthropic-only, SDK calls scattered throughout Provider-agnostic core + pluggable provider adapters
File handling file_backends/ — storage only media_backend/ — storage + resolution + sandbox orchestration
Tool results Flat strings or legacy ToolResult ToolResultEnvelope — dual projection (LLM vs UI)
Tool I/O isolation Tools call os, subprocess directly All I/O through Sandbox interface (swappable isolation)
Agent construction Monolithic constructor Composition of independent, swappable components
Philosophy

Design Principles

Six principles govern every architectural decision in agent_base. They are the litmus tests for "does this belong here?"

01
Separate Identity from Location from Representation
A media file's media_id is stable. Where its bytes live is a backend concern. How those bytes are projected depends on the consumer. Three concerns, three layers, zero coupling.
02
Tools Never Touch the OS Directly
All file I/O and command execution flows through a Sandbox interface. Swap LocalSandbox for DockerSandbox by changing config, not code.
03
The Tool Owns Its Result Shape
Each tool defines a ToolResultEnvelope subclass with two projections: one for the LLM context (minimal), one for the conversation log (rich). The agent loop is polymorphic — no if/elif chains.
04
Provider-Agnostic Core, Provider-Specific Edges
ContentBlock, Message, AgentConfig live in core/. No provider SDK imports. Translation to wire format lives in providers/<name>/.
05
Composition Over Inheritance
The agent is assembled from independently swappable components: Provider, Sandbox, MediaBackend, Compactor, MemoryStore, storage adapters. All wired at construction time.
06
Lazy Resolution at the Boundary
Content blocks store abstract references (media_id) internally. Resolution to base64/URL/bytes happens at the consumer boundary — never at creation time.

Extensibility

Axes of Change

Each axis of change is isolated so that modifications along one axis require changes in one place only.

AxisWhat ChangesWhere It LivesAdding One Touches
LLM Provider Wire format, API call, streaming providers/<name>/ 1 directory (4 files)
Tool Type Result shape, projections common_tools/<tool>.py 1 file
Sandbox Level Isolation mechanism sandbox/<level>.py 1 file
Storage Backend Persistence mechanism storage/adapters/<backend>.py 1 file
Media Backend Byte storage + resolution media_backend/<backend>.py 1 file
Compaction Strategy Context trimming approach providers/<name>/compaction.py 1 file per provider
Stream Transport SSE, WebSocket, CLI output streaming/formatters.py 1 file
Content Block Type New canonical content type core/types.py + formatters types.py + each provider's formatter
The Test If adding a new tool requires changes to core/, the architecture has failed. If adding a new provider requires changes to tools/, the architecture has failed. Each axis is its own island — connected by interfaces, never by implementation.

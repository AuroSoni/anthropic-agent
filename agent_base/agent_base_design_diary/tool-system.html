Who Owns the Tool Result?

A Developer Diary — The Tool System from Decorator to Registry to Schema Boundary

The Situation

Two Consumers, One Tool Result, Conflicting Needs

You've built an agentic system with a clean canonical message model. Content blocks are polymorphic — TextContent, ImageContent, ToolUseContent, etc. The system works. But a tension has been growing between two consumers of tool results, and it's about to break your abstractions.

Your agentic loop maintains two parallel states. AgentConfig manages the live LLM context window — it cares about what the model needs to see for its next turn. ConversationHistory is the full record for the UI — it cares about showing the human what happened, step by step, including execution logs, intermediate steps, and rich media.

Developer's inner voice "When a tool returns a result, AgentConfig just wants the final answer: a list of content blocks to stuff back into the context window. But ConversationHistory might want the entire execution trace — logs, intermediate steps, timing data, images with different storage references. These are fundamentally different projections of the same event."

And there's a second, deeper problem. Today, your ToolResultContent is a flat block in the canonical model. But tool results are not like other content blocks. A TextContent varies only across LLM providers (Anthropic says text, OpenAI says content). A tool result varies across tool types — a code execution tool returns stdout + files, a web search tool returns citations + snippets, a sub-agent returns an entire nested conversation. The canonical model can't anticipate all of these.

Fig 1 — Tool results have a fundamentally different axis of variation
🚨 The Design Tension

Your canonical ToolResultBase tries to represent all tool results with tool_result: str | List[Dict]. This works for simple tools but collapses into an untyped bag-of-dicts for rich tools. The type system can't help you, the serialization is lossy, and every consumer has to do ad-hoc parsing. Worse, you can't add tool-specific metadata (execution logs, timing, generated files) without polluting the canonical model with fields that only matter to one tool.

Chapter 1 — Ownership
The Key Insight

"The Tool Knows Best What Its Result Looks Like"

Let's think about who has the knowledge here. The canonical model knows about message structure — roles, content blocks, tool correlation IDs. But it doesn't know (and shouldn't know) that a code execution tool produces stdout, stderr, exit codes, and generated files. That knowledge lives with the tool itself.

Developer's inner voice "This is a classic 'information expert' problem from GRASP principles. The responsibility for defining a tool's result shape should go to the class that has the most information about it — the tool itself. My ConfigurableToolBase already owns the tool's schema and execution. Why not also let it own the result definition?"

But there's a constraint: AgentConfig needs a standardized projection of any tool result (content blocks for the LLM context window), while ConversationHistory needs a rich projection (everything the UI might want to display). The tool can produce the rich data, but it needs to know how to project it into both shapes.

This is where the pattern crystallizes. We need a Rich Result object owned by the tool, with two projection methods: one for the LLM context window, one for the conversation log.

Chapter 2 — The Envelope
Designing the Contract

ToolResultEnvelope — The Tool's Rich Return Type

Instead of having tools return a flat string or a ToolResult that only knows about text and images, we introduce a ToolResultEnvelope. This is an abstract base that every tool-specific result type extends. The key insight: the envelope is not a content block — it's a pre-projection object that knows how to produce content blocks for different consumers.

tool_result_envelope.py Abstract Contract 
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import Any

# Import the canonical content blocks from your model
from .core import ContentBlock, TextContent, ImageContent, ErrorContent


@dataclass
class ToolResultEnvelope(ABC):
    """Base class for tool-specific rich result objects.
    
    Every tool defines its own subclass with whatever fields it needs.
    The two abstract methods project this rich data into the shapes
    that each consumer (AgentConfig vs ConversationHistory) requires.
    
    This is NOT a ContentBlock. It's a pre-projection object that
    PRODUCES ContentBlocks. The tool owns the data; the projections
    adapt it to each consumer's needs.
    """

    # --- Shared metadata every result has ---
    tool_name: str = ""
    tool_id: str = ""
    is_error: bool = False
    error_message: str | None = None
    duration_ms: float | None = None

    # ─── Projection 1: For the LLM context window ───

    @abstractmethod
    def for_context_window(self) -> list[ContentBlock]:
        """Project this result into ContentBlocks for the LLM.
        
        This is what gets stuffed into the ToolResultContent block
        inside AgentConfig.messages. The LLM will see these blocks
        on its next turn.
        
        Rules:
          - Must return canonical ContentBlock instances
          - Should be MINIMAL — only what the LLM needs to continue
          - Images/docs should use the provider-appropriate source type
          - Errors should return [ErrorContent(...)] or [TextContent("Error: ...")]
        """
        ...

    # ─── Projection 2: For the conversation log (UI) ───

    @abstractmethod
    def for_conversation_log(self) -> dict[str, Any]:
        """Project this result into a rich dict for ConversationHistory.
        
        This is what gets stored in the conversation log for UI display.
        It can include anything the frontend might want to render:
        execution logs, timing, intermediate steps, file references,
        nested sub-agent conversations, etc.
        
        The dict should always include at minimum:
          {
            "tool_name": str,
            "tool_id": str,
            "is_error": bool,
            "summary": str,      # human-readable one-liner
            "content_blocks": [], # the same blocks as for_context_window
            ...tool-specific fields...
          }
        """
        ...

    # ─── Convenience: error envelope factory ───

    @classmethod
    def error(cls, tool_name: str, tool_id: str, message: str) -> "ToolResultEnvelope":
        """Create a generic error envelope (works for any tool)."""
        return GenericErrorEnvelope(
            tool_name=tool_name, tool_id=tool_id,
            is_error=True, error_message=message,
        )


# --- A simple fallback for tools that don't define their own envelope ---

@dataclass
class GenericErrorEnvelope(ToolResultEnvelope):
    """Fallback envelope for errors or legacy tools."""

    def for_context_window(self) -> list[ContentBlock]:
        return [TextContent(text=f"Error: {self.error_message}")]

    def for_conversation_log(self) -> dict[str, Any]:
        return {
            "tool_name": self.tool_name,
            "tool_id": self.tool_id,
            "is_error": True,
            "summary": self.error_message,
            "content_blocks": [TextContent(text=self.error_message or "").to_dict()],
        }
Developer's inner voice "Notice what I did NOT do: I didn't make this a ContentBlock subclass. That would pollute the canonical model with tool-specific data. The envelope is a separate layer — it sits between tool execution and the canonical message chain. It's the tool's 'rich return type' that gets projected down into the simpler shapes that each consumer needs."
🧩 Pattern Recognition

This is a Projection pattern (sometimes called "View Model" or "DTO projection"). The same source data gets transformed into different shapes for different consumers. The key is that the source (the tool) owns the projection logic, because only the tool knows what's meaningful in each context. AgentConfig gets a lean projection; ConversationHistory gets a rich one.

Chapter 3 — Concrete Envelopes
Building Real Envelopes

Each Tool Defines Its Own Rich Result

Now let's see how three very different tools — a code executor, a web search tool, and a sub-agent — each define their own envelope with tool-specific fields, while conforming to the same dual-projection contract.

Example A: Code Execution Tool

common_tools/code_execution_tool.py Tool-Owned 
@dataclass
class CodeExecutionResult(ToolResultEnvelope):
    """Rich result from a code execution tool."""

    # ── Tool-specific fields (the canonical model knows nothing about these) ──
    stdout: str = ""
    stderr: str = ""
    exit_code: int = 0
    generated_files: list[dict] = field(default_factory=list)  # [{path, mime_type, size}]
    generated_images: list[dict] = field(default_factory=list)  # [{path, mime_type, base64}]
    execution_log: list[str] = field(default_factory=list)     # line-by-line execution trace

    # ── Projection 1: What the LLM sees ──
    def for_context_window(self) -> list[ContentBlock]:
        blocks: list[ContentBlock] = []

        # LLM gets the stdout (the "answer")
        if self.stdout:
            blocks.append(TextContent(text=self.stdout))

        # LLM gets generated images inline (so it can reason about them)
        for img in self.generated_images:
            blocks.append(ImageContent(
                source_type=SourceType.BASE64,
                media_type=img["mime_type"],
                data=img["base64"],
            ))

        # LLM gets a FILE MANIFEST (not the files themselves)
        if self.generated_files:
            manifest = "Generated files:\n" + "\n".join(
                f"  - {f['path']} ({f['mime_type']}, {f['size']} bytes)"
                for f in self.generated_files
            )
            blocks.append(TextContent(text=manifest))

        # stderr only if error
        if self.is_error and self.stderr:
            blocks.append(TextContent(text=f"STDERR:\n{self.stderr}"))

        return blocks or [TextContent(text="(no output)")]

    # ── Projection 2: What the UI sees ──
    def for_conversation_log(self) -> dict[str, Any]:
        return {
            "tool_name": self.tool_name,
            "tool_id": self.tool_id,
            "is_error": self.is_error,
            "summary": self.stdout[:200] if self.stdout else "(no output)",
            "duration_ms": self.duration_ms,
            # ── UI-specific rich data ──
            "stdout": self.stdout,
            "stderr": self.stderr,
            "exit_code": self.exit_code,
            "execution_log": self.execution_log,
            "generated_files": self.generated_files,
            "generated_images": [
                {"path": img["path"], "mime_type": img["mime_type"]}
                for img in self.generated_images
            ],  # Note: no base64 in the log — just references
            "content_blocks": [b.to_dict() for b in self.for_context_window()],
        }

Notice the asymmetry in the two projections. The LLM gets the stdout and inline images (it needs to reason about them). The conversation log gets stdout, stderr, exit code, execution trace, and file references — but without base64 image blobs (the UI will load those from URLs). Each consumer gets exactly what it needs, no more.

Example B: Sub-Agent Tool

Sub-agents are the most extreme case. A sub-agent's result is an entire nested conversation. The LLM only needs to see the final answer. The UI might want to display the full nested conversation as a collapsible tree.

common_tools/sub_agent_tool.py Tool-Owned 
@dataclass
class SubAgentResult(ToolResultEnvelope):
    """Rich result from a sub-agent execution."""

    # ── Sub-agent specific data ──
    final_answer: str = ""
    nested_messages: list[dict] = field(default_factory=list)  # Full sub-conversation
    total_steps: int = 0
    sub_agent_model: str = ""
    sub_agent_usage: dict = field(default_factory=dict)

    def for_context_window(self) -> list[ContentBlock]:
        # The parent LLM ONLY sees the final answer — not the entire sub-conversation
        return [TextContent(text=self.final_answer)]

    def for_conversation_log(self) -> dict[str, Any]:
        return {
            "tool_name": self.tool_name,
            "tool_id": self.tool_id,
            "is_error": self.is_error,
            "summary": self.final_answer[:200],
            "duration_ms": self.duration_ms,
            # ── The UI gets the entire nested conversation tree ──
            "final_answer": self.final_answer,
            "nested_messages": self.nested_messages,
            "total_steps": self.total_steps,
            "sub_agent_model": self.sub_agent_model,
            "sub_agent_usage": self.sub_agent_usage,
            "content_blocks": [b.to_dict() for b in self.for_context_window()],
        }
✅ What We've Achieved

Each tool now owns its result shape completely. The CodeExecutionResult has stdout, stderr, and execution_log. The SubAgentResult has nested_messages and sub_agent_usage. These fields don't pollute the canonical model — they exist only in the tool's envelope. The canonical model stays clean: it just sees List[ContentBlock] from the context-window projection.

Chapter 4 — Wiring
Integration

How the Agentic Loop Consumes Envelopes

Now let's see how this changes the agentic loop. Previously, the loop received a flat (content, image_refs) tuple from tool execution. Now it receives a ToolResultEnvelope and projects it differently for each consumer.

agent_loop.py Consumption Site 
async def _handle_tool_result(
    self,
    tool_use_block: ToolUseContent,
    envelope: ToolResultEnvelope,
):
    """Process a tool result envelope for both consumers."""

    # ── Projection 1: Feed the LLM (AgentConfig) ──
    context_blocks = envelope.for_context_window()
    
    # Build the canonical ToolResultContent for the context window.
    # This is the standard content block that every LLM provider adapter knows.
    result_block = ToolResultContent(
        tool_name=tool_use_block.tool_name,
        tool_id=tool_use_block.tool_id,
        tool_result=[b.to_dict() for b in context_blocks],  # ← Nested content blocks!
        is_error=envelope.is_error,
    )
    
    # Append to the context window (AgentConfig.messages)
    self.agent_config.messages.append(
        Message(role=Role.USER, content=[result_block])
    )

    # ── Projection 2: Log for the UI (ConversationHistory) ──
    log_entry = envelope.for_conversation_log()
    
    self.conversation_history.append_tool_result(
        tool_use_id=tool_use_block.tool_id,
        log_entry=log_entry,  # ← Rich data: logs, files, timing, nested convos
    )

The agentic loop doesn't know whether it's dealing with a code execution result, a web search result, or a sub-agent result. It just calls the two projection methods. The tool's envelope handles the rest. This is polymorphism doing what it does best — eliminating if/elif chains in the consumer.

Updating ConfigurableToolBase

The final piece is making ConfigurableToolBase aware of envelopes. Tools that want rich results override a new method; legacy tools that return plain strings still work.

configurable_tool_base.py Updated Contract 
class ConfigurableToolBase(ABC):

          - ToolResultEnvelope    → new, used directly (preferred)
        """

          - Schema validation of tool results
          - UI component resolution (which React component renders this?)

And in the ToolRegistry.execute() method, we add automatic wrapping for legacy tools:

tool_registry.py Updated 
async def execute(self, tool_name, tool_input, **kwargs) -> ToolResultEnvelope:
    """Execute a tool. Always returns a ToolResultEnvelope."""
    try:
        result = await self._invoke(tool_name, tool_input)

        # New path: tool returned an envelope directly
        if isinstance(result, ToolResultEnvelope):
            return result

        # Legacy path: wrap plain str/ToolResult in a generic envelope
        if isinstance(result, ToolResult):
            return LegacyMediaEnvelope.from_tool_result(
                tool_name=tool_name, result=result, **kwargs
            )

        # Plain string fallback
        return GenericTextEnvelope(
            tool_name=tool_name,
            tool_id=kwargs.get("tool_id", ""),
            text=str(result),
        )

    except Exception as e:
        return ToolResultEnvelope.error(tool_name, kwargs.get("tool_id", ""), str(e))
💡 Backwards Compatibility

The auto-wrapping logic means every existing tool that returns a str or ToolResult continues to work without any changes. The envelope is purely additive — tools can opt into it when they need richer result semantics. This is the Open-Closed Principle in action: the system is open for extension (new envelope types) but closed for modification (existing tools don't break).

Chapter 6 — Rejected Alternatives
Developer's Decision Log

Paths Not Taken (And Why)

A senior developer's diary isn't just about what they built — it's about what they considered and rejected. Here are the alternatives that were weighed.

Alternative A: "Just Make ToolResultBase Fatter"

Rejected "I could add optional fields to ToolResultBase — stdout, stderr, nested_messages, execution_log, etc. But this creates a God Object. Every tool would see every other tool's fields. The type system can't enforce which combinations are valid. And every new tool type means modifying the canonical model, violating Open-Closed."

Alternative B: "Stuff Everything in kwargs"

Rejected "I could use the existing kwargs field to carry tool-specific data. But kwargs is a Dict[str, Any] — completely untyped. No autocomplete, no validation, no documentation. Every consumer would need to know the exact key names for each tool type, and there's no compile-time help. This is the 'stringly typed' anti-pattern."

Alternative C: "Two Separate Return Types from Tools"

Rejected "I could have tools return a tuple of (context_window_blocks, conversation_log_dict). But this couples the tool to knowledge of both consumers. If a third consumer appears (e.g., analytics/billing), every tool needs updating. The envelope pattern defers projection to method calls, so adding a third projection method is a single interface change."
🔑 The Decisive Factor

The envelope won because it's the only approach where adding a new tool type requires zero changes outside the tool's own code, and adding a new consumer requires only adding one abstract method to the base. Both axes of change are isolated. This is the hallmark of a well-factored design — changes in one dimension don't ripple into others.

Chapter 7 — The Registry
The Central Coordinator

ToolRegistry — Registration, Execution, Classification

With the envelope pattern solving what tools return, a separate question remains: who manages the tools themselves? Somewhere between the LLM requesting a tool call and the tool executing, there's a coordinator that knows which tools exist, routes calls to the right function, runs them in parallel when possible, and classifies them for relay.

That coordinator is the ToolRegistry. It has a deceptively simple interface — register tools, get schemas, execute calls — but the decisions beneath the surface are worth examining.

Developer's inner voice "I want the registry to be a dumb switchboard. It shouldn't know about providers. It shouldn't know about the LLM. It receives decorated functions, stores their schemas, and when asked to execute, it calls the function and wraps the result. The intelligence lives in the tools and their envelopes — the registry is just the dispatch layer."

Registration

Tools arrive at the registry as plain Python functions decorated with @tool. The decorator attaches three metadata attributes: __tool_schema__ (the canonical schema), __tool_executor__ (backend or frontend), and __tool_needs_confirmation__. The registry reads these off and stores them in a RegisteredTool dataclass.

registry.py Registration 
@dataclass
class RegisteredTool:
    """Internal metadata for a registered tool."""
    name: str
    func: Callable
    schema: dict
    executor: ExecutorType = "backend"
    needs_confirmation: bool = False

class ToolRegistry:
    def register_tools(self, tools: list[Callable]) -> None:
        for func in tools:
            if not hasattr(func, "__tool_schema__"):
                raise ValueError(
                    f"Function '{func.__name__}' is missing __tool_schema__."
                )
            schema = func.__tool_schema__
            executor = getattr(func, "__tool_executor__", "backend")
            needs_confirm = getattr(func, "__tool_needs_confirmation__", False)
            self._tools[schema["name"]] = RegisteredTool(
                name=schema["name"], func=func, schema=schema,
                executor=executor, needs_confirmation=needs_confirm,
            )

The critical constraint: only @tool-decorated functions can register. If you pass an undecorated function, you get a clear error. This eliminates an entire class of runtime surprises — every registered tool is guaranteed to have a well-formed schema.

Execution — Sync, Async, and Auto-Wrapping

The execution pipeline handles three concerns: dispatch (sync vs async), wrapping (raw returns into envelopes), and timing.

registry.py Execute 
async def execute(self, tool_name, tool_id, tool_input) -> ToolResultEnvelope:
    if tool_name not in self._tools:
        return ToolResultEnvelope.error(tool_name, tool_id, f"Unknown tool")

    registered = self._tools[tool_name]
    start = time.monotonic()

    try:
        if inspect.iscoroutinefunction(registered.func):
            result = await registered.func(**tool_input)
        else:
            result = await asyncio.to_thread(registered.func, **tool_input)

        envelope = self._wrap_result(result, tool_name, tool_id)

    except Exception as e:
        envelope = ToolResultEnvelope.error(tool_name, tool_id, str(e))

    envelope.duration_ms = (time.monotonic() - start) * 1000
    return envelope

Two key decisions here. First, sync tools run in a thread via asyncio.to_thread(). This prevents a blocking tool from freezing the event loop. Second, every code path returns an envelope — success, failure, unknown tool. The caller never has to handle raw exceptions from tools.

Bounded Parallel Execution

LLMs often request multiple tool calls in a single response. The registry executes them concurrently with a semaphore-based bound.

registry.py Parallel 
async def execute_tools(self, tool_calls, max_parallel=5):
    # Fast path: single call, no concurrency overhead
    if len(tool_calls) == 1:
        return [await self.execute(tc.name, tc.tool_id, tc.input)]

    semaphore = asyncio.Semaphore(max_parallel)
    results = [None] * len(tool_calls)

    async def _run_one(index, tc):
        async with semaphore:
            results[index] = await self.execute(tc.name, tc.tool_id, tc.input)

    await asyncio.gather(*[
        asyncio.create_task(_run_one(i, tc))
        for i, tc in enumerate(tool_calls)
    ])
    return results

Results are returned in input order regardless of execution order — the indexed results array guarantees this. The single-call fast path avoids semaphore and gather overhead for the common case.

Relay Classification

Not all tools execute the same way. Backend tools run immediately on the server. Frontend tools relay to the browser. Confirmation tools need user approval first. The registry classifies each batch of tool calls into three buckets:

registry.py Classification 
@dataclass
class ToolCallClassification:
    backend_calls: list[ToolCallInfo]       # Execute immediately on server
    frontend_calls: list[ToolCallInfo]      # Relay to browser for client-side execution
    confirmation_calls: list[ToolCallInfo]  # Requires user approval before execution

    @property
    def needs_relay(self) -> bool:
        return bool(self.frontend_calls or self.confirmation_calls)

The agent loop checks classification.needs_relay to decide whether to execute all tools immediately or pause for frontend/user input. Unknown tools go to the backend bucket — execute() will return an error envelope for them.

🧩 Design Principle — Separation of Dispatch and Execution

The registry is a dispatcher, not an executor. It routes calls to the right function and wraps results, but it never interprets tool semantics. It doesn't know what "code execution" means or what "frontend" means — it just reads the executor attribute and sorts. This keeps the registry stable as new tool types and execution modes appear.

Chapter 8 — The Decorator
Schema from Source

The @tool Decorator — Turning Functions into Tools

There's a question that comes before registration: how does a plain Python function become a tool? The answer is the @tool decorator, which generates a complete schema from the function's type hints and docstring — no hand-written JSON, no separate schema file, no divergence between implementation and description.

Developer's inner voice "I've been burned by hand-maintained schemas before. The function signature says limit: int but the schema says type: string. The docstring says 'Maximum number of results' but the schema says 'Limit parameter'. Two sources of truth that inevitably drift apart. The solution is obvious: there should be exactly one source of truth, and it should be the code itself."

How It Works

The decorator is non-invasive — it returns the original function unchanged, only adding metadata attributes. This means the function remains callable as normal Python, testable without framework scaffolding.

decorators.py Contract 
def tool(func=None, *, executor="backend", needs_user_confirmation=False):
    """Decorator that generates and attaches a tool schema to a function."""
    def decorator(fn):
        schema = generate_tool_schema(fn)
        fn.__tool_schema__ = schema           # The canonical schema dict
        fn.__tool_executor__ = executor        # "backend" or "frontend"
        fn.__tool_needs_confirmation__ = needs_user_confirmation
        return fn  # ← Returns the original function, unchanged

    if func is not None:
        return decorator(func)  # @tool (no parens)
    return decorator            # @tool(executor="frontend")

Three usage forms, all supported by a single implementation using the func=None sentinel:

Schema Generation — The Engine Inside

The real complexity lives in generate_tool_schema(). It does three things: parses type hints into JSON Schema types, parses Google-style docstrings for descriptions, and merges them into a canonical schema.

schema_utils.py Type Mapping 
# Python type → JSON Schema type
_BASE_TYPE_MAPPING = {
    int:   {"type": "integer"},
    float: {"type": "number"},
    str:   {"type": "string"},
    bool:  {"type": "boolean"},
    list:  {"type": "array"},
    dict:  {"type": "object"},
    Any:   {},  # Empty schema = unconstrained
}

# Recursive type parsing handles generics, unions, optionals, literals
def _parse_type_hint(hint) -> dict:
    origin = get_origin(hint)
    args = get_args(hint)

    if origin is None:
        return _get_json_schema_type(hint)

    elif origin is Union:
        return _parse_union_type(args)  # Handles Optional[X] → nullable

    elif origin is list:
        return {"type": "array", "items": _parse_type_hint(args[0])}

    elif origin is dict:
        return {"type": "object", "additionalProperties": _parse_type_hint(args[1])}

    elif origin is Literal:
        return {"enum": list(args)}  # Literal["a", "b"] → enum
    ...

The docstring parser extracts three pieces from Google-style docstrings: the function description (everything before Args:), per-parameter descriptions (the Args: block), and the return description (the Returns: block).

schema_utils.py Output

    "name": "add",
    "description": "Add two numbers.",
    "input_schema": {
        "type": "object",
        "properties": {
            "a": {"type": "number", "description": "First operand."},
            "b": {"type": "number", "description": "Second operand."},
        },
        "required": ["a", "b"],
    }
}
✅ Single Source of Truth

The function's type hints define the JSON Schema types. The function's docstring defines the description and parameter descriptions. The function's default values define which parameters are required vs optional. There is exactly one place to look, and it's the code itself. The schema is always correct because it's derived, not maintained.

Chapter 9 — Configurable Tools
When Simple Functions Aren't Enough

ConfigurableToolBase — Tools with Runtime Identity

The @tool decorator works beautifully for stateless functions. But some tools need configuration that changes their description. A code execution tool's prompt might say "Execute Python code. Maximum execution time: 30s" — but that 30s is a runtime value that varies per deployment. The library caller sets it, and the LLM needs to see it in the tool description.

Developer's inner voice "This is a tension between the decorator's simplicity and real-world needs. The decorator generates the schema at import time from a static docstring. But for configurable tools, the schema depends on constructor arguments — things that aren't known until runtime. I need a way to defer schema generation to instantiation time."

The solution is ConfigurableToolBase, an abstract base class that defers docstring rendering to instantiation and supports three levels of customization:

Level Mechanism When to Use
Default template Class-level DOCSTRING_TEMPLATE with {placeholders} Tool author provides sensible defaults; callers just instantiate
Custom template Constructor's docstring_template= parameter Caller wants a different description but same schema structure
Full override Constructor's schema_override= parameter Caller wants complete control — different name, different params
base.py Abstract Base 
class ConfigurableToolBase(ABC):
    """Abstract base for tools with templated docstrings."""

    DOCSTRING_TEMPLATE: str = ""  # Subclasses override with {placeholders}

    def __init__(self, docstring_template=None, schema_override=None):
        self._docstring_template = docstring_template
        self._schema_override = schema_override
        self._sandbox = None  # Injected later by ToolRegistry

    def _render_docstring(self) -> str:
        """Replace {placeholders} with values from _get_template_context()."""
        template = self._docstring_template or self.DOCSTRING_TEMPLATE
        context = self._get_template_context()
        return template.format(**context)

    def _apply_schema(self, func):
        """Apply schema to the inner function — override or generated."""
        if self._schema_override is not None:
            func.__tool_schema__ = self._schema_override
            func.__tool_executor__ = "backend"
            func.__tool_needs_confirmation__ = False
            return func
        # Normal path: render template → set as docstring → run @tool
        func.__doc__ = self._render_docstring()
        return tool(func)

    @abstractmethod
    def get_tool(self) -> Callable: ...

A Concrete Example

Here's how a code execution tool uses this pattern. The {timeout_seconds} placeholder gets replaced with the actual configured value:

code_exec_tool.py Concrete Tool 
class CodeExecTool(ConfigurableToolBase):
    DOCSTRING_TEMPLATE = """Execute Python code in a sandboxed environment.

    The code runs in an isolated sandbox with a maximum execution time of
    {timeout_seconds} seconds. Standard library imports are available.

    Args:
        code: The Python code to execute.
    """

    def __init__(self, timeout=30, **kwargs):
        super().__init__(**kwargs)
        self.timeout = timeout

    def _get_template_context(self):
        return {"timeout_seconds": self.timeout}

    def get_tool(self):
        instance = self
        def execute_code(code: str) -> str:
            """Placeholder"""  # Replaced by _apply_schema
            result = instance._sandbox.exec(code, timeout=instance.timeout)
            return result.stdout
        func = self._apply_schema(execute_code)
        func.__tool_instance__ = instance  # ← Key: enables sandbox injection
        return func

Sandbox Injection via the Registry

Notice func.__tool_instance__ = instance. This is the bridge that enables sandbox injection. When the agent initializes, the registry calls attach_sandbox(), which walks all registered tools looking for this attribute:

registry.py Injection 
def attach_sandbox(self, sandbox):
    self._sandbox = sandbox
    for registered in self._tools.values():
        instance = getattr(registered.func, "__tool_instance__", None)
        if instance and callable(getattr(instance, "set_sandbox", None)):
            instance.set_sandbox(sandbox)

The lifecycle flows naturally: construct tool → register → attach sandbox → tool calls use self._sandbox for all I/O. Tools never import or create sandboxes themselves.

💡 Template Method Pattern

ConfigurableToolBase is a textbook Template Method. The base class defines the algorithm skeleton (_render_docstring → _apply_schema → register), and subclasses fill in the blanks (DOCSTRING_TEMPLATE, _get_template_context(), get_tool()). The framework controls the flow; the tool author provides the specifics.

Chapter 10 — The Schema Boundary
A Refactoring Story

Moving Provider Knowledge Out of the Registry

The ToolRegistry originally had a method that knew about providers:

registry.py Before — The Smell 
def get_schemas(self, format="anthropic") -> list[dict]:
    schemas = [t.schema for t in self._tools.values()]

    if format == "anthropic":
        return [s.copy() for s in schemas]

    if format == "openai":
        return [{
            "type": "function",
            "function": {
                "name": s["name"],
                "description": s["description"],
                "parameters": s["input_schema"],
            }
        } for s in schemas]

    raise ValueError(f"Unsupported format '{format}'")
Developer's inner voice "This method is a code smell. The ToolRegistry lives in agent_base/tools/ — it's framework-level infrastructure. But it knows about Anthropic's schema format vs OpenAI's. Why should the tool registry care which LLM provider you're using? If I add a Google Gemini provider tomorrow, I'd have to edit registry.py. That violates the Open-Closed Principle and creates a dependency from the tool layer into the provider layer."
Fig 4 — Provider knowledge moves from tools layer to provider layer

The Refactoring

The fix is clean. ToolRegistry.get_schemas() drops the format parameter and returns only canonical schemas. A new abstract method on MessageFormatter takes responsibility for provider-specific conversion:

registry.py After — Canonical Only 
def get_schemas(self) -> list[dict]:
    """Get tool schemas in canonical format.

    Provider-specific conversion is the responsibility
    of each provider's MessageFormatter.format_tool_schemas().
    """
    return [t.schema.copy() for t in self._tools.values()]
messages.py New Abstract Method 
class MessageFormatter(ABC):
    """Translates canonical types into provider wire format."""

    @abstractmethod
    def format_messages(self, messages, params) -> dict: ...

    @abstractmethod
    def parse_response(self, raw_response) -> Message: ...

    @abstractmethod
    def format_tool_schemas(self, schemas: list[dict]) -> list[dict]:
        """Convert canonical tool schemas into provider wire format.
        
        ToolRegistry returns schemas in canonical form
        (name, description, input_schema). Each provider
        formatter converts them into its API's expected shape.
        """
        ...

The MessageFormatter already owns message translation — canonical Message objects in, provider wire format out. Tool schema conversion is the same kind of translation. It belongs in the same class.

Developer's inner voice "This refactoring follows a pattern I keep seeing: the canonical boundary. Everything inside agent_base/ speaks canonical types — ContentBlock, Message, ToolResultEnvelope, canonical schemas. Everything outside — the provider layer — speaks provider-specific types. The MessageFormatter is the membrane between these two worlds. The registry was poking through that membrane, and now it doesn't."
✅ The Canonical Boundary Principle

The framework core (agent_base/) should never import or reference provider-specific formats. It only speaks canonical types. The provider layer owns the translation. This means adding a new provider (Gemini, Mistral, local models) requires zero changes to the framework core — you implement three methods on a new MessageFormatter subclass and you're done.

Aspect Before After
Adding a provider Edit registry.py + write provider adapter Write provider adapter only
Registry's imports Knows Literal["anthropic", "openai"] No provider awareness at all
Schema conversion In get_schemas(format=...) In MessageFormatter.format_tool_schemas()
Responsibility Registry knows about providers Registry knows about tools; providers know about formats
Epilogue — The Thought Process
Meta-Reflection

How Senior Developers Extract Patterns

The journey we just walked through follows a repeatable mental process. It's not about knowing pattern names upfront — it's about recognizing structural tensions in code and resolving them by separating concerns.

Fig 5 — Pain → Axes of Change → Boundary → Named Pattern → Iterate

In this specific case, we identified two orthogonal axes of variation: tool type (code exec vs. web search vs. sub-agent) and consumer type (LLM context vs. UI log). The canonical model was trying to collapse both axes into one flat hierarchy, which is why it was breaking. The envelope introduces a new layer that handles the tool-type axis, while the projection methods handle the consumer-type axis. Each axis can now vary independently.

The same principle drove every design in this diary. The @tool decorator eliminates the axis of divergence between code and schema. ConfigurableToolBase separates the axis of "tool behavior" from the axis of "tool description." And the schema boundary refactoring separates the axis of "what tools exist" from "what format the provider expects." Each time we found two things changing for different reasons, we drew a boundary.

✅ The Litmus Test

For each axis of change, ask: "How many files do I touch?" New tool type → one file (tool + envelope). New consumer → one file (add projection method). New provider → one file (implement MessageFormatter). New tool configuration → one file (subclass ConfigurableToolBase). If the answer is always "one," the factoring is right.

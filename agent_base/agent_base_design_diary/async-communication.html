The Async Boundary

Developer Diary · Communication Patterns

Chapter 1 — The Prison
Origin of the Problem

The Prison of Synchronous RPC

The first instinct when a tool call needs to reach the front-end is the obvious one: emit a tool_call event, park the agent on an asyncio.Future, wait for the front-end to reply, resume. It's RPC over a streaming transport — familiar, trivial to implement, and perfectly serviceable for a single-user demo.

It breaks along two axes at once:

Failure mode 1 — Sub-agent nesting When a sub-agent spawned by SubAgentTool also needs front-end tool results, its relay calls must bubble up through the parent's channel. That introduces scope semantics (whose reply is this?), interleaving rules (can a parent reply while a child is waiting?), and bidirectional plumbing inside a tool's implementation — all for a pattern that should be invisible.
Failure mode 2 — Multi-server deployment The map[correlation_id → Future] lives inside one Python process. The moment a second application server exists — for redundancy, rolling deploys, or horizontal scale — a reply can land on the node that never sent the request. The request times out. The agent dies. “Sticky sessions” buys you a week of simplicity and a decade of operational pain.
Figure 1 · The Two Prisons
Both failure modes come from the same assumption: the reply must land in the same memory space that issued the call. Once you break that assumption, both prisons open at once.

Both failure modes share the same root cause: we're treating the front-end/back-end boundary as an in-process RPC when it's really a distributed coordination problem. The rest of this diary is about picking the right mental model for that coordination — without overbuilding on day one.

Chapter 2 — Six Patterns
The Design Space

Six Ways to Cross the Boundary

Each pattern below is a genuine alternative — not a straw man. We characterise each with a one-line description, a pseudo-code sketch, the trade-offs, and a verdict. The pattern we eventually recommend (Chapter 4) is assembled from two of these.

Pattern 01

Synchronous RPC + in-memory correlation

The baseline. Emit a tool call, park an asyncio.Future keyed by correlation_id, the front-end replies over the same connection, resolve the future, resume the loop.

agent_loop.py — baseline Sketch 
async def call_frontend_tool(name, args):
    cid = uuid4().hex
    self._pending[cid] = asyncio.Future()
    await self.stream.emit(ToolCallDelta(cid, name, args))
    return await self._pending[cid]   # blocks here

async def on_frontend_reply(self, cid, result):
    self._pending[cid].set_result(result)
Pros
  • Trivially implemented. No infra, no schema changes.
  • Latency is minimal — single in-process await.
  • Perfect for single-process demos and local development.
Cons
  • Dies instantly at two or more servers without sticky sessions.
  • Sub-agent relay requires hand-rolled nesting semantics.
  • A process crash loses every in-flight run. No replay.
Verdict The prison we're escaping. Keep it as the day-zero implementation; replace the primitive underneath without changing the call-site.
Pattern 02

Correlation-ID inbox

Decouple “send” from “receive”. Every tool call gets a correlation_id; both sides use a shared inbox (log, queue, table) to publish and consume events. The agent awaits on a predicate, not a process-local future.

agent_loop.py — correlation inbox Sketch 
async def call_frontend_tool(name, args):
    cid = uuid4().hex
    await log.append(run_id, ToolCallRequested(cid, name, args))
    reply = await log.wait_for(
        run_id,
        lambda e: isinstance(e, ToolCallReplied) and e.cid == cid,
    )
    return reply.result
Pros
  • Any server can receive the reply — the log is shared state.
  • Sub-agents get their own IDs; nesting semantics vanish.
  • Replay and audit come almost for free.
Cons
  • Needs a shared event log (even if in-memory for tests).
  • Still imperative — every tool call is a round-trip.
  • Write amplification versus a bare in-memory future.
Verdict Foundational. Every pattern below 3–6 either extends this or is orthogonal to it.
Pattern 03

Subscribe, don't call

Flip the polarity for front-end-owned state (selection, cursor, viewport, open file). The front-end publishes streams; the agent subscribes. A “tool call” becomes “read the latest value” or “wait for the next one”.

subscriptions.py — topic streams Sketch 
# front-end pushes whenever state changes, whether asked or not
front_end.publish("selection", SelectionChanged(range=...))

# back-end reads the latest cached value or blocks for next
async def get_selection():
    return await subs.latest_or_next("selection")
Pros
  • Matches the data model — front-end owns UI state natively.
  • No duplicate polling; state is always fresh.
  • Sub-agents subscribe to the same topics — no relay machinery.
Cons
  • Commands (“open URL”, “prompt user”) don't fit this shape.
  • Topic discovery / lifecycle adds a second protocol surface.
  • Subscriptions must be garbage-collected when runs end.
Verdict Right tool for a subset of front-end tools — UI-state reads. Wrong tool for imperative commands. Layer it on top of the inbox.
Pattern 04

Event-sourced conversation log

Treat the entire conversation as an append-only log of events authored by either side. Agents are reducers over the log; tool calls, replies, interrupts, and message edits are all entries. State at any time = fold over events up to sequence N.

reducer.py — log-driven agent Sketch 
state = initial_state()
async for event in log.subscribe(run_id):
    state = reduce(state, event)
    if needs_action(state):
        await log.append(run_id, determine_next(state))
Pros
  • Perfect audit trail — every transition is durable.
  • Time-travel debugging and replay are first-class features.
  • Handles concurrent writers (user interrupts, sub-agents) naturally.
Cons
  • Log management surface: snapshots, compaction, schema evolution.
  • Every actor must be reducer-shaped — a real refactor.
  • Overkill for a project that hasn't proven product fit yet.
Verdict The right answer at scale; the wrong answer on day one. Pattern 2 is a strict subset — grow into this only if the audit / replay needs materialise.
Pattern 05

Durable-workflow continuations

The agent loop is no longer a live coroutine. At each step it writes a checkpoint (“awaiting event matching P”) and releases the worker. When a matching event arrives, any worker can pick the run up and advance it. This is the shape Temporal / Restate / Inngest productise.

workflow.py — park & resume Sketch 
@workflow
async def agent_run(run_id):
    msg = await workflow.await_event("user_message")
    call = llm(msg)
    while call.has_tool_call:
        result = await workflow.await_event("tool_reply", cid=call.cid)
        call = llm(result)
    return call.text
Pros
  • Crash-tolerant by construction — restart-safe at every step.
  • Rolling deploys and horizontal scale are free.
  • Sub-agents are child workflows; scheduler handles them uniformly.
Cons
  • Adopting Temporal-class infra is a step change in operational surface.
  • Workflow code is non-deterministic-by-default — discipline required.
  • Debugging crosses a scheduler you didn't write.
Verdict The ceiling. Adopt only when operational pain from other patterns is real — not before.
Pattern 06

Shared CRDT document

No request/response at all. Both sides observe mutations to a shared, convergent document (Yjs / Automerge). A “tool call” is a write of an intent node; a “reply” is a write of a result node. The protocol is the document.

crdt.py — convergent state Sketch 
# back-end writes an intent
doc["tool_calls"][cid] = {"name": "open_url", "args": ...}

# front-end observes, writes the result
doc["tool_replies"][cid] = {"ok": True, "data": ...}

# back-end observes the reply appear — no RPC framing needed
Pros
  • Convergent under partitions; offline-capable by default.
  • Elegant when the UX is genuinely collaborative.
  • Unified model for state, commands, and conflicts.
Cons
  • High learning curve; CRDT semantics are hard to introspect.
  • Overkill for predominantly imperative tool dispatch.
  • Debugging “why did this value converge this way” is hard.
Verdict Right tool for collaborative documents. Wrong tool for an agent-tool boundary. Parked.
The axis that matters Patterns 1–6 vary along a single hidden axis: where state lives — in a process (1), in a shared log (2, 4, 5), in a publication stream (3), or in a convergent document (6). Multi-server robustness is really just “state does not live in any single process”.
Chapter 3 — Comparison
Side-by-Side

Patterns on Six Axes

Scoring is deliberately coarse — three levels per axis. The axes chosen are the ones the user flagged as load-bearing: robustness and multi-server readiness, plus the usual suspects (complexity, sub-agent fit, observability, ops overhead).

Pattern Build Complexity Crash Robustness Multi-Server Sub-Agent Fit Observability Ops Overhead
01 Sync RPC Low None No Poor Logs only None
02 Correlation inbox Medium Good Yes Excellent Log-backed Postgres
03 Subscribe Medium Partial Yes Excellent Stream taps Pub/sub
04 Event-sourced log High Excellent Yes Excellent First-class Snapshot + compact

Reading the table Pattern 2 is the only one that scores good on every axis without scoring bad on build-complexity or ops-overhead. That's not an accident — it's the minimum design that crosses the “state lives outside a single process” threshold.
Chapter 4 — The Recommendation
The Floor, Not the Ceiling

Correlation-ID Inbox on a Durable Log

The recommendation is Pattern 2 as the primary mechanism, with Pattern 3 layered in for UI-state tools. The agent loop stays a plain async def — no Temporal, no workflow DSL, no CRDT runtime. The multi-server escape hatch is a fourth storage adapter that the application code never has to care about directly.

Extend the three-adapter pattern to four

The agent_base/storage/ module already enforces a three-adapter discipline for agent config, conversations, and run logs. Each adapter has memory, filesystem, and Postgres backends. We add a fourth peer:

storage/base.py — proposed EventLogAdapter Planned ABC 
class EventLogAdapter(ABC):
    """Durable, run-scoped event log.

    All cross-boundary coordination (tool calls, replies, interrupts,
    reconnect replay) flows through append/read/wait primitives.
    """

    @abstractmethod
    async def append(self, run_id: str, event: Event) -> int: ...
    #                                                     ^ monotonic seq

    @abstractmethod
    async def read_after(
        self, run_id: str, seq: int
    ) -> AsyncIterator[Event]: ...

    @abstractmethod
    async def wait_for(
        self,
        run_id: str,
        predicate: Callable[[Event], bool],
        timeout: float | None = None,
    ) -> Event: ...

One call-site helper

The agent loop gets a single helper that hides the log behind the same shape a future used to take — making the migration a drop-in replacement:

core/agent_base.py — await_correlation Helper 
async def await_correlation(self, cid: str) -> ToolReply:
    return await self.event_log.wait_for(
        self.run_id,
        lambda e: isinstance(e, ToolCallReplied) and e.cid == cid,
        timeout=self.config.frontend_tool_timeout,
    )

Backend choices stay swappable

BackendWhenHow it works
memory.py Tests, local dev, single-process demos asyncio.Queue per run_id. Zero infra.
postgres.py Production, multi-server events(run_id, seq, payload) table with LISTEN/NOTIFY for push, FOR UPDATE SKIP LOCKED for worker pull.
redis.py If Postgres throughput ceilings are hit Redis Streams with consumer groups. Same EventLogAdapter interface — application code untouched.

Anatomy of the stream bus

The log's runtime shape is three streams keyed by run_id, with producers and consumers on opposite sides. Inbound events (user messages, tool replies, reconnects) flow left-to-right; outbound events (deltas, tool calls, checkpoints) flow right-to-left; a global control stream handles lifecycle. Postgres shadows the critical events as the durable book of record.

Figure 2 · The Stream Bus
Stream lanes keyed by run_id replace the per-process correlation map. The bus is the rendezvous; edges and workers are fungible pods that attach to it via consumer groups. Postgres shadows the critical events as the durable book of record.

Why this also solves sub-agents

Sub-agents spawned by agent_base/common_tools/sub_agent_tool.py write to the same log scoped by parent_run_id / child_run_id. A sub-agent's awaiting tool call is the same shape as the root agent's — one ToolCallRequested event, one matching ToolCallReplied event, picked up by whichever worker is free. The “relay” concept disappears because nesting is just another partition key.

Figure 3 · One Primitive, Three Rescues
The same primitive — append, read_after, wait_for on a run-scoped log — turns three different crises into the same non-event. Nesting, crashes, and disconnects all dissolve into sequence numbers.
What this gets you Crash survival (runs resume from last seq), reconnect replay (client subscribes from last acknowledged seq, no sticky session), uniform sub-agent semantics (same primitive at any depth), and user interrupts for free (just another inbound event with no pending correlation).
Chapter 5 — Migration Ladder
How We Get There

Four Steps, No Big-Bang Rewrites

The recommendation is designed so every step is independently shippable and preserves the application code's shape. We climb the ladder only as the next rung's pain becomes real.

  1. Today — in-process correlation
    Keep Pattern 1. asyncio.Future keyed by cid. Works for the FastAPI demo, works for single-box deployments. No change required.
  2. Step 1 — introduce EventLogAdapter
    Add the ABC with in-memory and Postgres backends. Replace self._pending[cid] = Future() with await_correlation(cid). Single-process behaviour is unchanged. Multi-server deployment now possible — any worker can complete any run.
  3. Step 2 — reconnect-safe front-end
    Edge servers become stateless: a WebSocket / SSE connection just tails read_after(run_id, last_seq). Clients reconnect to a different server transparently. No sticky sessions; rolling deploys are free.
  4. Step 3 — only if scale demands
    Swap the Postgres backend for Redis Streams / NATS JetStream, or promote the whole loop to a durable-workflow engine (Pattern 5). The EventLogAdapter interface doesn't change; the agent loop doesn't change. We earn the operational complexity only when it pays for itself.
A note on timing Step 1 is the only step we're signing up for near-term. Steps 2 and 3 are affordances the design preserves — not commitments. The point is to avoid painting ourselves into a corner, not to build everything at once.
Chapter 6 — Non-Commitments
Guard Rails

What This Page Does Not Commit To

The constraint from day one was: simple, maintainable, flexible — no over-engineering. The recommendation is the floor. This chapter enumerates the ceilings we deliberately do not commit to.

Not adopting Temporal / Restate / Inngest. A workflow engine is a step change in operational surface. We preserve the option to adopt one via the EventLogAdapter abstraction, but we do not build on one day one.
Not building A CRDT layer. Our domain is agent-tool dispatch, not collaborative document editing. The introspection cost is not worth the theoretical elegance.
Not forcing All tools into subscribe semantics. Imperative commands stay imperative. Subscribe is an additional shape available for front-end-owned state, not a replacement for request/reply everywhere.
Not promising Effectively-once semantics at the application layer. The log gives at-least-once delivery plus idempotency keys. Tool handlers are responsible for idempotency on their side — a deliberate, documented constraint, not a hidden invariant.
The principle Build the smallest thing that makes the right thing possible. The EventLogAdapter is that smallest thing. Everything beyond it is an option we've kept open — not a promise we've made.

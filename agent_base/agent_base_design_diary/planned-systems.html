Implemented Context Management

Compaction & Memory — Two Time Scales

These two systems serve different time horizons and never interact. Compaction manages the within-session context window. Memory manages cross-session knowledge. Both are wired into the agent loop but at different points.

CompactorMemoryStore
Scope Within-session context window Across-session knowledge
Frequency Every step of the agent loop Once at run start + once at run end
Input AgentConfig (full message list) Message list + conversation history
Output (bool, list[Message]) list[ContentBlock] (retrieve) / dict (update)

Compaction — Per-Provider Controllers

Unlike the original multi-strategy plan, compaction is now implemented as a provider-specific CompactionController co-located with each provider under providers/<name>/compaction.py. Each controller knows how to invoke its own provider to generate summaries, how to strip provider-specific artefacts (e.g. thinking blocks, cache markers), and how to find a safe user-message boundary before which history can be replaced with a summary.

ControllerLocationApproach
CompactionController (Anthropic) providers/anthropic/compaction.py LLM-summarized history replacement at a safe user-message boundary, with streamed progress deltas
CompactionController (LiteLLM) providers/litellm/compaction.py Same contract, adapted for LiteLLM's SDK
providers/<name>/compaction.py + memory/base.py Controller + ABC 
class CompactionController:
    def should_compact(self, messages, estimated_tokens) -> bool: ...
    def find_safe_boundary(self, messages, preserve_tokens) -> int: ...
    async def compact(self, context_messages, model,
                      agent_uuid, queue, stream_formatter, reason)
        -> list[Message]: ...

class MemoryStore(ABC):
    @abstractmethod
    async def retrieve(self, user_message, messages) -> list[ContentBlock]: ...
    @abstractmethod
    async def update(self, messages, conversation_log) -> dict: ...
Design Decision Compaction runs before each LLM call when the token threshold is exceeded. Memory retrieval runs once at run start (injected as context). Memory update runs once at run end. They share no state. Locating the controller next to the provider avoids a leaky generic ABC: summarization, safe-boundary detection, and artefact stripping all need provider-specific knowledge.
Implemented Persistence

Storage Adapters — Three Independent Axes

The three-adapter pattern is carried over from anthropic_agent largely as-is. Each adapter is independently swappable — you can run config in Postgres, conversations on the filesystem, and run logs in memory.

AdapterEntityPurpose
AgentConfigAdapter AgentConfig Session state — messages, model, relay state
ConversationAdapter Conversation Per-run records for UI history / pagination
AgentRunAdapter AgentRunLog Step-by-step execution logs for debugging

Backends: memory (default), filesystem, and Postgres. Factory: create_adapters(backend_type, **kwargs).

Recent Enhancements (March 2026)

The Postgres adapter gained several new capabilities to support the relay system and improved conversation management:

  • load_by_run_id() — Load a conversation by run_id (for relay state restoration)
  • upsert_conversation() — Insert or update conversations
  • Per-agent sequence numbering for pagination
  • Relay state serialization — PendingToolRelay persisted with tool call info and pre-computed backend results
  • Cost breakdown persistence — CostBreakdown stored per conversation
  • Media registry sync — MediaMetadata index stored alongside config
Design Decision Each adapter is truly independent. You can run AgentConfigAdapter on Postgres while ConversationAdapter runs on the filesystem. They share no state, no transactions, no connection pools. This lets teams optimize persistence per-concern: fast in-memory for config during development, durable Postgres for production conversations.
Implemented Providers

Provider Implementations

Each provider directory contains a small family of files — each owning one concern. Two providers are fully implemented: Anthropic and LiteLLM. Both support streaming, retry, cache control, relay/pause, and cost tracking. A native OpenAI provider remains on the roadmap. See the Provider Layer diary for the full design narrative.

Planned Still on the Roadmap

What Remains

With Anthropic + LiteLLM providers, streaming, relay, compaction, and pricing all shipped, the remaining planned work focuses on provider breadth, sandbox isolation, and advanced memory:

SystemStatusNotes
providers/openai/ Planned Native OpenAI provider — same file pattern as Anthropic/LiteLLM, different wire format. LiteLLM can already route to OpenAI today.
sandbox/docker.py Interface defined DockerSandbox container lifecycle — ABC exists, concrete implementation pending
sandbox/e2b.py Interface defined E2B cloud sandbox — ABC exists, concrete implementation pending
media_backend/memory.py Planned In-memory MediaBackend for tests and ephemeral runs
WebSocket StreamFormatter Planned Alternative transport for bi-directional streaming (sibling to JsonStreamFormatter)
Advanced memory stores Planned Vector-backed and graph-backed cross-session memory (only NoOpMemoryStore ships today)
storage/EventLogAdapter Designed Fourth storage adapter: durable, run-scoped event log backing the front-end / sub-agent async boundary. Full design rationale and pattern comparison in The Async Boundary.
Design Decision The architecture was designed so that each planned system slots into an existing extension point. Adding OpenAI touches providers/openai/ only. Adding DockerSandbox touches sandbox/docker.py only. The async-communication work extends the three-adapter storage pattern to four — no core changes required.

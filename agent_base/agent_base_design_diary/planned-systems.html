Planned Context Management

Compaction & Memory — Two Time Scales

These two systems serve different time horizons and never interact. Compaction manages the within-session context window. Memory manages cross-session knowledge. Both are wired into the agent loop but at different points.

CompactorMemoryStore
Scope Within-session context window Across-session knowledge
Frequency Every step of the agent loop Once at run start + once at run end
Input AgentConfig (full message list) Message list + conversation history
Output (bool, list[Message]) list[ContentBlock] (retrieve) / dict (update)

Compaction Strategies

StrategyApproach
NoOpCompactor Pass-through, no compaction
SlidingWindowCompactor Default. 4-phase progressive: remove thinking → truncate tool results → placeholder old results → remove old turns
ToolResultRemovalCompactor 2-phase: replace old tool results → remove old turns
SummarizingCompactor LLM-based summarization of older history
compaction/base.py + memory/base.py ABC 
class Compactor(ABC):
    @abstractmethod
    async def apply_compaction(self, agent_config: AgentConfig)
        -> tuple[bool, list[Message]]: ...

class MemoryStore(ABC):
    @abstractmethod
    async def retrieve(self, user_message, messages) -> list[ContentBlock]: ...
    @abstractmethod
    async def update(self, messages, conversation_history) -> dict: ...
Design Decision Compaction runs before each LLM call. Memory retrieval runs once at run start (injected as context). Memory update runs once at run end. They share no state. This separation means you can swap compaction strategies without affecting memory, and vice versa.
Planned Persistence

Storage Adapters — Three Independent Axes

The three-adapter pattern is carried over from anthropic_agent largely as-is. Each adapter is independently swappable — you can run config in Postgres, conversations on the filesystem, and run logs in memory.

AdapterEntityPurpose
AgentConfigAdapter AgentConfig Session state — messages, model, relay state
ConversationAdapter Conversation Per-run records for UI history / pagination
AgentRunAdapter AgentRunLog Step-by-step execution logs for debugging

Backends: memory (default), filesystem, and Postgres. Factory: create_adapters(backend_type, **kwargs).

Design Decision Each adapter is truly independent. You can run AgentConfigAdapter on Postgres while ConversationAdapter runs on the filesystem. They share no state, no transactions, no connection pools. This lets teams optimize persistence per-concern: fast in-memory for config during development, durable Postgres for production conversations.

Agent Construction

example — wiring an agent Composition 
agent = AnthropicAgent(
    system_prompt="You are a helpful assistant.",
    model="claude-sonnet-4-5",
    config=AnthropicLLMConfig(thinking_tokens=10000),

    tools=[my_tool_a, my_tool_b],
    subagents={"research": research_agent},

    sandbox=DockerSandbox(image="python:3.12-slim"),
    media_backend=S3MediaBackend(bucket="my-bucket"),
    compactor=SlidingWindowCompactor(),
    memory_store=NoOpMemoryStore(),

    config_adapter=PostgresAgentConfigAdapter(db_url),
    conversation_adapter=PostgresConversationAdapter(db_url),
    run_adapter=PostgresAgentRunAdapter(db_url),
)

  │
  ├── initialize() if needed
  ├── memory_store.retrieve() → inject context
  ├── append prompt to messages + conversation
  │
  └── _resume_loop():
       while step < max_steps:
       │
       ├── compactor.apply_compaction(agent_config)
       │
       ├── provider.generate[_stream](         ← Provider handles:
       │     messages, tool_schemas,              formatter.format_messages()
       │     llm_config, model, ...               SDK API call + retry
       │   ) → Message                            formatter.parse_response()
       │
       ├── append assistant message
       │
       └── match stop_reason:
            │
            ├── "pause_turn" → continue
            │
            ├── "tool_use":
            │    envelopes = tool_registry.execute_tools(tool_calls)
            │    for envelope:
            │      .for_context_window() → agent_config.messages
            │      .for_conversation_log() → conversation
            │    continue
            │
            └── "end_turn":
                 memory_store.update()
                 build AgentResult → return

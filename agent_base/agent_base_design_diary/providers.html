Chapter 1 — Provider ABC
The Contract

Provider as Boundary

The Provider ABC is the seam between the agent orchestration loop and the LLM SDK. The agent owns the loop — step counting, tool dispatch, compaction, memory, relay handling. The provider owns the transport — authentication, request building, retry, response parsing, stream event translation.

This separation is deliberate. The orchestration logic is the same whether you talk to Anthropic, OpenAI, or a local model. The wire format, authentication, retry semantics, and streaming event shapes are not. The Provider ABC captures exactly the two operations the agent needs:

core/provider.py ABC 
class Provider(ABC):
    """Abstract base class for LLM providers."""

    formatter: MessageFormatter

    @abstractmethod
    async def generate(
        self,
        system_prompt: str | None,
        messages: list[Message],
        tool_schemas: list[ToolSchema],
        llm_config: LLMConfig,
        model: str,
        max_retries: int,
        base_delay: float,
        agent_uuid: str = "",
    ) -> Message: ...

    @abstractmethod
    async def generate_stream(
        self,
        ...,
        queue: asyncio.Queue,
        stream_formatter: StreamFormatter,
        stream_tool_results: bool = True,
    ) -> Message: ...

Both methods accept canonical types: Message, ToolSchema, LLMConfig. Both return a canonical Message. The agent never sees wire-format dicts, Anthropic SDK objects, or OpenAI response classes. The Provider is the translation layer.

Developer's inner voice "The formatter attribute on Provider is the key. It's a MessageFormatter — another ABC that handles canonical-to-wire and wire-to-canonical translation. The Provider uses it, but the agent can also access it for schema formatting. The Provider has-a Formatter; it doesn't do the formatting itself."

Separation of Concerns

  Agent (orchestration)          Provider (transport)          LLM API
  =====================          ====================          =======

  step counting                  authentication
  tool dispatch        ------>   request building     ------>   HTTP
  compaction           <------   retry/backoff        <------   Response
  memory                         response parsing
  relay handling                  stream event translation

                                 MessageFormatter
                                 ================
                                 canonical ↔ wire format
                                 (pure translation, no I/O)
Provider as the boundary between orchestration and transport
The Module Rule

core/provider.py must NOT import any provider SDK. No import anthropic, no import openai. Provider-specific implementations live in providers/<name>/. This keeps the core layer SDK-agnostic and allows the agent to be instantiated without installing unused SDKs.

Chapter 2 — The Anthropic Provider
Concrete Implementation

AnthropicProvider — Owning the Transport

The AnthropicProvider is the concrete implementation that bridges the canonical world to Anthropic's Messages API. It owns four concerns: client lifecycle, request building, response parsing, and the cache control strategy.

providers/anthropic/provider.py Implementation 
class AnthropicProvider(Provider):

    def __init__(self,
        client: anthropic.AsyncAnthropic | None = None,
        formatter: AnthropicMessageFormatter | None = None,
    ) -> None:
        self.client = client or anthropic.AsyncAnthropic()
        self.formatter = formatter or AnthropicMessageFormatter()

The client is created lazily from ANTHROPIC_API_KEY if none is injected. The formatter defaults to AnthropicMessageFormatter. Both are injectable for testing.

Cache Control: 4-Block Priority Strategy

Anthropic limits cache_control to 4 blocks maximum. The provider implements a priority-based allocation strategy that picks the most impactful blocks to cache:

PriorityTargetRationale
1 System prompt Stable across turns, often large. Requires ≥ min_tokens threshold.
2 Document/image blocks Expensive to re-process. Likely to appear in first user message.
3 Large text blocks (by size desc.) Biggest bang per cache slot. Sorted descending to fill remaining slots.
4 Recent message blocks (fallback) Walk backwards from newest message. Skips thinking/redacted blocks.

Key insight "Thinking and redacted_thinking blocks are never cacheable. The _NON_CACHEABLE_CACHE_CONTROL_TYPES set ensures the fallback pass skips them when walking recent messages backward. Request Building

The _build_request_params() method assembles the complete API request dict. It handles a surprising amount of conditional logic:

_build_request_params() Builder 
# Always present
request_params = {
    "model": model,
    "max_tokens": llm_config.max_tokens or 16384,
    "messages": wire_messages,
}

# Conditional: system, thinking, tools, betas, container
if llm_config.thinking_tokens:

        "type": "enabled",

Response Parsing

The _build_response_message() method extracts usage data — including the provider-specific fields like context_management, inference_geo, service_tier, and speed that Anthropic returns on the response usage object:

_build_response_message() Usage

)

    content=content_blocks,

Developer's inner voice "The raw_usage dict is the escape hatch. We extract the canonical fields (input, output, cache) into typed attributes, but we also dump the entire usage object into raw_usage. The pricing calculator reads raw_usage to get cache TTL breakdowns and server tool counts — things that only Anthropic's API reports. New fields appear in raw_usage automatically without schema changes."
Chapter 3 — The Formatter
Wire Format Translation

AnthropicMessageFormatter — The Rosetta Stone

The formatter is a pure translator. No HTTP calls, no retries, no side effects. It has three public methods and a surprisingly deep private implementation.

  Canonical World                      Anthropic Wire Format
  ===============                      =====================

  TextContent          <--->           {"type": "text", "text": "..."}
  ThinkingContent      <--->           {"type": "thinking", ...} / {"type": "redacted_thinking", ...}
  ImageContent         <--->           {"type": "image", "source": {...}}
  DocumentContent      <--->           {"type": "document", "source": {...}}
  AttachmentContent    <--->           {"type": "container_upload", "file_id": "..."}
  ToolUseContent       <--->           {"type": "tool_use", "id": "...", ...}
  ServerToolUseContent <--->           {"type": "server_tool_use", ...}
  MCPToolUseContent    <--->           {"type": "mcp_tool_use", "server_name": "...", ...}
  ToolResultContent    <--->           {"type": "tool_result", "content": [...]}
  ServerToolResult...  <--->           {"type": "web_search_tool_result", ...}
  MCPToolResultContent <--->           {"type": "mcp_tool_result", "content": [...]}
  ErrorContent         <--->           {"type": "text", "text": "Error: ..."}

  format_blocks_to_wire()              parse_wire_to_blocks()
  format_tool_schemas()                _parse_citation()
Canonical ContentBlock types and their Anthropic wire-format equivalents

Source Resolution

Images and documents have multiple source types — base64, URL, and file_id. The formatter resolves each to the correct Anthropic wire structure:

formatters.py — _resolve_image_source() Resolver 
@staticmethod
def _resolve_image_source(block: ImageContent) -> dict:
    st = block.source_type or SourceType.BASE64
    if st in (SourceType.URL, "url"):
        return {"type": "url", "url": block.data}
    if st in (SourceType.FILE_ID, "file_id", SourceType.FILE, "file"):
        return {"type": "file", "file_id": block.data}
    # Default / base64
    return {"type": "base64", "media_type": block.media_type,
            "data": block.data}

Documents add an extra wrinkle: text/plain content uses {"type": "text"} for the source, while PDFs and other binary formats use {"type": "base64"}. The formatter handles this silently.

Citation Parsing: Five Types

Anthropic's API returns five distinct citation types. The formatter's _parse_citation() method maps each to a canonical dataclass:

Wire TypeCanonical ClassKey Fields
char_location CharCitation document_index, start/end char index
page_location PageCitation document_index, start/end page number
content_block_location ContentBlockCitation document_index, start/end block index
web_search_result_location WebSearchResultCitation url, title, encrypted_index
search_result_location SearchResultCitation search_result_index, source, title

Citations are stored in two forms on the canonical TextContent: the raw wire dicts in kwargs["citations"] for round-trip fidelity, and typed canonical objects in kwargs["canonical_citations"] for programmatic access. The wire dicts are stripped of response-only fields like file_id that the API rejects on input.

Compaction Blocks & Empty Block Filtering

Two subtle behaviors in _block_to_wire() that prevent API errors:

  • Compaction blocks: A TextContent with kwargs["compaction"] = True becomes {"type": "compaction", "content": ...}. Whitespace-only content is converted to null for wire safety.
  • Empty text filtering: Citation-only response blocks may have text="". The API rejects empty text content blocks, so _block_to_wire() returns None for empty text, and the list comprehension in format_blocks_to_wire() filters them out.
formatters.py — format_blocks_to_wire() Filter 
def format_blocks_to_wire(self, blocks: List[ContentBlock]) -> list[dict]:
    """Convert canonical ContentBlocks to Anthropic wire-format dicts."""
    return [
        wire for block in blocks
        if (wire := self._block_to_wire(block)) is not None
    ]

Thinking Block Graceful Degradation

Edge case "Anthropic requires a signature on thinking blocks for round-trip. But what if we have thinking content without a signature? Maybe from deserialization of an older format. We degrade it to a text block wrapped in <thinking> tags. The content is preserved; the API doesn't reject it. Log a warning so we can investigate."

Server Tool Result Parsing

The formatter handles a growing family of server tool result types: web_search_tool_result, web_fetch_tool_result, code_execution_tool_result, bash_code_execution_tool_result, text_editor_code_execution_tool_result, tool_search_tool_result. Rather than hardcoding each, it uses a pattern match with a generic fallback:

formatters.py — _parse_block() fallback Forward Compat 
# Explicit matches for known types
case "web_search_tool_result":
    return self._parse_server_tool_result(raw_block)
case "web_fetch_tool_result":
    return self._parse_server_tool_result(raw_block)

# Generic fallback for future server tools
case block_type if block_type.endswith("_tool_result"):
    return self._parse_server_tool_result(raw_block)
Forward Compatibility Pattern

Any new server tool result type that Anthropic adds will be caught by the endswith("_tool_result") fallback. The _parse_server_tool_result() method preserves the full content via model_dump(), so no data is lost even before we add explicit handling. The formatter degrades gracefully forward.

Chapter 4 — Retry & Backoff
Resilience

Exponential Backoff with Jitter

The retry module provides two mechanisms: a decorator for non-streaming calls and a specialized function for streaming. Both use exponential backoff, but the streaming path has additional responsibilities — it must translate Anthropic stream events into canonical StreamDelta objects.

Error Classification

CategoryError TypesBehavior
Retryable RateLimitError, APIConnectionError, APITimeoutError, InternalServerError, 5xx APIStatusError Retry with backoff: base_delay * 2^attempt + jitter
Non-retryable BadRequestError, AuthenticationError, PermissionDeniedError, NotFoundError, UnprocessableEntityError Raised immediately — no point retrying
Unexpected Any other Exception Retry (may be transient network issue)
providers/anthropic/retry.py Backoff 
async def anthropic_stream_with_backoff(
    client, request_params, queue,
    max_retries=5, base_delay=5.0, ...
) -> Any:
    for attempt in range(max_retries):
        try:
            async with client.beta.messages.stream(**request_params) as stream:
                await _process_stream_events(stream, queue, ...)
                return await stream.get_final_message()

        except (RateLimitError, APIConnectionError, ...):
            if attempt < max_retries - 1:
                delay = base_delay * (2 ** attempt) + random.uniform(0, 1)
                await asyncio.sleep(delay)
            else:
                raise

        except (BadRequestError, AuthenticationError, ...):
            raise  # Non-retryable

Stream Event Translation

The _process_stream_events() function is where Anthropic's native stream events become canonical StreamDelta objects. It tracks per-index block types and buffers tool input JSON across delta events:

retry.py — _process_stream_events() Translator 
block_types: dict[int, str] = {}    # index → "text"|"tool_use"|...
tool_buffers: dict[int, dict] = {}  # index → {id, name, input_json}

async for event in stream:
    match event.type:
        # content_block_start → track type, init buffer
        # content_block_delta → stream text/thinking, buffer tool JSON
        # content_block_stop  → emit finals, flush tool buffers

# Delta types produced:
#   TextDelta      — streamed character by character
#   ThinkingDelta  — streamed character by character
#   ToolCallDelta  — emitted once at block_stop (buffered)
#   ToolResultDelta— server tool results at block_stop
#   CitationDelta  — emitted after text block_stop
#   ErrorDelta     — on stream errors
Developer's inner voice "Text and thinking are streamed character by character — each delta goes straight to the formatter and onto the queue. Tool calls are different: we buffer the input_json_delta fragments and only emit a single ToolCallDelta at content_block_stop with the complete parsed JSON. Citations are emitted after the text final marker, so the client knows the text is complete before citations arrive."

The Non-Streaming Decorator

For non-streaming calls, a simpler retry_with_backoff decorator wraps any async function:

retry.py — retry_with_backoff Decorator 
def retry_with_backoff(max_retries, base_delay):
    def decorator(func):
        async def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return await func(*args, **kwargs)
                except Exception:
                    delay = base_delay * (2 ** attempt)
                    await asyncio.sleep(delay)
        return wrapper
    return decorator

# Used in AnthropicProvider.generate():
@retry_with_backoff(max_retries=3, base_delay=1.0)
async def _create():
    return await client.beta.messages.create(**request_params)
Streaming vs Non-Streaming Retry

The non-streaming decorator retries on any exception. The streaming function discriminates — it has explicit non-retryable error handling because a failed stream attempt may have already pushed partial deltas onto the queue. The two paths serve different failure modes.

Chapter 5 — The Relay System
Pausing for External Tools

PendingToolRelay — How the Agent Pauses

Some tool calls cannot be executed server-side. A UI-only tool like EnterPlanMode needs the frontend to execute. A destructive tool like DeleteFile needs user confirmation. The relay system handles this by pausing the agent loop and resuming when external results arrive.

Tool Call Classification

When the LLM returns stop_reason: "tool_use", the agent extracts tool calls and classifies them through the ToolRegistry:

  LLM Response (stop_reason: "tool_use")
      |
      v
  _extract_tool_calls()  →  [ToolCallInfo, ToolCallInfo, ...]
      |
      v
  tool_registry.classify_tool_calls()
      |
      +--> ToolCallClassification
           |
           +-- backend_calls:      [...]   ← execute immediately
           +-- frontend_calls:     [...]   ← send to client
           +-- confirmation_calls: [...]   ← ask user first
           +-- needs_relay:        bool    ← True if any frontend/confirmation
Tool call classification flow

If needs_relay is False, all calls are backend — execute them in parallel and continue the loop. If True, the agent enters the relay path.

The Relay Path

anthropic_agent.py — relay path Pause 
if classification.needs_relay:
    # 1. Execute backend calls immediately
    backend_results = await tool_registry.execute_tools(
        classification.backend_calls, max_parallel
    )

    # 2. Store pending state
    agent_config.pending_relay = PendingToolRelay(
        frontend_calls=classification.frontend_calls,
        confirmation_calls=classification.confirmation_calls,
        completed_results=completed_result_messages,
        run_id=self._run_id,
    )

    # 3. Persist state before pausing
    await self._persist_state()

    # 4. Notify frontend via SSE
    delta = MetaDelta(
        type="awaiting_frontend_tools",
        payload={"tools": pending_tools},
    )

    # 5. Return with stop_reason="relay"
    return self._build_agent_result(response, "relay")

The PendingToolRelay Dataclass

core/config.py Dataclass 
@dataclass
class PendingToolRelay:
    """Persisted state when the agent loop is paused."""
    frontend_calls: list[ToolCallInfo]      # sent to client
    confirmation_calls: list[ToolCallInfo]  # need approval
    completed_results: list[Message]       # backend results
    run_id: str | None = None

Resumption: resume_with_relay_results()

When the frontend sends back tool results, the agent combines them with the already-computed backend results and re-enters the loop:

  resume_with_relay_results(relay_results)
      |
      v
  1. Load pending_relay from agent_config
  2. Combine:
     - pending.completed_results  (backend, already computed)
     - relay_results              (frontend/user, just arrived)
      |
      v
  3. Build combined Message (user role, ToolResultContent blocks)
  4. Append to context_messages & conversation_history
      |
      v
  5. Fire on_relay_result() hook for each relay result
     (subclass can call reconfigure() here)
      |
      v
  6. Clear pending_relay
  7. _resume_loop()  — continue where we left off
The relay resumption flow
Key design decision "The on_relay_result() hook fires after results are in context but before the loop resumes. This is the perfect injection point for mid-run reconfiguration. A subclass can inspect the relay result (e.g., the user chose plan B in EnterPlanMode) and call reconfigure(tools=[...], system_prompt='...') to swap tools and prompts for the next LLM turn. The loop never knows the rug was pulled."
Why Persist Before Pausing?

The _persist_state() call before returning ensures the relay survives server restarts. On resumption, initialize() detects pending_relay on the loaded config, restores the partial Conversation by run_id, and waits for resume_with_relay_results(). The agent is crash-safe across the pause boundary.

Chapter 6 — Pricing & Cost
Cost Tracking

From Token Counts to Dollar Amounts

The pricing module computes per-step and cumulative costs from token usage and a bundled CSV of model pricing data. The design separates data (CSV), calculation (calculate_step_cost), and accumulation (_accumulate_usage on the agent).

pricing/calculator.py — ModelPricing Data 
@dataclass
class ModelPricing:
    model_id: str
    display_name: str
    input_per_mtok: float
    output_per_mtok: float
    cache_write_5m_per_mtok: float   # 5-minute TTL
    cache_write_1h_per_mtok: float   # 1-hour TTL

    long_context_input_multiplier: float
    long_context_output_multiplier: float

Multiplier Stacking

Anthropic's pricing has four conditional multipliers that stack multiplicatively on the base CSV rates:

  effective_rate = base_rate * M_fast * M_longctx * M_batch * M_geo

  M_fast     = fast_mode_multiplier   if speed == "fast"
  M_longctx  = long_context_*_mult    if total_input > threshold AND NOT fast
  M_batch    = batch_multiplier       if service_tier == "batch"
  M_geo      = data_residency_mult    if inference_geo == "us"

  Note: Fast mode and long context are mutually exclusive.
  Fast mode covers the full context window with no additional charge.
Pricing multiplier stacking rules

Cache Write Breakdown

Anthropic reports cache writes at two TTL tiers: 5-minute (ephemeral) and 1-hour (extended). The pricing differs, so the calculator extracts both from usage.raw_usage:

pricing/calculator.py Extraction 
def _extract_cache_write_breakdown(usage: Usage) -> tuple[int, int]:
    """Return (cache_write_5m_tokens, cache_write_1h_tokens)."""
    raw = usage.raw_usage or {}
    cache_creation = raw.get("cache_creation")

    if isinstance(cache_creation, dict):
        return (
            cache_creation.get("ephemeral_5m_input_tokens", 0),
            cache_creation.get("ephemeral_1h_input_tokens", 0),
        )

    # Fallback: treat all cache_write_tokens as 5-minute TTL
    return (usage.cache_write_tokens or 0, 0)

Server Tool Costs

Web search and web fetch have per-request costs, not per-token. The calculator extracts request counts from usage.raw_usage["server_tool_use"]:

pricing/calculator.py Server Tools 
def _extract_server_tool_counts(usage: Usage) -> tuple[int, int]:
    """Return (web_search_requests, web_fetch_requests)."""
    stu = (usage.raw_usage or {}).get("server_tool_use")
    if isinstance(stu, dict):
        return (
            stu.get("web_search_requests", 0),
            stu.get("web_fetch_requests", 0),

Model Resolution

resolve_model_pricing() handles versioned model names like claude-sonnet-4-5-20250929 by trying an exact match first, then longest-substring matching against the CSV model IDs. This future-proofs the calculator against API model name changes without requiring CSV updates for every dated variant.

Serialization Round-Trip

The base LLMConfig provides to_dict() / from_dict() using dataclasses.asdict() and field-filtered construction. Storage adapters check AgentConfig.provider to know which subclass to reconstruct:

core/config.py — LLMConfig ABC 
@classmethod
def from_dict(cls, data: dict) -> LLMConfig:
    """Reconstruct, filtering to valid dataclass fields."""
    valid_fields = {f.name for f in dataclasses.fields(cls)}
    filtered = {k: v for k, v in data.items() if k in valid_fields}
    return cls(**filtered)

Unknown keys are silently ignored. This means adding a new field to AnthropicLLMConfig is backward-compatible — older serialized configs simply won't have the key, and the field gets its default value.

Chapter 8 — The Agent Loop
Orchestration

AnthropicAgent._resume_loop() Deep Dive

The agent loop is a while loop bounded by max_steps. Each iteration: compact, call the provider, process the stop reason. The loop supports both fresh starts (run()) and relay resumptions (resume_with_relay_results()).

  _resume_loop(queue?, stream_formatter?)
      |
      v
  while current_step < max_steps:
      |
      +-- compactor.apply_compaction(agent_config)
      |
      +-- provider.generate_stream() or provider.generate()
      |
      +-- current_step += 1
      +-- _accumulate_usage(response.usage)
      +-- append response to context + conversation
      |
      +-- match stop_reason:
      |     |
      |     +-- "pause_turn"  → continue  (skills, long-running)
      |     |
      |     +-- "tool_use"    → classify tool calls
      |     |     |
      |     |     +-- needs_relay?
      |     |     |   YES → execute backend, persist, return "relay"
      |     |     |   NO  → execute all, append results, continue
      |     |
      |     +-- "end_turn"    → final_answer_check? → _finalize_run()
      |
      +-- (loop continues)
      |
      v
  max_steps reached → _finalize_run("max_steps")
The complete agent loop flow

Initialization: Fresh vs Resume

The initialize() method handles two paths. For a fresh agent (no UUID), it creates one, builds an empty AgentConfig, sets up the sandbox, and attaches tools. For a resuming agent, it loads state from the config adapter, re-parses llm_config as AnthropicLLMConfig, and restores the partial conversation if a relay is pending.

anthropic_agent.py — initialize() Lifecycle 
async def initialize(self):
    if not self._agent_uuid:
        # Fresh agent: generate UUID, empty config
        self._agent_uuid = str(uuid.uuid4())
        self.agent_config = AgentConfig(agent_uuid=...)
    else:
        # Resuming: load from storage
        loaded = await self.config_adapter.load(self._agent_uuid)
        # Re-parse llm_config as AnthropicLLMConfig
        if not isinstance(loaded.llm_config, AnthropicLLMConfig):
            loaded.llm_config = AnthropicLLMConfig.from_dict(...)
        # Restore partial conversation if relay pending
        if loaded.pending_relay:
            self.conversation = await conv_adapter.load_by_run_id(...)

Parallel Tool Execution

Tool calls are executed in parallel, bounded by max_parallel_tool_calls (default 5). The ToolRegistry.execute_tools() method uses asyncio.Semaphore internally. Results come back as ToolResultEnvelope objects, which the agent converts to ToolResultContent blocks in a user-role Message.

Sub-Agent Injection

When the agent has subagents, the SubAgentTool needs the parent's streaming context (queue and formatter) to forward SSE events. The loop injects this at the start of _resume_loop() and clears it in the finally block:

anthropic_agent.py Context 
try:
    self._inject_subagent_context(queue, stream_formatter)
    while ...:
        ...
finally:
    # Always clear to avoid stale references
    self._inject_subagent_context(None, None)

Logging with LogEntry

Each significant event during the loop is recorded as a LogEntry in _run_logs. At run finalization, these are persisted via the AgentRunAdapter. The log captures LLM calls (with stripped binary data for readability), tool executions, errors, and timing.

Chapter 9 — LiteLLM, the Second Provider
Provider Breadth

LiteLLM — Proving the Boundary

A single concrete implementation is an anecdote; a second is a test of the abstraction. providers/litellm/ implements the same ABCs — Provider, MessageFormatter, Agent, CompactionController — against the LiteLLM SDK, which itself speaks a dozen backends through an OpenAI-compatible facade. The agent loop, tool system, sandbox, media backend, and storage adapters are reused unchanged.

What the second provider proved

No changes were required in core/, tools/, sandbox/, media_backend/, storage/, streaming/, or pricing/ to add LiteLLM. The only new code lives in providers/litellm/. That is exactly what the "axes of change" table in the hub promised.

The AnthropicLLMConfig / LiteLLMConfig Split

Both configs subclass LLMConfig and carry the parameters only their transport understands. AnthropicLLMConfig holds thinking_tokens, beta_headers, container_id. LiteLLMConfig holds routing parameters (provider override, fallback list, router key), timeout, and any OpenAI-shape passthrough. The agent loop never reads them directly — it hands them to the provider, which knows what to do.

Developer's inner voice "The fact that LiteLLMAgent ended up as a near-identical twin to AnthropicAgent is a good sign, not a bad one. The orchestration loop should be near-identical — it's the same architecture. The difference is in the four other files."

What About Native OpenAI?

LiteLLM can already proxy to OpenAI today, so the urgency of a native providers/openai/ is lower. A native implementation would unlock tighter integration with OpenAI-specific features (structured outputs, Assistants API) that LiteLLM's lowest-common-denominator surface cannot expose. It remains on the roadmap in planned-systems.

The provider is the translator, not the director.
It speaks wire format so the agent doesn't have to —
and the boundary between them is the architecture.

Three Adapters, One Pattern

Developer Diary · Storage System

Agent config, conversation history, and run logs — three persistence concerns that change at different rates, for different reasons, and deserve different backends. This is how we kept them independent.

Chapter 1 — The Problem
Three Concerns, Three Rhythms

Why One Database Table Isn't Enough

An AI agent persists three fundamentally different kinds of data:

  • Agent configuration — the session state. Model, system prompt, tools, media registry. Written once at creation, updated occasionally. One row per agent session.
  • Conversation history — the run record. User message, assistant response, usage metrics, cost. Written once per run, read back for the chat UI. One row per run, many runs per session.
  • Run logs — the execution trace. Step-by-step events: LLM call started, tool executed, error encountered. Written in bulk at the end of a run. Diagnostic data, rarely read.

These three concerns change at different rates, have different access patterns, and serve different consumers. Forcing them into a single persistence mechanism means the diagnostic log viewer dictates the schema for the chat UI, or the configuration store's consistency requirements slow down append-only log writes.

Developer's inner voice "In the old system, I had a single save_state() method that serialized everything into one JSON blob. Configuration, conversation, logs — all tangled. Want to list sessions? Parse every blob and extract the title. Want to paginate conversations? Load the entire session blob and slice in Python. Want to switch run logs to a different backend? Rewrite everything."
The Design Constraint

Each persistence concern must be independently swappable. You should be able to store configs in PostgreSQL, conversations on the filesystem, and run logs in memory — without any adapter knowing about the others. The adapter interfaces must be narrow enough that implementing a new backend is a single file.

Chapter 2 — Three Adapters
The Adapter ABCs

Separating What Changes from What Doesn't

The solution is three independent abstract base classes, each owning exactly one persistence concern. They share a common lifecycle interface (StorageAdapter) but define different domain-specific operations:

storage/base.py ABC 
class StorageAdapter(ABC):
    """Common lifecycle for all storage adapters."""
    async def connect(self) -> None: ...
    async def close(self) -> None: ...
    async def __aenter__(self): ...
    async def __aexit__(self, *args): ...

class AgentConfigAdapter(StorageAdapter):
    async def save(self, config: AgentConfig) -> None: ...
    async def load(self, agent_uuid: str) -> AgentConfig | None: ...
    async def delete(self, agent_uuid: str) -> bool: ...
    async def update_title(self, agent_uuid, title) -> bool: ...
    async def list_sessions(self, limit, offset) -> tuple: ...

class ConversationAdapter(StorageAdapter):
    async def save(self, conversation: Conversation) -> None: ...
    async def load_history(self, agent_uuid, limit, offset) -> list[Conversation]: ...
    async def load_cursor(self, agent_uuid, before, limit) -> tuple: ...

class AgentRunAdapter(StorageAdapter):
    async def save_logs(self, agent_uuid, run_id, logs: list[LogEntry]) -> None: ...
    async def load_logs(self, agent_uuid, run_id) -> list[LogEntry]: ...
AdapterDomain TypeAccess PatternCardinality
AgentConfigAdapter AgentConfig CRUD by UUID, list with pagination 1 per session
ConversationAdapter Conversation Append, paginate newest-first, cursor N per session
AgentRunAdapter LogEntry Batch write, load by run ID N entries per run
Developer's inner voice "Notice that the adapters accept and return typed domain objects — AgentConfig, Conversation, LogEntry. Not dicts. Not JSON strings. The adapter's job is to accept a domain object, persist it somehow, and give it back later. How it serializes internally is its own concern."

The Independence Property

The three adapters share no state. You can mix and match backends freely:

mixing backends Usage 
# Config in PostgreSQL (for querying), conversations on disk, logs in memory
config_adapter = PostgresAgentConfigAdapter(dsn)
conv_adapter = FilesystemConversationAdapter(base_path="./data")
run_adapter = MemoryAgentRunAdapter()

# Or all three from the same backend via factory
config, conv, run = create_adapters("filesystem", base_path="./data")
Chapter 3 — Serialization
Who Owns to_dict()?

Why Domain Objects Don't Serialize Themselves

In many codebases, domain objects own their serialization: config.to_dict() and AgentConfig.from_dict(data). We deliberately chose not to do this for AgentConfig and Conversation.

The reason: these objects contain deeply nested typed fields — Message lists, ToolSchema dataclasses, MediaMetadata registries, a polymorphic LLMConfig subclass. A generic to_dict() on the domain object would either need to know about all these nested types (violating SRP) or produce a shallow dict that's useless for persistence.

Developer's inner voice "The serialization concern belongs to the storage layer, not the domain layer. AgentConfig is a configuration object — it holds typed fields so the rest of the framework can use them safely. How those fields are flattened to JSON is a storage problem. If we add a field to AgentConfig tomorrow, the serialization module updates — but AgentConfig itself doesn't change."

The solution: a centralized storage/serialization.py module with pure functions that convert between typed domain objects and JSON-safe dicts.

storage/serialization.py Helpers 
def serialize_config(config: AgentConfig) -> dict[str, Any]:
    """Serialize an AgentConfig to a JSON-safe dict."""
    return {
        "agent_uuid": config.agent_uuid,
        "context_messages": [m.to_dict() for m in config.context_messages],
        "tool_schemas": [dataclasses.asdict(ts) for ts in config.tool_schemas],
        "llm_config": config.llm_config.to_dict(),  # polymorphic!
        "media_registry": {k: v.to_dict() for k, v in config.media_registry.items()},
        "subagent_schemas": [dataclasses.asdict(s) for s in config.subagent_schemas],
        # ... primitives pass through ...
    }

def deserialize_config(
    data: dict,
    llm_config_class: type[LLMConfig] = LLMConfig,
) -> AgentConfig:
    """Reconstruct typed objects from a dict."""
    return AgentConfig(
        agent_uuid=data["agent_uuid"],
        context_messages=[Message.from_dict(m) for m in data.get("context_messages", [])],
        tool_schemas=[ToolSchema(**ts) for ts in data.get("tool_schemas", [])],
        llm_config=llm_config_class.from_dict(data.get("llm_config", {})),
        # ... reconstruct other typed fields ...
    )

The LLMConfig Polymorphism Problem

When deserializing, how does the serialization layer know whether to construct a base LLMConfig or an AnthropicLLMConfig? It doesn't — and it shouldn't. The caller (the agent layer) knows which provider is in use and passes the correct subclass:

caller decides the LLMConfig subclass Pattern 
# Each LLMConfig subclass owns to_dict() / from_dict()
@dataclass
class LLMConfig:
    def to_dict(self) -> dict:
        return dataclasses.asdict(self)

    @classmethod
    def from_dict(cls, data: dict) -> "LLMConfig":
        valid = {f.name for f in dataclasses.fields(cls)}
        return cls(**{k: v for k, v in data.items() if k in valid})

# The agent layer passes the right subclass
config = deserialize_config(data, llm_config_class=AnthropicLLMConfig)
Developer's inner voice "No registry. No type discriminator in the JSON. No magic. The caller that creates the agent knows the provider. The caller that loads the agent from storage also knows the provider. It passes the right class. Simple, explicit, and impossible to get wrong at runtime."

What Owns Its Own Serialization

Not everything avoids to_dict(). The rule is: leaf types own their serialization; composite types don't.

TypeOwns to_dict/from_dict?Reason
Message Yes Self-contained — delegates to ContentBlock.to_dict() for each content block
ContentBlock Yes Polymorphic hierarchy with match-based dispatch
Usage Yes Simple flat dataclass, no nested typed fields
LLMConfig Yes Provider subclasses need polymorphic control
MediaMetadata Yes Simple flat dataclass
AgentConfig No Composite — contains Messages, ToolSchemas, LLMConfig, MediaMetadata, PendingToolRelay
Conversation No Composite — contains Messages, Usage, MediaMetadata, CostBreakdown
The Round-Trip Guarantee

deserialize_config(serialize_config(config)) must produce an equivalent object for all populated field combinations. This is verified by smoke tests that create a fully-populated AgentConfig, round-trip it, and compare every field.

Chapter 4 — Backends
Three Backends, Three Trade-offs

Memory, Filesystem, PostgreSQL

Each backend implements all three adapter ABCs. The choice depends on the deployment context:

BackendUse CasePersistenceConcurrency
memory Unit tests, stateless demos Process lifetime only Single-process safe
filesystem Local development, single-server Durable (JSON files) Single-process safe
postgres Production multi-server Durable (ACID) Multi-process safe

Memory Adapter

The simplest possible implementation. Python dicts with deepcopy to prevent mutation:

storage/adapters/memory.py Implementation 
class MemoryAgentConfigAdapter(AgentConfigAdapter):
    def __init__(self):
        self._data: dict[str, AgentConfig] = {}

    async def save(self, config: AgentConfig) -> None:
        self._data[config.agent_uuid] = deepcopy(config)

    async def load(self, agent_uuid: str) -> AgentConfig | None:
        config = self._data.get(agent_uuid)
        return deepcopy(config) if config else None
Developer's inner voice "The deepcopy is important. Without it, the caller could mutate the stored object through the reference. The memory adapter simulates the behavior of real persistence: what you save is decoupled from what you hold. Tests shouldn't pass because of accidental reference sharing."

Filesystem Adapter

JSON files organized in a directory hierarchy. Atomic writes via temp file + rename. Async I/O via aiofiles:

filesystem layout Structure 
data/
├── agent_config/
│   └── {agent_uuid}.json           # Full AgentConfig via serialize_config()
├── conversation_history/
│   └── {agent_uuid}/
│       ├── 001.json                # serialize_conversation() per run
│       ├── 002.json
│       └── index.json              # {"last_sequence": 2, ...}
└── agent_runs/
    └── {agent_uuid}/
        └── {run_id}.jsonl          # One serialize_log_entry() per line

Conversations use an index.json file to track sequence numbers, avoiding the need to scan the directory. Run logs use JSONL format — one JSON object per line — for efficient append and line-by-line parsing.

PostgreSQL Adapter

Connection pooling via asyncpg. Lazy pool initialization. JSONB-centric schema (see next chapter). Each adapter manages its own pool independently — or they can share a connection string and the pools will be created separately.

One File Per Backend

Adding a new backend (Redis, S3, DynamoDB) means creating one file in storage/adapters/ and implementing the three adapter ABCs. No changes to the factory, the serialization module, or any other adapter. The axis of change is isolated.

Chapter 5 — JSONB Schema
Simplifying PostgreSQL

Why We Ditched Fine-Grained Columns

The original approach mapped every AgentConfig field to its own PostgreSQL column — 37 columns including arrays for tool names, JSONB for media registries, and nullable timestamps. Every time we added a field to the domain model, we needed a schema migration.

Developer's inner voice "37 columns. Every new field on AgentConfig meant an ALTER TABLE, an INSERT update, a SELECT update, and a serialization mapping. Four places to change for adding one field. This is framework code — the schema should be as easy to evolve as a JSON file."

The solution: a JSONB-centric schema. Keep a handful of frequently-queried columns at the top level for indexing and filtering. Everything else goes in a single data JSONB column.

PostgreSQL schema — agent_config DDL 
CREATE TABLE agent_config (
    agent_uuid  TEXT PRIMARY KEY,
    provider    TEXT NOT NULL DEFAULT '',      -- top-level: filterable
    model       TEXT NOT NULL DEFAULT '',      -- top-level: filterable
    title       TEXT,                           -- top-level: display
    data        JSONB NOT NULL DEFAULT '{}',   -- everything else
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW(),
    last_run_at TIMESTAMPTZ,
    total_runs  INTEGER NOT NULL DEFAULT 0
);
PostgreSQL schema — conversation_history & agent_runs DDL 
CREATE TABLE conversation_history (
    agent_uuid      TEXT NOT NULL,
    run_id          TEXT NOT NULL,
    sequence_number SERIAL,                     -- auto-incrementing
    data            JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (agent_uuid, run_id)
);

CREATE TABLE agent_runs (
    agent_uuid TEXT NOT NULL,
    run_id     TEXT NOT NULL,
    data       JSONB NOT NULL DEFAULT '[]',   -- array of LogEntry dicts
    created_at TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (agent_uuid, run_id)
);
Developer's inner voice "Now adding a field to AgentConfig is free. The serialization module puts it in the JSONB blob. No schema migration. The top-level columns are the fields we actually query on: provider, model, title, timestamps. Everything else is stored in PostgreSQL but interpreted in Python."

How Save and Load Work

On save, the adapter calls serialize_config() to get a JSON-safe dict, extracts the top-level columns, and stores the rest in the data JSONB column via UPSERT. On load, it merges the top-level columns back into the JSONB data and calls deserialize_config():

  Save:   AgentConfig
            |
            v
        serialize_config()  -->  JSON-safe dict
            |
            +-- extract provider, model, title, timestamps  -->  SQL columns
            +-- full dict  -->  JSONB data column
            |
            v
        INSERT ... ON CONFLICT DO UPDATE

  Load:   SELECT agent_uuid, provider, model, title, data, ...
            |
            +-- data (JSONB)  -->  Python dict
            +-- merge top-level columns into dict
            |
            v
        deserialize_config(merged_dict)  -->  AgentConfig
JSONB-centric save and load flow
Why Duplicate Top-Level Columns?

The top-level columns seem redundant — the same data exists in the JSONB blob. But they exist for PostgreSQL's benefit: you can CREATE INDEX ON agent_config (provider), ORDER BY updated_at DESC, or WHERE total_runs > 10 without JSONB path expressions. The JSONB blob is the source of truth for the application; the columns are the source of truth for the database.

Chapter 6 — The Factory
One Call, Three Adapters

The Factory Pattern

Most callers want all three adapters from the same backend. The factory function makes this a one-liner:

storage/registry.py Factory 
def create_adapters(
    backend: AdapterType,
    **kwargs
) -> tuple[AgentConfigAdapter, ConversationAdapter, AgentRunAdapter]:
    """Create all three adapters for the given backend type."""
    return (
        create_agent_config_adapter(backend, **kwargs),
        create_conversation_adapter(backend, **kwargs),
        create_agent_run_adapter(backend, **kwargs),
    )

# Usage
config, conv, run = create_adapters("filesystem", base_path="./data")
config, conv, run = create_adapters("postgres", connection_string=dsn)
config, conv, run = create_adapters("memory")

Individual factory functions exist for when you need to mix backends:

individual factories API 
# Fine-grained control
config = create_agent_config_adapter("postgres", connection_string=dsn)
conv = create_conversation_adapter("filesystem", base_path="./data")
run = create_agent_run_adapter("memory")

# List what's available
available_adapter_types()  # ["memory", "filesystem", "postgres"]

The Architecture Test

Adding a new storage backend (e.g., Redis, DynamoDB) requires exactly one new file in storage/adapters/ implementing the three adapter ABCs, plus a registration entry in the registry. Zero changes to serialization.py, zero changes to base.py, zero changes to any other adapter. The axis of change is isolated.

Design DecisionTrade-offWhy We Chose It
Three independent adapters More ABCs to implement per backend Independent evolution and mixed backends outweigh the small implementation cost
Centralized serialization One more module to maintain Domain objects stay clean; serialization is tested in one place
JSONB-centric Postgres Can't query arbitrary nested fields via SQL Schema evolution is free; the fields we query are top-level columns
Typed domain objects in adapter interfaces Adapters must handle serialization Callers get type safety; serialization is the adapter's internal concern

Separate the concerns that change at different rates.

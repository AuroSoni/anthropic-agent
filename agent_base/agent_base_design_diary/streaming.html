The Streaming Pipeline

Developer Diary · Real-Time Event Architecture

The LLM speaks in fragments. The agent loop needs completed messages. The UI needs every token the instant it arrives. This is how we built a pipeline that serves both without coupling either to the other.

Chapter One
Chapter 1

The Problem: Two Consumers, One Stream

An LLM generates tokens one at a time. Two very different consumers need those tokens, and they need them in fundamentally incompatible ways.

The agent loop needs complete messages. It waits for the full response — text content, tool calls with parsed arguments, thinking blocks — before deciding what to do next. It operates in terms of turn-level semantics: “the model wants to call read_file with these arguments.”

The UI needs incremental updates. Every token should appear the moment it’s generated. The user is watching. Latency is felt viscerally. A two-second pause while the model “thinks” feels like a hang; streaming the thinking tokens makes it feel alive.

Design Tension If you give the agent loop the raw token stream, it has to reassemble complete messages itself. If you give the UI only completed messages, the user stares at a spinner for seconds at a time. You need both — and neither should know about the other.

There’s a third constraint: transport independence. The streaming system can’t assume SSE. It can’t assume WebSockets. It can’t even assume a network — a CLI consumer might just print to stdout. The canonical event model must be transport-agnostic. 

                        +-----------------+
                        |   LLM Provider  |
                        |  (Anthropic API)|
                        +--------+--------+
                                 |
                     raw SSE tokens / events
                                 |
                    +------------+------------+
                    |                         |
           +-------v--------+       +--------v-------+
           |   Agent Loop   |       |   UI Consumer  |
           | (needs complete |       | (needs every   |
           |  messages)      |       |  token NOW)    |
           +----------------+       +----------------+

    Two consumers. Incompatible timing requirements.
    One stream. Zero coupling.
The fundamental tension

The solution is a canonical event type — a provider-agnostic dataclass that represents every kind of streaming event — paired with a formatter abstraction that serializes those events for whatever transport layer is in use. The provider translates its native stream into canonical deltas. The formatter translates canonical deltas into wire format. Neither knows about the other.

StreamDelta — The Canonical Event

Every streaming event in agent_base flows through a single type hierarchy rooted at StreamDelta. It doesn’t matter whether the event originated from the Anthropic API, a tool execution, or the framework itself — by the time it reaches the transport layer, it’s a StreamDelta.

streaming/types.py
BASE
@dataclass
class StreamDelta:
    """Base class for all stream events.

    Attributes:
        agent_uuid: UUID of the agent emitting this delta.
        type: Event type string (auto-set by subclasses).
        is_final: Whether this is the last delta in its logical chunk.
    """

    agent_uuid: str
    type: str = ""
    is_final: bool = False

Three fields. That’s the contract every streaming event must satisfy:

  • agent_uuid — identifies which agent emitted the event, enabling multiplexing when sub-agents run concurrently
  • type — a string discriminator, auto-set by each subclass in __post_init__
  • is_final — signals the last chunk of a logical event (critical for chunked payloads)
Why agent_uuid? In a multi-agent system, the UI might be rendering output from an orchestrator and two sub-agents simultaneously. Without agent_uuid, the consumer can’t demultiplex — it can’t tell which agent’s text is arriving. Every delta carries its origin.

Seven concrete subtypes cover the full vocabulary of streaming events:

StreamFormatter — The Transport Adapter

A StreamDelta is a Python dataclass. The transport layer speaks JSON strings (for SSE), binary frames (for WebSockets), or plain text (for a CLI). Something has to bridge the gap. That something is the StreamFormatter.

streaming/base.py
ABC
StreamFormatterType = Literal["json"]


class StreamFormatter(ABC):
    """Abstract base class for stream formatters.

    Implementations serialize StreamDelta objects to a specific wire
    format (JSON envelopes, raw text, etc.) and put the result on a
    queue for the transport layer to consume.
    """

    @abstractmethod
    async def format_delta(
        self, delta: StreamDelta, queue: asyncio.Queue
    ) -> None:
        """Serialize a StreamDelta and put the result on the queue."""
        ...

The contract is minimal by design: one method, two arguments. The formatter receives a canonical delta and an output queue. It serializes the delta into whatever wire format it supports and puts the result on the queue. The transport layer drains the queue.

Design Decision Why pass the queue to format_delta() instead of returning a string? Because a single delta might produce multiple wire-format messages. A large tool result gets chunked into several envelopes. Returning a list would work, but the queue-based API naturally supports backpressure and keeps the async flow clean.

JsonStreamFormatter — The Default Implementation

The concrete implementation produces compact JSON envelopes — the wire format consumed by SSE endpoints and WebSocket handlers:

streaming/formatters.py
IMPL
class JsonStreamFormatter(StreamFormatter):
    """Stream formatter producing compact JSON envelopes.

    Each delta is serialized to one or more JSON envelope strings
    and put on the output queue. Large payloads (tool calls, tool
    results) are automatically chunked with UTF-8-safe splitting.
    """

    def __init__(self, stream_tool_results: bool = True) -> None:
        self.stream_tool_results = stream_tool_results

    async def format_delta(
        self, delta: StreamDelta, queue: asyncio.Queue
    ) -> None:
        agent = delta.agent_uuid

        if isinstance(delta, TextDelta):
            env = build_envelope("text", agent, delta.is_final, delta.text)
            await queue.put(env)

        elif isinstance(delta, ToolResultDelta):
            # Gate server tool results behind the flag.
            if delta.is_server_tool and not self.stream_tool_results:
                return
            # ... chunk_and_emit for large results

The formatter dispatches on the delta’s concrete type using isinstance checks. Simple deltas like TextDelta and ThinkingDelta produce a single envelope. Complex deltas like ToolCallDelta and ToolResultDelta are routed through chunk_and_emit() for automatic payload splitting.

The stream_tool_results Gate

Server-side tool results can be voluminous — code execution output, file contents, search results. The stream_tool_results flag lets the caller suppress these from the wire format entirely. The agent loop still sees them (it works with the raw StreamDelta objects), but the UI never receives the noise. Client tool results always stream through — the user initiated those and expects to see the output.

The JSON Envelope Format

Every wire-format message is a compact JSON object with a fixed shape:

streaming/utils.py — build_envelope()
SCHEMA
def build_envelope(
    msg_type: str,
    agent: str,
    final: bool,
    delta: str,
    **extra: Any,
) -> str:
    obj: dict[str, Any] = {
        "type":  msg_type,   # "text", "tool_call", "error", ...
        "agent": agent,      # agent UUID for demuxing
        "final": final,      # last chunk of this logical event?
        "delta": delta,      # the payload string
    }
    obj.update(extra)     # tool_id, tool_name, envelope_log, ...
    return json.dumps(obj, ensure_ascii=False, separators=(",", ":"))

A sample envelope on the wire looks like:

wire format examples
JSON

{"type":"text","agent":"a1b2c3","final":false,"delta":"Hello"}

// Tool call (might be chunked)
{"type":"tool_call","agent":"a1b2c3","final":true,

// Error
{"type":"error","agent":"a1b2c3","final":true,

Chunking and the UTF-8 Boundary Problem

Text tokens are small — a few bytes each. But tool results can be enormous. A read_file call might return 50 KB of source code. A code execution tool might produce pages of output. These payloads need to be broken into SSE-sized chunks.

The naive approach is simple: split the byte string every N bytes. But text is UTF-8, and UTF-8 is a variable-width encoding. A single character can span 1 to 4 bytes. If you split at an arbitrary byte boundary, you can land in the middle of a multi-byte character, producing invalid UTF-8 on both sides of the split.

The Danger Consider the string “Hello 🚀 World”. The rocket emoji is 4 bytes in UTF-8: F0 9F 9A 80. If your chunk boundary falls at byte offset 7, you split it into F0 9F and 9A 80. Neither fragment is valid UTF-8. The JSON serializer will choke. The client will see a replacement character. The user will see garbage.

The solution is UTF-8-safe splitting. When the split point lands on a continuation byte (a byte matching 0b10xxxxxx), back up until you reach a leading byte:

streaming/utils.py
UTF-8 SAFE SPLIT
def _utf8_safe_split(payload_bytes: bytes, max_bytes: int) -> list[bytes]:
    """Split payload_bytes into chunks of at most max_bytes each.

    Never splits inside a multi-byte UTF-8 character.
    """
    chunks: list[bytes] = []
    offset = 0
    length = len(payload_bytes)
    while offset < length:
        end = min(offset + max_bytes, length)
        if end < length:
            # Back up if we landed on a continuation byte (0b10xxxxxx)
            while end > offset and (payload_bytes[end] & 0xC0) == 0x80:
                end -= 1
        chunks.append(payload_bytes[offset:end])
        offset = end
    return chunks
How It Works UTF-8 continuation bytes always have the bit pattern 10xxxxxx (hex 0x80–0xBF). Leading bytes have patterns 0xxxxxxx (ASCII), 110xxxxx (2-byte), 1110xxxx (3-byte), or 11110xxx (4-byte). The bitmask (byte & 0xC0) == 0x80 precisely identifies continuation bytes. When we find one at the split point, we back up until we reach a leading byte — keeping the multi-byte character intact in the next chunk.

chunk_and_emit — The Orchestrator

The chunk_and_emit() function ties it all together. It measures the envelope overhead, calculates the maximum delta size per chunk, splits if necessary, and emits each chunk as a separate envelope:

streaming/utils.py
CHUNKING
MAX_SSE_CHUNK_BYTES = 2048

async def chunk_and_emit(
    queue: asyncio.Queue,
    msg_type: str,
    agent: str,
    payload: str,
    final_on_last: bool,
    **extra: Any,
) -> None:
    # Measure overhead (envelope with empty delta)
    overhead = len(build_envelope(msg_type, agent, False, "", **extra).encode("utf-8"))
    max_delta_bytes = MAX_SSE_CHUNK_BYTES - overhead
    if max_delta_bytes < 64:
        max_delta_bytes = MAX_SSE_CHUNK_BYTES  # degenerate case fallback

    payload_bytes = payload.encode("utf-8")

    if len(payload_bytes) <= max_delta_bytes:
        # Small payload — emit directly, no splitting
        env = build_envelope(msg_type, agent, final_on_last, payload, **extra)
        await queue.put(env)
        return

    # Large payload — split safely and emit chunks
    chunks = _utf8_safe_split(payload_bytes, max_delta_bytes)
    for i, chunk_bytes in enumerate(chunks):
        is_last = i == len(chunks) - 1
        delta_str = chunk_bytes.decode("utf-8")
        env = build_envelope(
            msg_type, agent,
            final=(final_on_last and is_last),  # only the last chunk is "final"
            delta=delta_str, **extra,
        )
        await queue.put(env)

  50 KB tool result payload
  ========================

  Step 1: Measure envelope overhead
    {"type":"tool_result","agent":"...","final":false,"delta":"","id":"...","name":"..."}
    ~120 bytes of overhead

  Step 2: max_delta_bytes = 2048 - 120 = 1928 bytes per chunk

  Step 3: Split payload at UTF-8-safe boundaries
    [chunk_0: 1928 bytes] [chunk_1: 1928 bytes] ... [chunk_N: remainder]

  Step 4: Emit envelopes
    {"type":"tool_result","final":false,"delta":"...chunk_0...","id":"tc_01","name":"read_file"}
    {"type":"tool_result","final":false,"delta":"...chunk_1...","id":"tc_01","name":"read_file"}
    ...
    {"type":"tool_result","final":true, "delta":"...chunk_N...","id":"tc_01","name":"read_file"}
                           ^^^^^^^^^
                      Only the last chunk carries final=true
Chunking a large tool result into SSE-safe envelopes
Why Measure Overhead Dynamically? The envelope overhead depends on the agent UUID length, the type string, and the extra fields (tool_id, tool_name). Rather than guessing, chunk_and_emit() builds an actual envelope with an empty delta, measures its byte length, and subtracts that from the chunk budget. This guarantees every emitted envelope fits within MAX_SSE_CHUNK_BYTES.

Wiring — The Full Pipeline

Let’s trace a token from the moment it leaves the LLM to the moment the user sees it. The pipeline has four stages, each decoupled from the others by a well-defined interface: 

  +-------------------+
  |   LLM Provider    |    Stage 1: Provider emits native events
  | (Anthropic API)   |    (content_block_delta, tool_use, etc.)
  +---------+---------+
            |
            | translate to canonical type
            v
  +-------------------+
  |    StreamDelta     |    Stage 2: Canonical event objects
  |  (TextDelta,       |    (provider-agnostic, typed, carry agent_uuid)
  |   ToolCallDelta,   |
  |   MetaDelta, ...)  |
  +---------+---------+
            |
            | format_delta()
            v
  +-------------------+
  |  StreamFormatter   |    Stage 3: Serialize to wire format
  | (JsonStreamFormatter|    (compact JSON envelopes, auto-chunked)
  |  or custom impl)   |
  +---------+---------+
            |
            | await queue.put(envelope_str)
            v
  +-------------------+
  |  asyncio.Queue     |    Stage 4: Async queue bridges to transport
  |                    |    (SSE endpoint, WebSocket handler, CLI printer)
  +---------+---------+
            |
            | transport drains queue
            v
  +-------------------+
  |  Transport Layer   |    Delivery
  | (SSE / WS / CLI)  |    (user sees each token as it arrives)
  +-------------------+
The four-stage streaming pipeline

Each boundary is a clean abstraction:

Boundary Interface Decouples
Provider → Delta StreamDelta subclass constructors Provider-specific event formats from the rest of the system
Delta → Formatter format_delta(delta, queue) Event semantics from serialization format
Formatter → Transport asyncio.Queue Serialization from delivery mechanism
Why This Matters Want to add a WebSocket transport? Write a consumer that drains the queue and sends each string as a WS frame. Want to support a new LLM provider? Translate its native events into StreamDelta subclasses. Want to change the wire format to Protocol Buffers? Write a new StreamFormatter. Each change is isolated to a single layer.

The emit_stream_delta Convenience Function

For cases where you have a StreamDelta and just want to get it onto the wire without constructing a formatter, utils.py provides a standalone dispatch function:

streaming/utils.py
CONVENIENCE
async def emit_stream_delta(
    queue: asyncio.Queue,
    delta: StreamDelta,
) -> None:
    """Dispatches on delta subclass and emits the appropriate
    JSON envelope(s) to the queue.

    Bridges StreamDelta objects to the wire-format emission layer.
    For large payloads it uses chunk_and_emit for automatic
    UTF-8-safe splitting.
    """
    agent = delta.agent_uuid

    if isinstance(delta, TextDelta):
        env = build_envelope("text", agent, delta.is_final, delta.text)
        await queue.put(env)

    elif isinstance(delta, ThinkingDelta):
        env = build_envelope("thinking", agent, delta.is_final, delta.thinking)
        await queue.put(env)

    # ... ToolCallDelta, ToolResultDelta, CitationDelta, MetaDelta, RollbackDelta, ErrorDelta
    # each dispatched to build_envelope or chunk_and_emit as appropriate

    else:
        logger.warning("unknown_stream_delta_type",
                       delta_type=type(delta).__name__,
                       agent_uuid=agent)

This function mirrors the logic in JsonStreamFormatter.format_delta() but lives outside the formatter class. It’s useful for framework-internal code that needs to emit deltas without going through the full formatter pipeline — for example, error handling paths that need to push an ErrorDelta directly onto the queue.

MetaDelta — Framework-Level Events

Not all streaming events come from the LLM. The framework itself has lifecycle events that the UI needs to know about: an agent starting up, a run completing, files being produced, the system waiting for a human relay. These are meta events, and they flow through the same streaming pipeline as LLM tokens.

streaming/types.py
META
@dataclass
class MetaDelta(StreamDelta):
    """Framework-level meta events (init, final, files, awaiting, etc.).

    The type is set by the caller (e.g. "meta_init", "meta_final").
    """

    payload: dict[str, Any] = field(default_factory=dict)

    # type is set by the caller — no __post_init__ override.
Design Decision MetaDelta is the only subtype that does not override __post_init__. Its type is set by the caller — "meta_init", "meta_final", "meta_files", "meta_awaiting_relay". This makes it an extensible escape hatch: any framework-level event can be piped through the stream without defining a new subclass.

The Meta Event Vocabulary

Type String When Emitted Typical Payload
"meta_init" Agent run begins Agent config, model name, run ID
"meta_final" Agent run completes Usage stats, total tokens, cost
"meta_files" Files produced during execution File paths, sizes, types
"meta_awaiting_relay" Agent is waiting for human input Prompt text, relay context
Unified Event Model The alternative would be a separate signaling channel for lifecycle events — a second queue, a callback system, or WebSocket control frames. But that means the UI has to listen on two channels and merge them into a single timeline. By piping everything through the same StreamDelta hierarchy and the same queue, the UI gets a single ordered stream of events. One consumer. One timeline. No merge logic.

Serialization Path

When the formatter encounters a MetaDelta, it JSON-serializes the payload dict and routes it through chunk_and_emit() like any other complex delta:

streaming/formatters.py
META HANDLING
elif isinstance(delta, MetaDelta):
    payload = json.dumps(
        delta.payload, ensure_ascii=False, separators=(",", ":")
    )
    await chunk_and_emit(
        queue, delta.type, agent, payload, final_on_last=delta.is_final
    )

The delta.type here is the caller-provided string — "meta_init", "meta_final", etc. — which becomes the "type" field in the JSON envelope. The UI can switch on this field to render lifecycle events differently from LLM content.

ErrorDelta — The Safety Valve

ErrorDelta always emits with final=True, regardless of the is_final field on the delta itself. This is intentional: an error is always the last event in its logical sequence. The UI can treat it as a terminal signal and display the error immediately without waiting for more chunks.

· · ·

The streaming pipeline is the nervous system of agent_base. It carries signals from the LLM provider, through canonical types, through transport-agnostic formatters, into whatever delivery mechanism the deployment requires. Each layer knows only its immediate neighbors. The provider doesn’t know about SSE. The formatter doesn’t know about Anthropic. The transport doesn’t know about tool calls. And the user sees every token the moment it’s born.

Building the Vocabulary

Developer Diary · Core Domain Model

Your tool receives an anthropic.types.ToolUseBlock. Now try switching to OpenAI. This is the story of building provider-agnostic types that let the framework speak one language while providers speak many.

Chapter 1 — The Problem
Provider Lock-in Through Types

When Your Types Are Someone Else's API

In the original anthropic_agent, provider types leaked everywhere. A tool handler received anthropic.types.ToolUseBlock. The message list was list[anthropic.types.MessageParam]. The streaming code parsed anthropic.types.RawContentBlockDeltaEvent. Every file in the codebase imported from anthropic.types.

This worked fine — until someone asked: "Can we support OpenAI too?"

Developer's inner voice "The problem isn't that we use Anthropic's types internally. The problem is that we expose them at every seam. Tools see them. Storage sees them. Compaction sees them. Streaming sees them. To add a second provider, I'd have to touch every file in the project. That's not coupling — that's entanglement."

The coupling was structural, not just nominal:

  • Tool code imported anthropic.types to read tool inputs and construct results
  • Storage code serialized anthropic.types.Message objects — whose shape is Anthropic-specific
  • Compaction code counted tokens using Anthropic's message format
  • Streaming code parsed Anthropic-specific SSE events

None of these modules should know which LLM provider is being used. A tool that reads a file doesn't care whether Claude or GPT-4 asked for it. A compactor that trims old messages doesn't care what wire format those messages originally arrived in.

💡 The Core Question

What vocabulary does the framework actually need? Not Anthropic's vocabulary, not OpenAI's. A canonical vocabulary that captures the semantics of multi-modal, tool-using conversations — without tying to any provider's wire format.

Chapter 2 — The Vocabulary
The ContentBlock Hierarchy

Defining What Content Can Be

The content model needed to answer one question: what kinds of things can appear in a conversation turn? Not "what does Anthropic support" or "what does OpenAI support" — but what are the semantically meaningful content types the framework understands?

core/types.py Type Hierarchy 
ContentBlock (ABC, dataclass)
├── TextContent                          # text
├── ThinkingContent                      # thinking, signature
├── MediaContent                         # media_type, media_id (required)
│   ├── ImageContent                     # source_type, data (transient)
│   ├── DocumentContent                  # source_type, data (transient)
│   └── AttachmentContent                # filename, source_type
├── ToolUseBase                          # tool_name, tool_id, tool_input
│   ├── ToolUseContent                   # client-side tool invocation
│   ├── ServerToolUseContent             # server-side (provider-managed)
│   └── MCPToolUseContent                # MCP protocol (+ mcp_server_name)
├── ToolResultBase                       # tool_name, tool_id, tool_result, is_error
│   ├── ToolResultContent                # client-side result
│   ├── ServerToolResultContent          # server-side result
│   └── MCPToolResultContent             # MCP result (+ mcp_server_name)
├── CitationBase                         # cited_text
│   ├── CharCitation / PageCitation
│   ├── ContentBlockCitation
│   ├── SearchResultCitation
│   └── WebSearchResultCitation
└── ErrorContent                         # error_message, error_type
Developer's inner voice "This isn't the LCD of all providers, and it's not the superset. It's the set of semantically meaningful types the framework needs to reason about. A thinking block matters because compaction treats it differently. A media block matters because it needs resolution. A tool result matters because it has dual projections. If a type doesn't influence framework behavior, it doesn't need to be in the hierarchy."

The Serialization Contract

Every content block must round-trip through JSON. The base class defines the contract:

core/types.py — ContentBlock base Contract 
@dataclass
class ContentBlock(ABC):
    """Base for all content types. raw is transient, kwargs are persisted."""
    content_block_type: ContentBlockType = field(init=False)
    kwargs: Dict[str, Any] = field(default_factory=dict)
    raw: Any = field(default=None, repr=False)  # ← never serialized

    @abstractmethod
    def to_dict(self) -> Dict[str, Any]:
        """JSON-native primitives only. Enums emit .value strings."""
        ...

    @classmethod
    def from_dict(cls, data: Dict) -> "ContentBlock":
        """Match-based polymorphic dispatcher."""
        block_type = data.get("content_block_type")
        match block_type:
            case ContentBlockType.TEXT.value:
                return _build_dataclass_block(TextContent, data)
            # ... other cases ...

Two key design choices:

  • raw is transient. The original provider SDK object is stored for debugging but never serialized. This prevents provider types from leaking into persistence.
  • kwargs is the escape hatch. Provider-specific fields that don't map to canonical fields go into kwargs. They're persisted but not interpreted by the framework. This lets providers preserve information without polluting the canonical model.

Media Content: Identity vs Data

Media blocks encode a critical separation: identity (media_id) is required and persisted; data (base64, bytes) is transient and resolved on demand.

Developer's inner voice "The data field on ImageContent is marked transient — never serialized. When you save a conversation, you save media_id: 'abc123', not 2MB of base64. When you reload, the media backend resolves the ID to bytes on demand. This keeps persisted conversations lightweight and makes backend swaps trivial."
Chapter 3 — Messages & Usage
The Conversation Envelope

Wrapping Content in Context

A Message is a single turn in a conversation — one or more ContentBlocks, a role, optional usage metrics, and provider/model identity.

core/messages.py Dataclass 
@dataclass
class Message:
    id: str = field(default_factory=lambda: str(uuid.uuid4()))
    role: Role = Role.USER
    content: List[ContentBlock] = field(default_factory=list)
    stop_reason: Optional[str] = None
    usage: Optional[Usage] = None
    provider: str = ""
    model: str = ""

The MessageFormatter ABC sits at the translation boundary:

core/messages.py ABC 
class MessageFormatter(ABC):
    @abstractmethod
    def format_messages(self, messages, params) -> Dict:
        """Canonical messages → provider wire format."""
        ...

    @abstractmethod
    def parse_response(self, raw_response) -> Message:
        """Provider wire format → canonical message."""
        ...
Developer's inner voice "The formatter is the gateway. Canonical types flow in, provider types flow out, and vice versa. The formatter is the only place where provider SDK types are imported. If I grep for import anthropic in the entire codebase, it should only hit files inside providers/anthropic/."
Chapter 4 — The Agent Contract
Separating Orchestration from Transport

Agent, Provider, Formatter — Three Concerns

In the original anthropic_agent, a single class did everything: manage the agentic loop, make SDK API calls, parse responses, and format messages. The refactoring split this into three classes, each owning one concern:

core/agent.py + core/provider.py ABC 
class Agent(ABC):
    """Owns orchestration: step counting, tool dispatch, compaction."""
    @abstractmethod
    async def run(self, prompt: str | Message) -> AgentResult: ...
    @abstractmethod
    async def resume_run_with_tool_results(self, results) -> AgentResult: ...

class Provider(ABC):
    """Owns transport: HTTP calls, retry, streaming, response parsing."""
    @abstractmethod
    async def generate(self, system_prompt, messages, tool_schemas,
                       llm_config, model, ...) -> Message: ...
    @abstractmethod
    async def generate_stream(self, ..., queue, stream_formatter) -> Message: ...
ClassConcernProvider SDK imports?
Agent Orchestration — step counting, tool dispatch, compaction No
Provider Transport — HTTP, retry, streaming, accumulation Yes (in concrete subclass)
MessageFormatter Translation — canonical types ↔ wire format Yes (in concrete subclass)
Developer's inner voice "The Agent calls self.provider.generate() — it never imports a provider SDK directly. The Provider calls self.formatter.format_messages() then self.formatter.parse_response(). Each class does one job. To add OpenAI, I write three files: OpenAIAgent, OpenAIProvider, OpenAIMessageFormatter. Zero changes to core/."
Chapter 5 — Provider Mapping
Where Translation Lives

The Formatter as the Gateway

Each provider directory contains exactly four files — each owning one concern:

Developer's inner voice "The formatter does media resolution too. When formatting messages for the LLM, it encounters a MediaContent block with just a media_id. It calls media_backend.to_base64(media_id) to get the actual bytes. This is lazy resolution at the boundary — the content block carries an identity reference, and the formatter resolves it right before the API call."
Chapter 6 — Connecting the Dots
Architectural Principles

What This Design Buys Us

🔑 Dependency Inversion

High-level modules (tools, compaction, storage) depend on abstractions (ContentBlock, Message, Agent), not on concrete provider types. The dependency arrow points inward toward the canonical model. Providers depend on the core; the core never depends on providers.

✅ Liskov Substitution

Any Provider subclass can be swapped in wherever a Provider is expected. AnthropicProvider and a future OpenAIProvider both accept canonical Message lists and return canonical Message objects. The agent loop doesn't know or care which one it's talking to.

💡 The Grep Test

A well-factored codebase passes the grep test. grep -r "import anthropic" agent_base/ should only match files inside providers/anthropic/. grep -r "from anthropic" agent_base/core/ should return zero results. grep -r "from anthropic" agent_base/tools/ should return zero results. If any of these fail, provider types have leaked into the framework core.

PrincipleHow It Applied
Dependency Inversion Tools, storage, compaction depend on canonical types in core/, not on provider SDKs. The dependency direction is always inward.
Liskov Substitution Any Provider can replace any other Provider. The contract is generate(canonical) → canonical. The agent loop never breaks.
Open/Closed Adding a new provider means adding a new directory with four files. Zero modifications to core/, tools/, storage/, or any existing provider.
Single Responsibility Agent owns orchestration. Provider owns transport. Formatter owns translation. Three concerns, three classes, three files.
Lazy Resolution Content blocks carry identity references (media_id), not data. Resolution happens at the consumer boundary — the formatter, the streaming layer, the persistence layer. Never at creation time.
· · ·

The canonical model is the fulcrum of the entire architecture. Everything above it (tools, compaction, storage, streaming) speaks one language. Everything below it (Anthropic, OpenAI, LiteLLM) speaks its own language. The formatter translates between them. Add a new language by adding a translator. The core never changes.

Chapter 7 — Session State
AgentConfig & LLMConfig

Everything the Agent Needs to Resume

An agent session isn't just a model name. It's a system prompt, a conversation history, tool schemas, media registries, provider-specific parameters, and tracking state. AgentConfig is the single dataclass that holds all of this:

core/config.py — AgentConfig Dataclass 
@dataclass
class AgentConfig:
    # Identity
    agent_uuid: str = field(default_factory=lambda: str(uuid4()))
    description: str | None = None
    provider: str = ""
    model: str = ""
    max_steps: int = 50
    system_prompt: str | None = None

    # LLM context
    context_messages: list[Message] = field(default_factory=list)
    conversation_history: list[Message] = field(default_factory=list)

    # Tools
    tool_schemas: list[ToolSchema] = field(default_factory=list)
    tool_names: list[str] = field(default_factory=list)

    # Provider config
    llm_config: LLMConfig = field(default_factory=LLMConfig)

    # Media
    media_registry: dict[str, MediaMetadata] = field(default_factory=dict)

    # Relay (frontend/confirmation tool calls)
    pending_relay: PendingToolRelay | None = None

    # Hierarchy
    parent_agent_uuid: str | None = None
    subagent_schemas: list[SubAgentSchema] = field(default_factory=list)

    # Timestamps & tracking
    title: str | None = None
    created_at: str | None = None
    total_runs: int = 0
    extras: dict[str, Any] = field(default_factory=dict)
Developer's inner voice "Every highlighted field used to be dict or list[dict] in the old anthropic_agent. A tool schema was a dict you hoped had the right keys. An LLM config was a dict that might or might not contain thinking_tokens. Now the types enforce structure at the Python level. You can't accidentally pass a media registry entry without a media_type field — the MediaMetadata dataclass won't let you."

LLMConfig: Polymorphic Provider Parameters

Different providers need different parameters. Anthropic needs thinking_tokens and beta_headers. OpenAI needs frequency_penalty and response_format. The base LLMConfig holds common fields; provider subclasses add their own:

core/config.py — LLMConfig Base 
@dataclass
class LLMConfig:
    """Base LLM configuration. Provider subclasses extend this."""
    temperature: float | None = None
    max_tokens: int = 16384
    top_p: float | None = None
    stop_sequences: list[str] | None = None

    def to_dict(self) -> dict[str, Any]:
        return dataclasses.asdict(self)

    @classmethod
    def from_dict(cls, data: dict) -> "LLMConfig":
        valid = {f.name for f in dataclasses.fields(cls)}
        return cls(**{k: v for k, v in data.items() if k in valid})
providers/anthropic/config.py Subclass 
@dataclass
class AnthropicLLMConfig(LLMConfig):
    """Anthropic-specific parameters."""
    thinking_budget_tokens: int | None = None
    beta_headers: list[str] | None = None
    container_id: str | None = None
    # Inherits to_dict() / from_dict() from LLMConfig
    # The field-filtering from_dict handles extra fields automatically
Developer's inner voice "The from_dict method filters to only known fields before constructing. This means you can safely call AnthropicLLMConfig.from_dict(data) even if the dict has extra keys — they're silently ignored. And since each subclass inherits the base implementation, adding a new provider config is just adding a new dataclass with the right fields."
No Registry, No Discriminator

How does the system know to deserialize as AnthropicLLMConfig vs base LLMConfig? It doesn't use a type discriminator in the JSON. The caller (agent layer) knows the provider and passes the correct class: deserialize_config(data, llm_config_class=AnthropicLLMConfig). Simple, explicit, impossible to misroute.

Chapter 8 — Run Records
Conversation, Usage & Cost

Recording What Happened in a Run

Every time the agent runs (one user message → one final response, possibly with multiple tool-use steps), the result is captured in a Conversation record. This is distinct from AgentConfig.conversation_history (which holds the live LLM context). A Conversation is a complete, immutable snapshot of one run:

core/config.py — Conversation Dataclass 
@dataclass
class Conversation:
    agent_uuid: str
    run_id: str = field(default_factory=lambda: str(uuid4()))

    # Timing
    started_at: str | None = None
    completed_at: str | None = None

    # The conversation
    user_message: Message | None = None
    final_response: Message | None = None
    messages: list[Message] = field(default_factory=list)  # full turn log

    # Outcome
    stop_reason: str | None = None
    total_steps: int | None = None

    # Metrics
    usage: Usage = field(default_factory=Usage)
    generated_files: list[MediaMetadata] = field(default_factory=list)
    cost: CostBreakdown | None = None

    # Ordering
    sequence_number: int | None = None
    extras: dict[str, Any] = field(default_factory=dict)

Usage: Token Accounting

Usage tracks cumulative token consumption across all LLM calls in a run. It supports addition for aggregating across steps:

core/messages.py — Usage Dataclass 
@dataclass
class Usage:
    input_tokens: int = 0
    output_tokens: int = 0
    cache_creation_input_tokens: int = 0
    cache_read_input_tokens: int = 0

    def __add__(self, other: "Usage") -> "Usage":
        return Usage(
            input_tokens=self.input_tokens + other.input_tokens,
            output_tokens=self.output_tokens + other.output_tokens,
            # ... same for cache fields
        )

    def to_dict(self) -> dict: ...
    @classmethod
    def from_dict(cls, data) -> "Usage": ...

CostBreakdown: Dollar Amounts

While Usage counts tokens, CostBreakdown translates them to dollars. Separate fields for input vs output pricing, with a computed total:

core/config.py — CostBreakdown Dataclass 
@dataclass
class CostBreakdown:
    input_cost: float = 0.0
    output_cost: float = 0.0
    total_cost: float = 0.0
    currency: str = "USD"
Developer's inner voice "Cost is separate from usage because the pricing logic is provider-specific. The agent computes cost after each run using the provider's pricing table and attaches it to the Conversation. The Conversation itself doesn't know how to calculate cost — it just stores the result."
Chapter 9 — Supporting Cast
PendingToolRelay, SubAgentSchema, LogEntry

The Typed Helpers

Several supporting dataclasses replace what were previously raw dicts. Each enforces structure at the Python level so that bugs surface at construction time, not at serialization time.

PendingToolRelay

When the agent encounters tool calls that need frontend interaction or user confirmation, the relay state holds them between runs:

core/config.py — PendingToolRelay Dataclass 
@dataclass
class PendingToolRelay:
    frontend_calls: list[ToolCallInfo] = field(default_factory=list)
    confirmation_calls: list[ToolCallInfo] = field(default_factory=list)
    completed_results: list[Message] = field(default_factory=list)

ToolCallInfo captures the tool name, ID, and input for each pending call. completed_results holds results that have already been resolved (e.g., the user approved a confirmation) but haven't yet been fed back to the LLM.

SubAgentSchema

When an agent can spawn sub-agents, each sub-agent's configuration template is described by a schema:

core/config.py — SubAgentSchema Dataclass 
@dataclass
class SubAgentSchema:
    name: str
    description: str = ""
    model: str | None = None
    system_prompt: str | None = None
    tool_names: list[str] = field(default_factory=list)
    max_steps: int = 25

LogEntry & AgentResult

LogEntry captures a single event during agent execution. AgentResult wraps the final outcome of a run:

core/result.py Dataclasses 
@dataclass
class LogEntry:
    step: int
    event_type: str                          # "llm_call", "tool_exec", "error", ...
    timestamp: str
    message: str = ""
    duration_ms: float | None = None
    usage: Usage | None = None             # per-step token usage
    extras: dict[str, Any] = field(default_factory=dict)

@dataclass
class AgentResult:
    config: AgentConfig                    # updated session state
    conversation: Conversation              # the run record
    run_log: AgentRunLog                    # execution trace
    stop_reason: str = "end_turn"           # "end_turn", "relay", "max_steps"
    cost: CostBreakdown | None = None      # cumulative USD cost
    agent_logs: list[LogEntry] = field(default_factory=list)
Developer's inner voice "Every AgentResult carries three things: the updated config (with new conversation_history), the conversation record (for the chat UI), and the run log (for diagnostics). Three concerns, three objects, all typed. The storage layer saves each one through its respective adapter."

The Typing Guarantee

Every field that was dict or list[dict] in the original anthropic_agent is now a typed dataclass or a list of typed dataclasses. Type errors surface at construction time. Missing fields surface at construction time. The serialization layer converts between typed Python and untyped JSON — but everywhere else in the framework, the types are enforced.

Define the vocabulary first. Let the vocabulary define the architecture.
A canonical type system isn't a luxury — it's the foundation
that makes everything else swappable.

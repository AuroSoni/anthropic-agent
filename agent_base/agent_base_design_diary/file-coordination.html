When Ephemeral Meets Durable

Developer Diary · File Coordination

We built a 640-line state machine to move files in and out. Then we deleted it. This is the story of recognizing over-engineering and finding where responsibilities actually belong.

Chapter 1 — The Problem
Two Worlds, One Agent

Files Need to Cross the Boundary

The agent lives in two worlds at once. Its sandbox is an ephemeral workspace — files created during a session, destroyed at teardown. Its media backend is durable storage — files that survive across sessions, uploaded by users, persisted for later retrieval.

These worlds need to talk to each other. When a user uploads a PDF and asks the agent to analyze it, that PDF needs to get from the media backend into the sandbox where tools can read it. When the agent generates a report, that report needs to get from the sandbox into the media backend where the user can download it later.

Developer's inner voice "So we need to move files between two systems. That's a bridge pattern, right? One class that knows about both sides and moves bytes across the boundary. How hard can it be?"

The requirements seemed simple:

  • Inbound: User uploads → media backend → sandbox (so tools can read them)
  • Outbound: Tools write files → sandbox → media backend (so users can download them)
  • Idempotent: Materializing the same file twice shouldn't duplicate anything
  • Session-scoped: Sandbox dies at teardown; durable storage survives

Two verbs. Move in, move out. Simple enough. Or so we thought.

Chapter 2 — First Attempt
The Bridge Pattern

SandboxFileBridge & FileManifest

The first design introduced two new classes: a SandboxFileBridge to orchestrate file movement, and a FileManifest to track what's been moved.

sandbox/bridge.py — first attempt Deleted 
class SandboxFileBridge:
    """Moves files between the storage backend and the sandbox."""

    def __init__(self,
        sandbox: Sandbox,
        media_backend: MediaBackend,
        manifest: FileManifest,
    ):
        self._sandbox = sandbox
        self._backend = media_backend
        self._manifest = manifest

    async def materialize(self, media_id: str, agent_uuid: str) -> str:
        """Download from backend, place in sandbox, track in manifest."""
        record = self._manifest.get(media_id)
        if record and record.lifecycle == FileLifecycle.IMPORTED:
            return record.sandbox_path  # already materialized

        metadata = await self._backend.get_metadata(media_id, agent_uuid)
        content = await self._backend.retrieve(media_id, agent_uuid)
        path = f".imported/{media_id}/{metadata.media_filename}"
        await self._sandbox.write_file_bytes(path, content)

        self._manifest.register(media_id, path, FileLifecycle.IMPORTED)
        return path

    async def persist_exports(self, agent_uuid: str) -> list[MediaMetadata]:
        """Scan exports zone, store each file, track in manifest."""
        results = []
        for record in self._manifest.by_lifecycle(FileLifecycle.EXPORTED):
            if not record.is_dirty:
                continue
            content = await self._sandbox.read_file_bytes(record.sandbox_path)
            meta = await self._backend.store(content, ...)
            record.mark_persisted(meta.media_id)
            results.append(meta)
        return results

The bridge needed to track state — what's been imported, what's been exported, what's been modified since last persist. That tracking lived in the FileManifest:

class FileLifecycle(Enum):

Both classes had comprehensive test coverage: 55 tests for the bridge, 31 for the manifest. They worked correctly. The state machine was deterministic. Lifecycle transitions were validated.

Developer's inner voice "This feels thorough. Five lifecycle states, dirty tracking, a manifest that survives across operations. But... something's nagging me. The tests pass, the design is clean, but I keep looking at it and thinking: is this really what two verbs need?"
Chapter 3 — The Smell
Recognizing Over-Engineering

640 Lines for "Move Bytes In, Move Bytes Out"

Let's count what we built:

FileManifest ~162 31 Track what's been moved and its state
FileLifecycle — — 5-state enum for file transitions
FileRecord — — Per-file tracking dataclass
Total ~642 86 —

642 lines of code. 86 tests. Two new classes. One enum with five values. One dataclass. All for two verbs: move in and move out.

Developer's inner voice "Here's what's really bothering me: the bridge is a third party that knows about both the sandbox and the media backend. It's an intermediary. But the sandbox already knows how to write files, and the media backend already knows how to store files. Why does a third class need to exist just to call methods on both?"

The code smells started to crystallize:

  • Artificial indirection. The bridge just calls backend.retrieve() then sandbox.write_file_bytes(). It's a secretary forwarding phone calls.
  • Zone leakage. The bridge constructs paths like .imported/{media_id}/{filename}. It knows about the sandbox's internal layout. That's the sandbox's concern.
  • Phantom state. The manifest tracks lifecycle transitions that the filesystem already knows. Is the file there? Check the filesystem. Is it dirty? That's an application concern, not a file-tracking concern.
  • Three-way coupling. The bridge depends on Sandbox, MediaBackend, and FileManifest. Changing any of the three ripples into the bridge.
⚠️ The Smell Test

If you can describe what a class does in under 10 words — "moves files between sandbox and backend" — and it takes 480 lines to implement, something is wrong. Either the description is hiding complexity, or the implementation is creating it.

Chapter 4 — The Insight
Finding the Natural Owners

Responsibilities Belong to Their Natural Owners

The bridge existed because we assumed file coordination was a third concern — something that lives between the sandbox and the media backend. But look at the responsibilities more carefully:

Bridge does this:

  • Write file to .imported/ zone
  • Track that it was imported
  • Scan .exports/ zone
  • Read exported file content
  • Construct zone paths

Who should own it:

  • Sandbox — it owns the filesystem
  • Sandbox — it knows what's been written
  • Sandbox — it owns the directory
  • Sandbox — it reads files
  • Sandbox — it owns zone layout

Bridge does this:

  • Get metadata from backend
  • Retrieve bytes from backend
  • Store exported bytes in backend
  • Infer MIME types

Who should own it:

  • MediaBackend — its own API
  • MediaBackend — its own API
  • MediaBackend — its own API
  • MediaBackend — it does this for store()
Developer's inner voice "Every single thing the bridge does is already someone else's job. The sandbox owns the filesystem. The media backend owns storage. The bridge is just glue — and glue shouldn't have 480 lines and its own state machine."

The fix: give each system its half of the coordination.

  • Sandbox gets a small cluster of semantic abstract methods: import_file() on the ingress side, and list_exported_files() / get_exported_file() / get_exported_file_metadata() on the egress side. They say what to do, not how to lay out zones.
  • MediaBackend gets three concrete methods: attach_sandbox(), materialize(), and flush_exports(). These compose abstract storage operations with sandbox coordination.
  • The bridge class, the manifest, the lifecycle enum, the file record — all deleted. Zero new classes.
Chapter 5 — The Simplification
The Final Design

Two Methods on Sandbox, Three on MediaBackend

Sandbox: Own Your Zones

The sandbox already owns the filesystem. Now it also owns the concept of importing and exporting files. The zone layout (.imported/, workspace/, .exports/) is an internal detail — callers get semantic methods.

sandbox/sandbox_types.py ABC 
# Added to the Sandbox ABC

@abstractmethod

                      data: AsyncIterator[bytes]) -> str:
    """Accept a file for tool use. Streams bytes in — no full buffering.
    Idempotent per (sandbox, filename). Returns the sandbox-relative path.
    Callers must not assume path structure."""
    ...

# Egress is split into three methods so callers can stream content
# independently of listing / metadata.

@abstractmethod
async def list_exported_files(self) -> list[str]: ...

@abstractmethod
async def get_exported_file(self, path: str)
    -> AsyncIterator[bytes]: ...

@abstractmethod
async def get_exported_file_metadata(self)
    -> list[ExportedFileMetadata]: ...

Two design choices worth calling out. First, import and export speak streams, not buffers — an AsyncIterator[bytes] on the way in, another one on the way out — so a 100 MB image never materialises as a single bytes blob in memory. Second, egress is deliberately split into three calls: a cheap name list, a streamed content fetch, and a metadata bundle (size, BLAKE3 hash, extension). This lets flush_exports() decide which files to persist before committing to transferring their bytes.

Idempotency is still handled by the filesystem: import_file() checks if the path exists before writing. No manifest needed. No lifecycle enum. The filesystem is the state tracker.

sandbox/local.py — implementation Implementation 
async def import_file(self, filename: str,
                      data: AsyncIterator[bytes]) -> str:
    rel = f".imported/{filename}"
    target = self._resolve(rel)       # path containment check

    if target.exists():                # idempotent — already imported
        return rel

    target.parent.mkdir(parents=True, exist_ok=True)
    async with aiofiles.open(target, "wb") as f:
        async for chunk in data:
            await f.write(chunk)
    return rel

async def list_exported_files(self) -> list[str]:
    exports_dir = self.root / ".exports"
    if not exports_dir.exists():
        return []
    return [str(p.relative_to(exports_dir))
            for p in sorted(exports_dir.rglob("*"))
            if p.is_file()]

MediaBackend: Compose, Don't Bridge

The media backend already has store(), retrieve(), and get_metadata(). Three new concrete methods compose those with sandbox operations. Concrete — not abstract — because every backend inherits the same orchestration logic. Only storage is polymorphic; coordination isn't.

media_backend/types.py Concrete 
# Concrete methods on MediaBackend ABC — not abstract

_sandbox: Sandbox | None = None

def attach_sandbox(self, sandbox: Sandbox) -> None:
    """Store reference to active sandbox."""
    self._sandbox = sandbox

async def materialize(self, media_id: str, agent_uuid: str) -> str:
    """Retrieve from storage, import into sandbox. Idempotent."""
    metadata = await self.get_metadata(media_id, agent_uuid)
    stream   = self.retrieve(media_id, agent_uuid)   # AsyncIterator[bytes]
    return await self._sandbox.import_file(
        metadata.media_filename, stream)

async def flush_exports(self, agent_uuid: str) -> list[MediaMetadata]:
    """Collect exported files, store each in backend."""
    results = []
    for info in await self._sandbox.get_exported_file_metadata():
        stream = self._sandbox.get_exported_file(info.path)   # streamed
        mime   = mimetypes.guess_type(info.filename)[0] \
                 or "application/octet-stream"
        meta = await self.store(stream, info.filename, mime, agent_uuid)
        if info.path != info.filename:
            meta.extras["export_path"] = info.path
        results.append(meta)
    return results
Developer's inner voice "Notice the TYPE_CHECKING import. MediaBackend imports Sandbox only for type hints, never at runtime. No circular dependency. No runtime coupling between modules. Just a type annotation that says 'I can work with any Sandbox.'"

Zone Encapsulation

The critical invariant: MediaBackend never knows about zone paths. It calls sandbox.import_file(name, stream) and gets back an opaque sandbox-relative path. It asks for list_exported_files() / get_exported_file_metadata() to decide what to persist, and streams each file back through get_exported_file(). A DockerSandbox could organize files completely differently from LocalSandbox — the media backend wouldn't notice.

    ┌─────────────────────┐         ┌─────────────────────────────────┐

    │                     │         │                                 │
    │  store()            │         │   .imported/{filename}          │
    │  retrieve()    ─────┼── attach_sandbox() ──►   workspace/      │
    │  get_metadata()     │         │   .exports/...                  │
    │                     │         │                                 │
    │  materialize() ─────┼── import_file(name, stream) ──►          │
    │                     │         │                                 │
    │  flush_exports()  ──┼── list_exported_files() ──────►          │
    │                  ◄──┼── get_exported_file_metadata() ─         │

Chapter 6 — What We Learned
Principles

The Best Abstraction Is the One You Delete

💡 YAGNI Applied to Abstractions

The FileManifest tracked five lifecycle states. In practice, only two mattered: "is the file in the sandbox?" and "are there files in exports?" The filesystem already answers both questions. The manifest was duplicating the filesystem's state in a data structure — and then we had to keep them in sync. Every sync point is a bug waiting to happen.

🔑 The Intermediary Test

When you design a class, ask: "Does this class have unique knowledge, or is it just forwarding calls?" The SandboxFileBridge had no unique knowledge. Everything it did was: call backend method, pass result to sandbox method. The information it tracked (the manifest) was redundant with the filesystem. A class that holds no unique knowledge and does no unique computation is an intermediary. Delete it, and give each real owner its share of the work.

✅ Interface Segregation in Practice

Rather than one fat bridge interface, the final design puts two focused methods on Sandbox (import_file, get_exported_files) and three focused methods on MediaBackend (attach_sandbox, materialize, flush_exports). Each method does one thing. Each owner provides what it naturally owns. The caller (the agent loop) composes them — materialize() internally calls both get_metadata() + retrieve() (its own) and sandbox.import_file() (the sandbox's). Composition over intermediation.

PrincipleHow It Applied
YAGNI Five lifecycle states when two booleans suffice. Dirty tracking when the filesystem tracks state. Delete what you don't need.
Single Responsibility The bridge had three responsibilities: know zone paths, track state, orchestrate. Split them: sandbox owns zones, filesystem tracks state, MediaBackend orchestrates.
Interface Segregation Two methods on Sandbox instead of one fat bridge. Three methods on MediaBackend instead of one fat bridge. Small interfaces, focused contracts.
Dependency Inversion MediaBackend depends on the Sandbox ABC, not on LocalSandbox. TYPE_CHECKING import prevents runtime coupling. The dependency is abstract, not concrete.
Zone Encapsulation MediaBackend never constructs paths like .imported/{id}/. That's the sandbox's concern. Swap the sandbox implementation, and zone layout can change freely.
· · ·

The arc of this diary is the arc of most over-engineering stories: a reasonable-sounding abstraction that turned out to be solving a problem that didn't exist. The bridge pattern is valuable when two systems genuinely can't know about each other. But when one system owns the filesystem and the other owns storage, they can each handle their half. The bridge was the answer to a question nobody was asking.

The final design is five methods spread across two existing classes. No new files. No new classes. No state machine. The filesystem is the manifest.

The best code is code you don't write.
The best abstraction is the one you delete.
The best state machine is the filesystem.

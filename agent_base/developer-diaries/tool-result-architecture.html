Who Owns the Tool Result?

A Developer Diary — Designing Result Ownership in an Agentic Architecture

The Situation

Two Consumers, One Tool Result, Conflicting Needs

You've built an agentic system with a clean canonical message model. Content blocks are polymorphic — TextContent, ImageContent, ToolUseContent, etc. The system works. But a tension has been growing between two consumers of tool results, and it's about to break your abstractions.

Your agentic loop maintains two parallel states. AgentConfig manages the live LLM context window — it cares about what the model needs to see for its next turn. ConversationHistory is the full record for the UI — it cares about showing the human what happened, step by step, including execution logs, intermediate steps, and rich media.

Developer's inner voice "When a tool returns a result, AgentConfig just wants the final answer: a list of content blocks to stuff back into the context window. But ConversationHistory might want the entire execution trace — logs, intermediate steps, timing data, images with different storage references. These are fundamentally different projections of the same event."

And there's a second, deeper problem. Today, your ToolResultContent is a flat block in the canonical model. But tool results are not like other content blocks. A TextContent varies only across LLM providers (Anthropic says text, OpenAI says content). A tool result varies across tool types — a code execution tool returns stdout + files, a web search tool returns citations + snippets, a sub-agent returns an entire nested conversation. The canonical model can't anticipate all of these.

Fig 1 — Tool results have a fundamentally different axis of variation
🚨 The Design Tension

Your canonical ToolResultBase tries to represent all tool results with tool_result: str | List[Dict]. This works for simple tools but collapses into an untyped bag-of-dicts for rich tools. The type system can't help you, the serialization is lossy, and every consumer has to do ad-hoc parsing. Worse, you can't add tool-specific metadata (execution logs, timing, generated files) without polluting the canonical model with fields that only matter to one tool.

Chapter 1 — Ownership
The Key Insight

"The Tool Knows Best What Its Result Looks Like"

Let's think about who has the knowledge here. The canonical model knows about message structure — roles, content blocks, tool correlation IDs. But it doesn't know (and shouldn't know) that a code execution tool produces stdout, stderr, exit codes, and generated files. That knowledge lives with the tool itself.

Developer's inner voice "This is a classic 'information expert' problem from GRASP principles. The responsibility for defining a tool's result shape should go to the class that has the most information about it — the tool itself. My ConfigurableToolBase already owns the tool's schema and execution. Why not also let it own the result definition?"

But there's a constraint: AgentConfig needs a standardized projection of any tool result (content blocks for the LLM context window), while ConversationHistory needs a rich projection (everything the UI might want to display). The tool can produce the rich data, but it needs to know how to project it into both shapes.

This is where the pattern crystallizes. We need a Rich Result object owned by the tool, with two projection methods: one for the LLM context window, one for the conversation log.

Chapter 2 — The Envelope
Designing the Contract

ToolResultEnvelope — The Tool's Rich Return Type

Instead of having tools return a flat string or a ToolResult that only knows about text and images, we introduce a ToolResultEnvelope. This is an abstract base that every tool-specific result type extends. The key insight: the envelope is not a content block — it's a pre-projection object that knows how to produce content blocks for different consumers.

tool_result_envelope.py Abstract Contract 
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import Any

# Import the canonical content blocks from your model
from .core import ContentBlock, TextContent, ImageContent, ErrorContent


@dataclass
class ToolResultEnvelope(ABC):
    """Base class for tool-specific rich result objects.
    
    Every tool defines its own subclass with whatever fields it needs.
    The two abstract methods project this rich data into the shapes
    that each consumer (AgentConfig vs ConversationHistory) requires.
    
    This is NOT a ContentBlock. It's a pre-projection object that
    PRODUCES ContentBlocks. The tool owns the data; the projections
    adapt it to each consumer's needs.
    """

    # --- Shared metadata every result has ---
    tool_name: str = ""
    tool_id: str = ""
    is_error: bool = False
    error_message: str | None = None
    duration_ms: float | None = None

    # ─── Projection 1: For the LLM context window ───

    @abstractmethod
    def for_context_window(self) -> list[ContentBlock]:
        """Project this result into ContentBlocks for the LLM.
        
        This is what gets stuffed into the ToolResultContent block
        inside AgentConfig.messages. The LLM will see these blocks
        on its next turn.
        
        Rules:
          - Must return canonical ContentBlock instances
          - Should be MINIMAL — only what the LLM needs to continue
          - Images/docs should use the provider-appropriate source type
          - Errors should return [ErrorContent(...)] or [TextContent("Error: ...")]
        """
        ...

    # ─── Projection 2: For the conversation log (UI) ───

    @abstractmethod
    def for_conversation_log(self) -> dict[str, Any]:
        """Project this result into a rich dict for ConversationHistory.
        
        This is what gets stored in the conversation log for UI display.
        It can include anything the frontend might want to render:
        execution logs, timing, intermediate steps, file references,
        nested sub-agent conversations, etc.
        
        The dict should always include at minimum:
          {
            "tool_name": str,
            "tool_id": str,
            "is_error": bool,
            "summary": str,      # human-readable one-liner
            "content_blocks": [], # the same blocks as for_context_window
            ...tool-specific fields...
          }
        """
        ...

    # ─── Convenience: error envelope factory ───

    @classmethod
    def error(cls, tool_name: str, tool_id: str, message: str) -> "ToolResultEnvelope":
        """Create a generic error envelope (works for any tool)."""
        return GenericErrorEnvelope(
            tool_name=tool_name, tool_id=tool_id,
            is_error=True, error_message=message,
        )


# --- A simple fallback for tools that don't define their own envelope ---

@dataclass
class GenericErrorEnvelope(ToolResultEnvelope):
    """Fallback envelope for errors or legacy tools."""

    def for_context_window(self) -> list[ContentBlock]:
        return [TextContent(text=f"Error: {self.error_message}")]

    def for_conversation_log(self) -> dict[str, Any]:
        return {
            "tool_name": self.tool_name,
            "tool_id": self.tool_id,
            "is_error": True,
            "summary": self.error_message,
            "content_blocks": [TextContent(text=self.error_message or "").to_dict()],
        }
Developer's inner voice "Notice what I did NOT do: I didn't make this a ContentBlock subclass. That would pollute the canonical model with tool-specific data. The envelope is a separate layer — it sits between tool execution and the canonical message chain. It's the tool's 'rich return type' that gets projected down into the simpler shapes that each consumer needs."
🧩 Pattern Recognition

This is a Projection pattern (sometimes called "View Model" or "DTO projection"). The same source data gets transformed into different shapes for different consumers. The key is that the source (the tool) owns the projection logic, because only the tool knows what's meaningful in each context. AgentConfig gets a lean projection; ConversationHistory gets a rich one.

Chapter 3 — Concrete Envelopes
Building Real Envelopes

Each Tool Defines Its Own Rich Result

Now let's see how three very different tools — a code executor, a web search tool, and a sub-agent — each define their own envelope with tool-specific fields, while conforming to the same dual-projection contract.

Example A: Code Execution Tool

code_exec_envelope.py Tool-Owned 
@dataclass
class CodeExecEnvelope(ToolResultEnvelope):
    """Rich result from a code execution tool."""

    # ── Tool-specific fields (the canonical model knows nothing about these) ──
    stdout: str = ""
    stderr: str = ""
    exit_code: int = 0
    generated_files: list[dict] = field(default_factory=list)  # [{path, mime_type, size}]
    generated_images: list[dict] = field(default_factory=list)  # [{path, mime_type, base64}]
    execution_log: list[str] = field(default_factory=list)     # line-by-line execution trace

    # ── Projection 1: What the LLM sees ──
    def for_context_window(self) -> list[ContentBlock]:
        blocks: list[ContentBlock] = []

        # LLM gets the stdout (the "answer")
        if self.stdout:
            blocks.append(TextContent(text=self.stdout))

        # LLM gets generated images inline (so it can reason about them)
        for img in self.generated_images:
            blocks.append(ImageContent(
                source_type=SourceType.BASE64,
                media_type=img["mime_type"],
                data=img["base64"],
            ))

        # LLM gets a FILE MANIFEST (not the files themselves)
        if self.generated_files:
            manifest = "Generated files:\n" + "\n".join(
                f"  - {f['path']} ({f['mime_type']}, {f['size']} bytes)"
                for f in self.generated_files
            )
            blocks.append(TextContent(text=manifest))

        # stderr only if error
        if self.is_error and self.stderr:
            blocks.append(TextContent(text=f"STDERR:\n{self.stderr}"))

        return blocks or [TextContent(text="(no output)")]

    # ── Projection 2: What the UI sees ──
    def for_conversation_log(self) -> dict[str, Any]:
        return {
            "tool_name": self.tool_name,
            "tool_id": self.tool_id,
            "is_error": self.is_error,
            "summary": self.stdout[:200] if self.stdout else "(no output)",
            "duration_ms": self.duration_ms,
            # ── UI-specific rich data ──
            "stdout": self.stdout,
            "stderr": self.stderr,
            "exit_code": self.exit_code,
            "execution_log": self.execution_log,
            "generated_files": self.generated_files,
            "generated_images": [
                {"path": img["path"], "mime_type": img["mime_type"]}
                for img in self.generated_images
            ],  # Note: no base64 in the log — just references
            "content_blocks": [b.to_dict() for b in self.for_context_window()],
        }

Notice the asymmetry in the two projections. The LLM gets the stdout and inline images (it needs to reason about them). The conversation log gets stdout, stderr, exit code, execution trace, and file references — but without base64 image blobs (the UI will load those from URLs). Each consumer gets exactly what it needs, no more.

Example B: Sub-Agent Tool

Sub-agents are the most extreme case. A sub-agent's result is an entire nested conversation. The LLM only needs to see the final answer. The UI might want to display the full nested conversation as a collapsible tree.

subagent_envelope.py Tool-Owned 
@dataclass
class SubAgentEnvelope(ToolResultEnvelope):
    """Rich result from a sub-agent execution."""

    # ── Sub-agent specific data ──
    final_answer: str = ""
    nested_messages: list[dict] = field(default_factory=list)  # Full sub-conversation
    total_steps: int = 0
    sub_agent_model: str = ""
    sub_agent_usage: dict = field(default_factory=dict)

    def for_context_window(self) -> list[ContentBlock]:
        # The parent LLM ONLY sees the final answer — not the entire sub-conversation
        return [TextContent(text=self.final_answer)]

    def for_conversation_log(self) -> dict[str, Any]:
        return {
            "tool_name": self.tool_name,
            "tool_id": self.tool_id,
            "is_error": self.is_error,
            "summary": self.final_answer[:200],
            "duration_ms": self.duration_ms,
            # ── The UI gets the entire nested conversation tree ──
            "final_answer": self.final_answer,
            "nested_messages": self.nested_messages,
            "total_steps": self.total_steps,
            "sub_agent_model": self.sub_agent_model,
            "sub_agent_usage": self.sub_agent_usage,
            "content_blocks": [b.to_dict() for b in self.for_context_window()],
        }
✅ What We've Achieved

Each tool now owns its result shape completely. The CodeExecEnvelope has stdout, stderr, and execution_log. The SubAgentEnvelope has nested_messages and sub_agent_usage. These fields don't pollute the canonical model — they exist only in the tool's envelope. The canonical model stays clean: it just sees List[ContentBlock] from the context-window projection.

Chapter 4 — Wiring
Integration

How the Agentic Loop Consumes Envelopes

Now let's see how this changes the agentic loop. Previously, the loop received a flat (content, image_refs) tuple from tool execution. Now it receives a ToolResultEnvelope and projects it differently for each consumer.

agent_loop.py Consumption Site 
async def _handle_tool_result(
    self,
    tool_use_block: ToolUseContent,
    envelope: ToolResultEnvelope,
):
    """Process a tool result envelope for both consumers."""

    # ── Projection 1: Feed the LLM (AgentConfig) ──
    context_blocks = envelope.for_context_window()
    
    # Build the canonical ToolResultContent for the context window.
    # This is the standard content block that every LLM provider adapter knows.
    result_block = ToolResultContent(
        tool_name=tool_use_block.tool_name,
        tool_id=tool_use_block.tool_id,
        tool_result=[b.to_dict() for b in context_blocks],  # ← Nested content blocks!
        is_error=envelope.is_error,
    )
    
    # Append to the context window (AgentConfig.messages)
    self.agent_config.messages.append(
        Message(role=Role.USER, content=[result_block])
    )

    # ── Projection 2: Log for the UI (ConversationHistory) ──
    log_entry = envelope.for_conversation_log()
    
    self.conversation_history.append_tool_result(
        tool_use_id=tool_use_block.tool_id,
        log_entry=log_entry,  # ← Rich data: logs, files, timing, nested convos
    )

The agentic loop doesn't know whether it's dealing with a code execution result, a web search result, or a sub-agent result. It just calls the two projection methods. The tool's envelope handles the rest. This is polymorphism doing what it does best — eliminating if/elif chains in the consumer.

Updating ConfigurableToolBase

The final piece is making ConfigurableToolBase aware of envelopes. Tools that want rich results override a new method; legacy tools that return plain strings still work.

configurable_tool_base.py Updated Contract 
class ConfigurableToolBase(ABC):
    """Updated to support ToolResultEnvelope returns."""

    # ... existing __init__, _render_docstring, _apply_schema ...

    @abstractmethod
    def get_tool(self) -> Callable:
        """Return a @tool decorated function.
        
        The function may return:
          - str                   → legacy, auto-wrapped in GenericTextEnvelope
          - ToolResult            → legacy, auto-wrapped in GenericMediaEnvelope
          - ToolResultEnvelope    → new, used directly (preferred)
        """
        ...

    # --- Optional: Override to customize the envelope type ---

    def get_envelope_class(self) -> type[ToolResultEnvelope] | None:
        """Return the envelope class this tool produces.
        
        Used by the framework for:
          - Type introspection / documentation generation
          - Schema validation of tool results
          - UI component resolution (which React component renders this?)
        
        Return None if the tool uses legacy str/ToolResult returns.
        """
        return None

And in the ToolRegistry.execute() method, we add automatic wrapping for legacy tools:

tool_registry.py Updated 
async def execute(self, tool_name, tool_input, **kwargs) -> ToolResultEnvelope:
    """Execute a tool. Always returns a ToolResultEnvelope."""
    try:
        result = await self._invoke(tool_name, tool_input)

        # New path: tool returned an envelope directly
        if isinstance(result, ToolResultEnvelope):
            return result

        # Legacy path: wrap plain str/ToolResult in a generic envelope
        if isinstance(result, ToolResult):
            return LegacyMediaEnvelope.from_tool_result(
                tool_name=tool_name, result=result, **kwargs
            )

        # Plain string fallback
        return GenericTextEnvelope(
            tool_name=tool_name,
            tool_id=kwargs.get("tool_id", ""),
            text=str(result),
        )

    except Exception as e:
        return ToolResultEnvelope.error(tool_name, kwargs.get("tool_id", ""), str(e))
💡 Backwards Compatibility

The auto-wrapping logic means every existing tool that returns a str or ToolResult continues to work without any changes. The envelope is purely additive — tools can opt into it when they need richer result semantics. This is the Open-Closed Principle in action: the system is open for extension (new envelope types) but closed for modification (existing tools don't break).

Naming the Patterns

Let's step back and name the design patterns we just discovered by solving this problem. The interesting thing is we didn't start with patterns — we started with pain and the patterns emerged.

Pattern Where It Appears What Pain It Solves
Projection / View Model for_context_window() and for_conversation_log() Same data, different consumers need different shapes. The source object owns the transformation logic.
Envelope / Rich Result ToolResultEnvelope subclasses Tool-specific data that doesn't belong in the canonical model but needs to travel with the result.
Information Expert (GRASP) Tools own their envelope definitions The class with the most knowledge about data shape should own its serialization.
Adapter (legacy wrapping) GenericTextEnvelope, auto-wrapping in execute() Old tools returning str get wrapped into the new interface without code changes.
Open-Closed Principle New tools add new envelope subclasses Adding a new tool type never requires modifying the agentic loop, AgentConfig, or ConversationHistory.
Chapter 6 — Rejected Alternatives
Developer's Decision Log

Paths Not Taken (And Why)

A senior developer's diary isn't just about what they built — it's about what they considered and rejected. Here are the alternatives that were weighed.

Alternative A: "Just Make ToolResultBase Fatter"

Rejected "I could add optional fields to ToolResultBase — stdout, stderr, nested_messages, execution_log, etc. But this creates a God Object. Every tool would see every other tool's fields. The type system can't enforce which combinations are valid. And every new tool type means modifying the canonical model, violating Open-Closed."

Alternative B: "Stuff Everything in kwargs"

Rejected "I could use the existing kwargs field to carry tool-specific data. But kwargs is a Dict[str, Any] — completely untyped. No autocomplete, no validation, no documentation. Every consumer would need to know the exact key names for each tool type, and there's no compile-time help. This is the 'stringly typed' anti-pattern."

Alternative C: "Two Separate Return Types from Tools"

Rejected "I could have tools return a tuple of (context_window_blocks, conversation_log_dict). But this couples the tool to knowledge of both consumers. If a third consumer appears (e.g., analytics/billing), every tool needs updating. The envelope pattern defers projection to method calls, so adding a third projection method is a single interface change."
🔑 The Decisive Factor

The envelope won because it's the only approach where adding a new tool type requires zero changes outside the tool's own code, and adding a new consumer requires only adding one abstract method to the base. Both axes of change are isolated. This is the hallmark of a well-factored design — changes in one dimension don't ripple into others.

Epilogue — The Thought Process
Meta-Reflection

How Senior Developers Extract Patterns

The journey we just walked through follows a repeatable mental process. It's not about knowing pattern names upfront — it's about recognizing structural tensions in code and resolving them by separating concerns.

Fig 4 — Pain → Axes of Change → Boundary → Named Pattern → Iterate

In this specific case, we identified two orthogonal axes of variation: tool type (code exec vs. web search vs. sub-agent) and consumer type (LLM context vs. UI log). The canonical model was trying to collapse both axes into one flat hierarchy, which is why it was breaking. The envelope introduces a new layer that handles the tool-type axis, while the projection methods handle the consumer-type axis. Each axis can now vary independently.

✅ The Litmus Test

Ask yourself these two questions about the design: "If I add a new tool type (say, a database query tool), how many files do I touch?" Answer: one — the new tool's file, where I define both the envelope and the tool. "If I add a new consumer (say, a billing/analytics system), how many files do I touch?" Answer: one — the ToolResultEnvelope base, where I add a third projection method. If both answers are "one", the factoring is right.

Patterns are the names we give to structural solutions after we discover them.
The skill isn't knowing the names — it's feeling the pain that calls for them.

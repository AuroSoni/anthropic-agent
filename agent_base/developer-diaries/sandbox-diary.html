Sandboxing Agent Tools:
From Base Paths to Containers

Developer Diary — Designing for Incremental Isolation

The tools work. Now we need to cage them — gently at first, then seriously — without rewriting a single tool.

The Problem

Why Sandboxing Is Harder Than It Looks in an Agentic System

Your agent has a toolkit: file_read, file_write, list_dir, grep, apply_patch, bash, code_exec. Right now, they all operate directly on the host filesystem. bash runs subprocess.run(). file_read opens whatever path the LLM provides. And you're starting to realize this is terrifying.

The LLM can hallucinate a path like /etc/passwd and your file_read will cheerfully return its contents. The bash tool can run rm -rf / if the model is confused enough. Two concurrent agent sessions can step on each other's files. There is no isolation, no containment, and no cleanup boundary.

You need sandboxing. But here's the constraint that makes this architecturally interesting: you can't pause everything for two months while you build a Docker-based sandbox. You need to ship isolation incrementally. Today: simple path restriction. Next month: Docker containers. Eventually: cloud sandboxes like E2B. And through all of this, the tools themselves should not change.

Developer's inner voice "This is a classic interface-extraction problem. Right now, tools do I/O directly — open(), subprocess.run(), os.listdir(). I need to put something between the tools and the OS. That 'something' is the sandbox. But the sandbox's internal implementation will evolve from 'path restriction' to 'Docker container' to 'cloud VM.' The tools can't know or care which level they're running in. This means I need an abstraction that captures what tools need — filesystem access and command execution — while hiding how those operations are implemented."
🚨 The Core Constraint

Tools must be sandbox-agnostic. A file_read tool that works with a path-restricted sandbox today must work unchanged inside a Docker container tomorrow and inside an E2B cloud VM next quarter. This means tools cannot do raw open() or subprocess.run(). They must delegate all I/O through a sandbox interface. The sandbox is the seam between the agent's intent and the operating system's reality.

Chapter 1 — What We're Replacing
The Current State

Tools That Talk Directly to the OS

Here's what your tools look like before sandboxing. They do raw I/O. They trust whatever path the LLM provides. They run commands as the current user with full filesystem access. Every tool is a little root shell.

tools/file_tools.py Before — No Sandbox 
class FileReadTool(ConfigurableToolBase):
    def get_tool(self):
        async def file_read(path: str) -> str:
            # Opens ANY path on the host filesystem. /etc/passwd? Sure!
            async with aiofiles.open(path, "r") as f:
                return await f.read()
        return self._apply_schema(file_read)


class BashTool(ConfigurableToolBase):
    def get_tool(self):
        async def bash(command: str) -> str:
            # Runs ANY command as the current user. rm -rf /? No problem!
            proc = await asyncio.create_subprocess_shell(
                command,
                stdout=asyncio.subprocess.PIPE,
                stderr=asyncio.subprocess.PIPE,
            )
            stdout, stderr = await proc.communicate()
            return stdout.decode() + (f"\nSTDERR: {stderr.decode()}" if stderr else "")
        return self._apply_schema(bash)

The issue isn't just security. It's also the absence of a cleanup boundary. When an agent session ends, how do you know which files on the filesystem belong to that session? You don't. There's no namespace, no fencing, no way to say "delete everything this agent touched." And when two agent sessions run concurrently, their files are interleaved in whatever directory they happen to use. One agent's rm *.py can nuke another agent's work.

Chapter 2 — The Abstraction
Designing the Seam

What Do Tools Actually Need From "the Outside World"?

Before writing any sandbox code, let's inventory what our tools actually do with the OS. Not what they could do — what they need to do to serve the agent's purpose. This inventory drives the interface.

Developer's inner voice "Let me trace through each tool and list every syscall-level operation it performs. file_read needs to read a file's content. file_write needs to create or overwrite a file. list_dir needs to list entries in a directory. grep needs to search through file contents. apply_patch needs to read, modify, and write a file. bash needs to execute an arbitrary shell command and capture stdout/stderr. code_exec needs to write a temp file, execute it, and capture the output. If I squint, there are really only three fundamental capabilities: read/write files in a namespace, execute commands, and manage lifecycle (setup and teardown). Everything else is composition."

This gives us the interface. It has three faces: a filesystem, a command executor, and a lifecycle manager. Notably, the filesystem face uses relative paths — paths relative to the sandbox's workspace root. This is crucial. The LLM generates paths like src/main.py, not /var/sandboxes/agent_abc123/workspace/src/main.py. The sandbox resolves relative paths to absolute paths internally. No consumer ever sees or touches the real filesystem location.

sandbox/base.py Abstract Contract 
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import AsyncIterator


@dataclass
class ExecResult:
    """The output of a command execution inside a sandbox."""
    exit_code: int = 0
    stdout: str = ""
    stderr: str = ""
    timed_out: bool = False
    duration_ms: float = 0.0


@dataclass
class FileEntry:
    """A single entry in a directory listing."""
    name: str
    is_dir: bool = False
    size_bytes: int = 0


class Sandbox(ABC):
    """Abstract execution environment for agent tools.
    
    A Sandbox provides two capabilities:
      1. Filesystem — read, write, list, delete files in an isolated namespace
      2. Execution  — run shell commands and code within the sandbox boundary
    
    Tools interact with the Sandbox through this interface.
    They never do raw open(), subprocess.run(), or os.listdir().
    
    All file paths are RELATIVE to the sandbox workspace root.
    The sandbox resolves them to real paths internally.
    
    Implementations:
      - LocalSandbox:  path-restricted directory on the host
      - DockerSandbox: Docker container with mounted workspace
      - E2BSandbox:    remote E2B cloud sandbox via API
    """

    # ─── Lifecycle ───

    @abstractmethod
    async def setup(self) -> None:
        """Initialize the sandbox.
        
        LocalSandbox:   create the workspace directory
        DockerSandbox:  pull image, create container, start it
        E2BSandbox:     create a remote sandbox instance via API
        
        Must be called before any other method.
        """
        ...

    @abstractmethod
    async def teardown(self) -> None:
        """Destroy the sandbox and all its contents.
        
        LocalSandbox:   rm -rf the workspace directory
        DockerSandbox:  stop and remove the container
        E2BSandbox:     terminate the remote sandbox
        
        After teardown, no other method should be called.
        """
        ...

    # ─── Filesystem ───

    @abstractmethod
    async def read_file(self, path: str) -> str:
        """Read a text file at the given relative path."""
        ...

    @abstractmethod
    async def write_file(self, path: str, content: str) -> None:
        """Write content to a file. Creates parent directories as needed."""
        ...

    @abstractmethod
    async def read_file_bytes(self, path: str) -> bytes:
        """Read a binary file."""
        ...

    @abstractmethod
    async def write_file_bytes(self, path: str, content: bytes) -> None:
        """Write binary content to a file."""
        ...

    @abstractmethod
    async def list_dir(self, path: str = ".") -> list[FileEntry]:
        """List entries in a directory."""
        ...

    @abstractmethod
    async def file_exists(self, path: str) -> bool:
        """Check whether a file or directory exists."""
        ...

    @abstractmethod
    async def delete(self, path: str) -> bool:
        """Delete a file or directory."""
        ...

    # ─── Command Execution ───

    @abstractmethod
    async def exec(
        self,
        command: str,
        timeout: float = 30.0,
        cwd: str | None = None,    # Relative path within the sandbox
        env: dict[str, str] | None = None,
    ) -> ExecResult:
        """Execute a shell command inside the sandbox.
        
        LocalSandbox:   subprocess with cwd set to workspace
        DockerSandbox:  docker exec inside the container
        E2BSandbox:     API call to run command in remote VM
        
        All commands run within the sandbox boundary. They cannot
        escape to the host filesystem (for Docker and E2B backends).
        """
        ...

    # ─── Streaming Execution (for long-running commands) ───

    @abstractmethod
    async def exec_stream(
        self,
        command: str,
        timeout: float = 30.0,
        cwd: str | None = None,
        env: dict[str, str] | None = None,
    ) -> AsyncIterator[str]:
        """Execute a command and stream output line by line.
        
        Useful for long-running builds, test suites, etc. where the
        agent (or UI) needs incremental output.
        """
        ...

    # ─── Context Manager (convenience) ───

    async def __aenter__(self):
        await self.setup()
        return self

    async def __aexit__(self, *exc):
        await self.teardown()
🧩 Why Relative Paths Are Non-Negotiable

The LLM generates paths like src/main.py, tests/test_auth.py, ./config.json. These are workspace-relative paths — exactly what a developer would type. The sandbox resolves them internally. In a LocalSandbox, src/main.py resolves to /var/sandboxes/abc123/workspace/src/main.py. In a DockerSandbox, it resolves to /workspace/src/main.py inside the container. In an E2BSandbox, it resolves to an API call targeting /home/user/src/main.py on the remote VM. The LLM and the tools never know the real absolute path. This is what makes the sandbox swappable.

One design decision worth calling out: exec() takes a command: str, not a list[str]. This is deliberate. The LLM generates shell commands as strings ("python main.py --verbose", "grep -rn 'TODO' src/"). Parsing these into argument lists ourselves would be fragile and would lose shell features like pipes, redirection, and globbing. We pass the string to a shell inside the sandbox — which in a Docker or E2B context is safely contained. In the local sandbox, this is the least-isolated part of the system, which is precisely why you upgrade to Docker when security matters.

Chapter 3 — Level 1: Path Restriction
The Simplest Sandbox

LocalSandbox — Fenced Directories, Ship It Today

The local sandbox is not a security boundary. It's an organizational boundary. It gives each agent session its own directory, resolves all paths relative to that directory, prevents path traversal attacks (no ../../etc/passwd), and provides a clean teardown by simply deleting the directory. It's what you ship today while you build the real container-based sandbox.

Isolation Path only — no process isolation
Security Path traversal prevention
Setup Time ~1ms (mkdir)
Best For Dev, trusted agents
sandbox/local.py Level 1 
import asyncio, os, shutil, time
from pathlib import Path


class LocalSandbox(Sandbox):
    """Path-restricted sandbox on the local filesystem.
    
    Each agent session gets its own directory under base_dir:
        {base_dir}/{sandbox_id}/workspace/
    
    All file paths are resolved relative to workspace/.
    Path traversal is blocked by resolving and checking containment.
    Commands run as subprocesses with cwd set to the workspace.
    
    This provides organizational isolation, not security isolation.
    Use DockerSandbox or E2BSandbox when you need real containment.
    """

    def __init__(
        self,
        sandbox_id: str,
        base_dir: str = "/var/sandboxes",
        default_timeout: float = 30.0,
    ):
        self.sandbox_id = sandbox_id
        self.base_dir = Path(base_dir)
        self.workspace = self.base_dir / sandbox_id / "workspace"
        self.default_timeout = default_timeout

    # ─── Path Resolution (the critical security check) ───

    def _resolve(self, path: str) -> Path:
        """Resolve a relative path to an absolute path within the workspace.
                
                Raises ValueError if the resolved path escapes the workspace.
                This blocks path traversal attacks like '../../etc/passwd'.
                """
        # Resolve relative to workspace, then check containment
        resolved = (self.workspace / path).resolve()
        workspace_resolved = self.workspace.resolve()
        
        if not str(resolved).startswith(str(workspace_resolved)):
            raise ValueError(
                f"Path traversal blocked: '{path}' resolves outside workspace"
            )
        return resolved

    # ─── Lifecycle ───

    async def setup(self):
        self.workspace.mkdir(parents=True, exist_ok=True)

    async def teardown(self):
        sandbox_root = self.base_dir / self.sandbox_id
        if sandbox_root.exists():
            shutil.rmtree(sandbox_root)

    # ─── Filesystem ───

    async def read_file(self, path: str) -> str:
        resolved = self._resolve(path)
        async with aiofiles.open(resolved, "r") as f:
            return await f.read()

    async def write_file(self, path: str, content: str):
        resolved = self._resolve(path)
        resolved.parent.mkdir(parents=True, exist_ok=True)
        async with aiofiles.open(resolved, "w") as f:
            await f.write(content)

    async def list_dir(self, path: str = ".") -> list[FileEntry]:
        resolved = self._resolve(path)
        entries = []
        for item in sorted(resolved.iterdir()):
            entries.append(FileEntry(
                name=item.name,
                is_dir=item.is_dir(),
                size_bytes=item.stat().st_size if item.is_file() else 0,
            ))
        return entries

    async def file_exists(self, path: str) -> bool:
        return self._resolve(path).exists()

    async def delete(self, path: str) -> bool:
        resolved = self._resolve(path)
        if resolved.is_dir():
            shutil.rmtree(resolved)
        elif resolved.is_file():
            resolved.unlink()
        else:
            return False
        return True

    # ─── Command Execution ───

    async def exec(self, command, timeout=None, cwd=None, env=None):
        timeout = timeout or self.default_timeout
        work_dir = self._resolve(cwd) if cwd else self.workspace

        start = time.monotonic()
        try:
            proc = await asyncio.create_subprocess_shell(
                command,
                cwd=str(work_dir),  # Restrict working directory
                stdout=asyncio.subprocess.PIPE,
                stderr=asyncio.subprocess.PIPE,
                env={**(os.environ | (env or {}))},
            )
            stdout, stderr = await asyncio.wait_for(
                proc.communicate(), timeout=timeout
            )
            return ExecResult(
                exit_code=proc.returncode,
                stdout=stdout.decode(errors="replace"),
                stderr=stderr.decode(errors="replace"),
                duration_ms=(time.monotonic() - start) * 1000,
            )
        except asyncio.TimeoutError:
            proc.kill()
            return ExecResult(exit_code=-1, timed_out=True,
                              duration_ms=(time.monotonic() - start) * 1000)

    async def exec_stream(self, command, timeout=None, cwd=None, env=None):
        work_dir = self._resolve(cwd) if cwd else self.workspace
        proc = await asyncio.create_subprocess_shell(
            command, cwd=str(work_dir),
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.STDOUT,  # Merge stderr into stdout for streaming
            env={**(os.environ | (env or {}))},
        )
        async for line in proc.stdout:
            yield line.decode(errors="replace")
        await proc.wait()
Developer's inner voice "The _resolve() method is the entire security model of the local sandbox. It's one function: resolve the path, check that it's still under the workspace. If someone passes '../../etc/passwd', Path.resolve() follows the '..' segments, and the containment check catches it. This is enough to prevent accidental path escapes and LLM hallucinations. It is NOT enough to prevent a determined attacker — the subprocess can still escape via bash commands like 'cat /etc/passwd' or symlink tricks. That's why this is Level 1. Real isolation needs a process boundary."
Chapter 4 — Rewriting Tools
The One-Time Migration

Tools That Delegate to the Sandbox

Here's the critical payoff. We rewrite each tool once to delegate through the Sandbox interface. After this, the tools never change again — regardless of whether the sandbox is a local directory, a Docker container, or a cloud VM.

tools/file_tools.py After — Sandbox-Aware 
class FileReadTool(ConfigurableToolBase):
    """Read file contents. Delegates to the Sandbox — works in any environment."""
    
    def __init__(self, sandbox: Sandbox, **kwargs):
        super().__init__(**kwargs)
        self.sandbox = sandbox

    def get_tool(self):
        sandbox = self.sandbox
        async def file_read(path: str, offset: int = 0, limit: int | None = None) -> str:
            """Read the contents of a file.

            Args:
                path: Relative path to the file (e.g., "src/main.py").
                offset: Line offset to start from.
                limit: Maximum number of lines to return.
            """
            # One line. Works on local disk, Docker, E2B — doesn't matter.
            content = await sandbox.read_file(path)
            
            lines = content.splitlines(keepends=True)
            if limit:
                lines = lines[offset:offset + limit]
            else:
                lines = lines[offset:]
            return "".join(lines)

        return self._apply_schema(file_read)


class FileWriteTool(ConfigurableToolBase):
    def __init__(self, sandbox: Sandbox, **kwargs):
        super().__init__(**kwargs)
        self.sandbox = sandbox

    def get_tool(self):
        sandbox = self.sandbox
        async def file_write(path: str, content: str) -> str:
            await sandbox.write_file(path, content)
            return f"Wrote {len(content)} chars to {path}"
        return self._apply_schema(file_write)


class BashTool(ConfigurableToolBase):
    def __init__(self, sandbox: Sandbox, **kwargs):
        super().__init__(**kwargs)
        self.sandbox = sandbox

    def get_tool(self):
        sandbox = self.sandbox
        async def bash(command: str, timeout: float = 30.0) -> str:
            """Execute a shell command.

            Args:
                command: The shell command to run.
                timeout: Maximum execution time in seconds.
            """
            result = await sandbox.exec(command, timeout=timeout)
            
            output = result.stdout
            if result.stderr:
                output += f"\nSTDERR:\n{result.stderr}"
            if result.timed_out:
                output += "\n[Command timed out]"
            if result.exit_code != 0:
                output += f"\n[Exit code: {result.exit_code}]"
            return output
        return self._apply_schema(bash)


class GrepTool(ConfigurableToolBase):
    def __init__(self, sandbox: Sandbox, **kwargs):
        super().__init__(**kwargs)
        self.sandbox = sandbox

    def get_tool(self):
        sandbox = self.sandbox
        async def grep(pattern: str, path: str = ".", include: str = "") -> str:
            """Search for a pattern in files.

            Args:
                pattern: Regex pattern to search for.
                path: Directory or file to search in.
                include: Glob pattern for files to include (e.g., "*.py").
            """
            cmd = f'grep -rn "{pattern}" {path}'
            if include:
                cmd += f' --include="{include}"'
            result = await sandbox.exec(cmd, timeout=15.0)
            return result.stdout or "No matches found."
        return self._apply_schema(grep)

Notice what happened. Every tool that used to do open() now calls sandbox.read_file(). Every tool that used to do subprocess.run() now calls sandbox.exec(). The tools have no idea whether the sandbox is a local directory or a Docker container. They don't import os, subprocess, or aiofiles. All OS interaction is mediated through the sandbox interface. This is the Dependency Inversion Principle applied to the operating system itself — tools depend on an abstraction (Sandbox), not a concretion (the host OS).

✅ The Invariant

After this migration, tools are finished. The entire sandbox evolution — from local directories to Docker containers to cloud VMs — happens behind the Sandbox interface. No tool file is ever touched again. The litmus test: grep -r "import os\|import subprocess\|import aiofiles" tools/ should return zero results. If a tool imports an OS module, it's leaking through the abstraction.

Chapter 5 — Level 2: Docker Containers
Real Process Isolation

DockerSandbox — Same Interface, Different Universe

A month later, you need real isolation. An enterprise client wants their agents sandboxed so that a misbehaving LLM can't read host files, exhaust host memory, or interfere with other sessions. The Sandbox interface means you can build this without touching any tool code.

Isolation Process + filesystem + network
Security Container boundary, resource limits
Setup Time ~2–5s (container start)
Best For Production, untrusted code
Developer's inner voice "The architecture is: create a container from a prebuilt image (with Python, Node, common tools installed), mount the workspace as a volume so files are shared between host and container, and run all commands via 'docker exec'. File operations go through the mounted volume — so read_file and write_file are still local filesystem operations on the host, but exec() now runs inside the container's process namespace. This gives me real process isolation (the container can't see host processes), filesystem isolation (the container only sees its own root plus the mounted workspace), and resource limits (CPU, memory caps via Docker)."
sandbox/docker.py Level 2 
import aiodocker

class DockerSandbox(Sandbox):
    """Docker-based sandbox with real process and filesystem isolation.
    
    Architecture:
      - A container is created from a prebuilt image
      - The workspace directory is bind-mounted into the container
      - File operations go through the mount (fast, no copying)
      - Command execution happens via 'docker exec' (fully isolated)
      - Container has CPU/memory limits
      - Teardown removes the container entirely
    """

    def __init__(
        self,
        sandbox_id: str,
        image: str = "agent-sandbox:latest",
        base_dir: str = "/var/sandboxes",
        cpu_limit: float = 1.0,       # Number of CPU cores
        mem_limit: str = "512m",      # Memory limit
        network_enabled: bool = False, # Block network by default
    ):
        self.sandbox_id = sandbox_id
        self.image = image
        self.workspace_host = Path(base_dir) / sandbox_id / "workspace"
        self.workspace_container = "/workspace"  # Path INSIDE the container
        self.cpu_limit = cpu_limit
        self.mem_limit = mem_limit
        self.network_enabled = network_enabled
        self._docker = aiodocker.Docker()
        self._container = None

    # ─── Lifecycle ───

    async def setup(self):
        # Create workspace on host
        self.workspace_host.mkdir(parents=True, exist_ok=True)

        # Create the container with resource limits and the bind mount
        self._container = await self._docker.containers.create_or_replace(
            name=f"sandbox-{self.sandbox_id}",
            config={
                "Image": self.image,
                "Cmd": ["sleep", "infinity"],  # Keep alive for exec
                "HostConfig": {
                    "Binds": [f"{self.workspace_host}:{self.workspace_container}"],
                    "NanoCpus": int(self.cpu_limit * 1e9),
                    "Memory": self._parse_mem(self.mem_limit),
                    "NetworkMode": "none" if not self.network_enabled else "bridge",
                },
                "WorkingDir": self.workspace_container,
            },
        )
        await self._container.start()

    async def teardown(self):
        if self._container:
            await self._container.kill()
            await self._container.delete(force=True)
        # Clean up host workspace
        if self.workspace_host.exists():
            shutil.rmtree(self.workspace_host.parent)

    # ─── Filesystem (through the bind mount — fast, no docker cp) ───

    def _resolve(self, path: str) -> Path:
        # Same containment check as LocalSandbox, but on the host mount path
        resolved = (self.workspace_host / path).resolve()
        if not str(resolved).startswith(str(self.workspace_host.resolve())):
            raise ValueError(f"Path traversal blocked: '{path}'")
        return resolved

    # read_file, write_file, list_dir, file_exists, delete →
    # IDENTICAL to LocalSandbox (they go through the bind mount on the host)
    # In practice, you'd extract a _HostFilesystemMixin for DRY.

    async def read_file(self, path: str) -> str:
        resolved = self._resolve(path)
        async with aiofiles.open(resolved, "r") as f:
            return await f.read()

    async def write_file(self, path: str, content: str):
        resolved = self._resolve(path)
        resolved.parent.mkdir(parents=True, exist_ok=True)
        async with aiofiles.open(resolved, "w") as f:
            await f.write(content)

    # ... list_dir, file_exists, delete: same as LocalSandbox ...

    # ─── Command Execution (via docker exec — ISOLATED) ───

    async def exec(self, command, timeout=30.0, cwd=None, env=None):
        work_dir = f"{self.workspace_container}/{cwd}" if cwd else self.workspace_container

        # Build the docker exec command
        exec_config = {
            "Cmd": ["sh", "-c", command],
            "AttachStdout": True,
            "AttachStderr": True,
            "WorkingDir": work_dir,
        }
        if env:
            exec_config["Env"] = [f"{k}={v}" for k, v in env.items()]

        start = time.monotonic()
        # This runs INSIDE the container — can't see host filesystem
        exec_instance = await self._container.exec(exec_config)
        try:
            # Collect output with timeout
            stream = exec_instance.start()
            output = await asyncio.wait_for(stream.read_out(), timeout=timeout)
            inspect = await exec_instance.inspect()

            return ExecResult(
                exit_code=inspect["ExitCode"],
                stdout=output.decode(errors="replace") if output else "",
                duration_ms=(time.monotonic() - start) * 1000,
            )
        except asyncio.TimeoutError:
            return ExecResult(exit_code=-1, timed_out=True,
                              duration_ms=(time.monotonic() - start) * 1000)

The crucial architectural observation is in the asymmetry between filesystem operations and command execution. File operations go through the bind mount — the host reads and writes to the workspace directory, and the container sees those changes immediately because it's the same physical directory. This is fast (no copying) and simple (no Docker API needed for file I/O). Command execution goes through docker exec — the command runs inside the container's isolated process namespace, with no access to host processes or the host filesystem outside the mount. This split gives you the best of both worlds: fast file I/O and isolated execution.

Fig 2 — File I/O is fast (through the mount); execution is isolated (through docker exec)
🧩 The Bind Mount Insight

The bind mount is what makes the Docker sandbox practical for agentic workloads. Without it, every read_file and write_file would require a docker cp round-trip — copying bytes in and out of the container, adding seconds of latency to every file operation. With the bind mount, file I/O is local-speed on the host, and the container sees changes instantly. Execution is the only thing that crosses the Docker API boundary. This asymmetry — fast filesystem, isolated execution — is exactly what agent tools need.

Chapter 6 — Level 3: Cloud Sandboxes
Remote Execution

E2BSandbox — Everything Over the Network

Eventually you need sandboxes that run on entirely separate machines. Maybe you need GPUs. Maybe you need sandboxes in the same region as the user. Maybe you need to scale to thousands of concurrent agents and your Docker host can only run fifty containers. E2B (and similar cloud sandbox providers) give you on-demand VMs via an API.

Isolation Full VM — separate machine
Security VM boundary, network isolation
Setup Time ~5–15s (VM boot)
Best For Scale, GPU, full isolation
Developer's inner voice "This is where the Sandbox abstraction really proves its worth. With E2B, there's no bind mount. There's no shared filesystem. from e2b_code_interpreter import AsyncSandbox as E2BClient

class E2BSandbox(Sandbox):
    """Remote cloud sandbox via E2B.
    
    Every operation is an API call to a remote VM.
    No shared filesystem — files are transferred over the network.
    Full VM isolation. Billed per second of usage.
    
    The key design difference from Docker: there is no bind mount.
    File operations require explicit network transfer. This means
    we need to be smart about caching and batching — but the tools
    don't know or care about any of this.
    """

    def __init__(
        self,
        sandbox_id: str,
        template: str = "base",         # E2B sandbox template
        api_key: str = "",
        timeout: int = 300,               # Sandbox lifetime in seconds
    ):
        self.sandbox_id = sandbox_id
        self.template = template
        self.api_key = api_key
        self.timeout = timeout
        self._sbx: E2BClient | None = None

    # ─── Lifecycle ───

    async def setup(self):
        # Creates a fresh VM in the cloud — takes a few seconds
        self._sbx = await E2BClient.create(
            template=self.template,
            api_key=self.api_key,
            timeout=self.timeout,
        )

    async def teardown(self):
        if self._sbx:
            await self._sbx.kill()

    # ─── Filesystem (all network calls — no local disk) ───

    async def read_file(self, path: str) -> str:
        # API call to the remote VM's filesystem
        content = await self._sbx.files.read(path)
        return content

    async def write_file(self, path: str, content: str):
        await self._sbx.files.write(path, content)

    async def read_file_bytes(self, path: str) -> bytes:
        return await self._sbx.files.read_bytes(path)

    async def write_file_bytes(self, path: str, content: bytes):
        await self._sbx.files.write_bytes(path, content)

    async def list_dir(self, path: str = ".") -> list[FileEntry]:
        entries = await self._sbx.files.list(path)
        return [
            FileEntry(name=e.name, is_dir=e.is_dir, size_bytes=e.size or 0)
            for e in entries
        ]

    async def file_exists(self, path: str) -> bool:
        try:
            await self._sbx.files.read(path)
            return True
        except Exception:
            return False

    async def delete(self, path: str) -> bool:
        await self._sbx.files.remove(path)
        return True

    # ─── Command Execution (API call to remote VM) ───

    async def exec(self, command, timeout=30.0, cwd=None, env=None):
        start = time.monotonic()
        result = await self._sbx.commands.run(
            command,
            cwd=cwd or "/home/user",
            envs=env or {},
            timeout=timeout,
        )
        return ExecResult(
            exit_code=result.exit_code,
            stdout=result.stdout,
            stderr=result.stderr,
            duration_ms=(time.monotonic() - start) * 1000,
        )

    async def exec_stream(self, command, timeout=30.0, cwd=None, env=None):
        async for event in self._sbx.commands.stream(
            command, cwd=cwd, envs=env, timeout=timeout
        ):
            if event.type == "stdout":
                yield event.data

The most profound thing about this implementation is how different it is internally from the Docker sandbox. There's no bind mount. There's no _resolve() path containment check (the VM is the containment). Every file operation is a network round-trip. And yet, every tool works identically because the interface is the same. This is the Repository pattern applied to execution environments — the abstraction hides not just different storage locations, but entirely different computational models.

async def create_sandbox(settings: Settings, session_id: str) -> Sandbox:
    """Factory: create a sandbox from config."""

    match settings.sandbox_backend:
        case "local":
            sandbox = LocalSandbox(
                sandbox_id=session_id,
                base_dir=settings.sandbox_base_dir,
            )
        case "docker":
            sandbox = DockerSandbox(
                sandbox_id=session_id,
                image=settings.sandbox_docker_image,
                cpu_limit=settings.sandbox_cpu_limit,
                mem_limit=settings.sandbox_mem_limit,
                network_enabled=settings.sandbox_network,
            )
        case "e2b":
            sandbox = E2BSandbox(
                sandbox_id=session_id,
                template=settings.e2b_template,
                api_key=settings.e2b_api_key,
            )
        case _:
            raise ValueError(f"Unknown sandbox: {settings.sandbox_backend}")

    return sandbox

Chapter 8 — Connecting the Dots
Composing With the Media Backend

How the Sandbox Relates to the File Cache and Storage Backend

If you've been following the earlier diaries, you might be wondering: where does the LocalFileCache from Part II fit now that we have sandboxes? The answer requires distinguishing between two different kinds of files in the system.

Working files are files the agent creates and manipulates during its session — source code it's writing, scripts it's running, data it's generating. These live inside the sandbox. The sandbox is their only home during the session. They're created by sandbox.write_file(), read by sandbox.read_file(), and destroyed when sandbox.teardown() runs.

Uploaded files are files the user uploaded (PDFs, images, spreadsheets) or files that need to persist across sessions. These live in the storage backend (S3, local disk via FileStorageBackend). They're identified by file_id and managed by the media system.

The bridge between them is straightforward. When the agent needs to work with an uploaded file, the system materializes it into the sandbox. When the agent produces a file that should persist, the system extracts it from the sandbox into the storage backend.

sandbox/bridge.py The Bridge 
class SandboxFileBridge:
    """Moves files between the storage backend and the sandbox.
    
    Inbound:  storage backend → sandbox (user uploads available to tools)
    Outbound: sandbox → storage backend (agent outputs persisted for later)
    """

    def __init__(self, storage: FileStorageBackend, metadata_store: FileMetadataStore):
        self.storage = storage
        self.metadata_store = metadata_store

    async def materialize_into_sandbox(
        self, file_id: str, sandbox: Sandbox, sandbox_path: str,
    ):
        """Download a file from storage and place it in the sandbox."""
        meta = await self.metadata_store.get(file_id)
        content = await self.storage.retrieve(file_id, meta.agent_uuid)
        await sandbox.write_file_bytes(sandbox_path, content)

    async def persist_from_sandbox(
        self, sandbox: Sandbox, sandbox_path: str, agent_uuid: str,
    ) -> FileMetadata:
        """Extract a file from the sandbox and store it in the backend."""
        content = await sandbox.read_file_bytes(sandbox_path)
        filename = sandbox_path.split("/")[-1]
        file_id = f"file_{uuid.uuid4().hex[:16]}"

        meta = await self.storage.store(
            file_id=file_id, filename=filename,
            content=content, agent_uuid=agent_uuid,
        )
        await self.metadata_store.save(meta)
        return meta
Developer's inner voice "So what happens to the LocalFileCache? It's still useful — but its role shifts. For a LocalSandbox or DockerSandbox (where the workspace is on the host), the cache can serve as the materializer: it downloads files from the storage backend to the host, and the sandbox workspace can be the same directory or linked. For an E2BSandbox (where the workspace is remote), the cache is bypassed — the bridge calls storage.retrieve() and then sandbox.write_file_bytes() to push the bytes directly to the remote VM. The cache was always an optimization for local I/O; when I/O is inherently remote, it doesn't help."

Epilogue — The Architecture Map
Reflection

What Changed, What Didn't, and Why

Pattern Where What It Resolves
Strategy Sandbox interface + 3 implementations The execution environment is a pluggable strategy. Swap local → Docker → E2B without touching tools or the agent loop.
Dependency Inversion Tools depend on Sandbox, not os / subprocess High-level modules (tools) depend on an abstraction (Sandbox), not low-level modules (the OS). This makes tools portable across execution environments.
Path Containment _resolve() in Local and Docker sandboxes All relative paths are resolved and checked for containment before any I/O. Blocks path traversal. Not needed in E2B (the VM is the boundary).
Asymmetric I/O (Docker) Bind mount for files, docker exec for commands File operations are fast (local disk through the mount). Command execution is isolated (inside the container). Best of both worlds.
Bridge SandboxFileBridge Moves files between two separate systems (the durable storage backend and the ephemeral sandbox) that don't directly know about each other.
Context Manager Lifecycle async with sandbox Setup and teardown are tied to scope. Whether it's a 1ms mkdir or a 10s container start, the lifecycle is managed by the with block. Resource leaks are impossible.
🔑 The Unifying Insight

The entire sandbox architecture flows from one question: "What does a tool actually need from the world?" The answer was surprisingly small — read a file, write a file, list a directory, run a command. Not "access the filesystem." Not "create a subprocess." Just those four verbs. Once you have the verbs, the implementations are just different translations of those verbs into different environments. LocalSandbox translates them into open() and subprocess.run(). DockerSandbox translates file verbs into bind-mount I/O and command verbs into docker exec. E2BSandbox translates everything into HTTP API calls to a remote VM. The tools say what they want. The sandbox figures out how to get it done. That's the whole pattern.

✅ The Litmus Test

Three scenarios verify the factoring. "Add a new sandbox backend (Firecracker microVMs)?" — One new file implementing the Sandbox interface. Zero tool changes. "Add a new tool (apply_patch)?" — One new file that calls sandbox.read_file() and sandbox.write_file(). Zero sandbox changes. "Switch an entire deployment from Docker to E2B?" — One environment variable: SANDBOX_BACKEND=e2b. Zero code changes. Each axis — sandbox backends, tool implementations, deployment configuration — changes independently.

Tools should say what they want, not how to get it.
The sandbox is the translator between intent and operating system.
Change the translator, never the speaker.

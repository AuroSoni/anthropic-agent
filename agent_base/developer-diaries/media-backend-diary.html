Designing a Media Backend
for an Agentic AI System

A Developer Diary — From Hardcoded Paths to Pluggable Storage

The Setup

Why Files in an Agentic System Are Uniquely Hard

In a normal web app, file uploads are straightforward — accept the file, put it on S3, return a URL. But an agentic AI system has a constellation of consumers that all need the same file in different forms, at different times, through different channels.

A user uploads a PDF. The LLM provider adapter needs it as base64 bytes in the API request. The conversation history needs a lightweight reference for the database. The frontend needs a URL to render a preview. The tool execution sandbox might need it as a local file path. A sub-agent might need to pass it to another LLM with different format requirements. And if you're streaming the agent's work to the frontend in real time, you need to send image references before the agent turn is complete.

Developer's inner voice "This isn't a file upload problem. It's a reference management problem. The file itself is just bytes sitting somewhere. What matters is having a stable identity for those bytes and being able to resolve that identity into whatever representation each consumer needs."

Let's start where every project starts — with the simplest thing that works — and watch the pain accumulate until it forces an architecture.

Chapter 1 — The Naive Version
Day 1

Just Save the Bytes, Figure It Out Later

Day one, you write the most direct code possible. When a user uploads an image, save it to disk. When the LLM needs it, read it back. When the frontend needs it, serve it from the same disk path.

file_utils.py V1 — Hardcoded 
import os, base64, uuid

# Just... save the file somewhere
UPLOAD_DIR = "/tmp/agent_uploads"

def save_upload(filename: str, data: bytes) -> str:
    """Save a file. Return the path."""
    file_id = f"{uuid.uuid4().hex[:12]}_{filename}"
    path = os.path.join(UPLOAD_DIR, file_id)
    os.makedirs(UPLOAD_DIR, exist_ok=True)
    with open(path, "wb") as f:
        f.write(data)
    return path

def load_as_base64(path: str) -> str:
    """Read a file back as base64 for the LLM API."""
    with open(path, "rb") as f:
        return base64.standard_b64encode(f.read()).decode()

def get_serve_url(path: str) -> str:
    """Build a URL for the frontend. Hope the path matches the static mount."""
    filename = os.path.basename(path)
    return f"/static/uploads/{filename}"   # 🤞 fingers crossed

This works on your laptop. Then three things happen in the same week, and each one breaks it in a different way.

Fig 1 — Three failures that expose three separate coupling points
🚨 The Core Realization

Your code has three separate concerns welded together: where bytes live (local filesystem path), how bytes are identified (filesystem path is the ID), and how bytes are accessed (direct filesystem read, static URL path). Changing any one of these forces changes in the other two. A senior developer recognizes this as a classic case for the Repository pattern — abstract storage behind a stable identity scheme, and resolve access format at the boundary.

Chapter 2 — Identity First
The Key Architectural Decision

A File's Identity Must Be Independent of Its Location

Before writing any storage code, we need to get one thing right: how files are identified. This is the decision that everything else hangs on, and getting it wrong creates pain that propagates through every layer of the system.

Developer's inner voice "In the naive version, a file's identity was its filesystem path. That's why scaling broke — the path only works on the machine that wrote the file. I need an identity that's location-independent. A UUID-based file_id that's generated at upload time, stored in the database alongside the file's metadata, and used everywhere internally. The actual bytes could be on local disk, S3, GCS, or Mars — nobody cares, because they reference the file by its abstract ID."

The file metadata record is the anchor of the entire system. It's what gets stored in the database. Every other operation — storage, retrieval, URL generation — starts from this record.

models.py Identity Model 
from dataclasses import dataclass, field
from datetime import datetime, timezone
import uuid

@dataclass
class FileMetadata:
    """The identity record for a file in the system.
    
    This is what gets persisted in the database. It describes the file
    without containing the file. The actual bytes live in whatever storage
    backend is configured.
    """

    # ── Identity (immutable once created) ──
    file_id: str = field(default_factory=lambda: f"file_{uuid.uuid4().hex[:16]}")
    filename: str = ""           # Original user-supplied filename
    mime_type: str = ""          # MIME type (image/png, application/pdf, etc.)
    size_bytes: int = 0          # File size in bytes
    checksum: str = ""           # SHA-256 content hash for integrity/dedup

    # ── Ownership ──
    agent_uuid: str = ""         # Which agent session this file belongs to
    uploaded_by: str = ""        # "user" | "tool:{tool_name}" | "agent"
    
    # ── Storage location (set by the backend after storing) ──
    storage_backend: str = ""    # "local" | "s3" | "gcs" | "azure" — which backend holds the bytes
    storage_key: str = ""        # Backend-specific locator (filesystem path, S3 key, blob name)
    storage_bucket: str = ""     # For object stores: the bucket/container name

    # ── Lifecycle ──
    created_at: str = field(
        default_factory=lambda: datetime.now(timezone.utc).isoformat()
    )
    expires_at: str | None = None   # TTL for auto-cleanup (None = permanent)
    is_deleted: bool = False         # Soft delete flag

    # ── Extension point ──
    extras: dict = field(default_factory=dict)  # Tags, labels, custom metadata

    def to_dict(self) -> dict:
        return {
            "file_id": self.file_id,
            "filename": self.filename,
            "mime_type": self.mime_type,
            "size_bytes": self.size_bytes,
            "checksum": self.checksum,
            "agent_uuid": self.agent_uuid,
            "uploaded_by": self.uploaded_by,
            "storage_backend": self.storage_backend,
            "storage_key": self.storage_key,
            "storage_bucket": self.storage_bucket,
            "created_at": self.created_at,
            "expires_at": self.expires_at,
            "is_deleted": self.is_deleted,
            "extras": self.extras,
        }
🧩 The Separation of Identity from Location

Notice the split: file_id is the stable identity that everything else in the system references. storage_backend + storage_key + storage_bucket are the location details that only the storage backend cares about. The rest of the system — canonical content blocks, conversation history, tool results — only ever stores file_id. This is the same principle as a database primary key: the ID is stable, the physical storage location is an implementation detail.

Chapter 3 — The Repository
The Abstract Contract

FileStorageBackend — Where Bytes Go To Live

Now we define what it means to "store a file" and "retrieve a file" in the abstract. This is the Repository pattern applied to binary objects. The interface says what operations are available; the implementations decide where and how.

Developer's inner voice "I need five operations: store, retrieve, delete, check-exists, and generate a URL. That last one is important — URL generation is backend-specific. A local backend returns a path-based URL served by your API. An S3 backend returns a presigned URL. A GCS backend returns a signed URL. The URL shape is fundamentally tied to the storage backend, so it belongs on this interface, not on a separate service."
storage/base.py Abstract Contract 
from abc import ABC, abstractmethod


class FileStorageBackend(ABC):
    """Abstract repository for binary file storage.
    
    Implementors handle the actual byte persistence — local disk, S3,
    GCS, Azure Blob, etc. The rest of the system interacts with files
    only through file_id references and this interface.
    
    Contract guarantees:
      - store() returns FileMetadata with storage_* fields populated
      - retrieve() returns raw bytes given a file_id
      - get_url() returns a consumer-ready URL (presigned if needed)
      - All operations are async for I/O-bound backends
    """

    # ─── Core Operations ───

    @abstractmethod
    async def store(
        self,
        file_id: str,
        filename: str,
        content: bytes,
        agent_uuid: str,
        mime_type: str = "",
    ) -> FileMetadata:
        """Store file bytes. Return metadata with storage location populated.
        
        The backend decides the storage_key (path/object name) internally.
        The caller provides the file_id — it is the stable identity.
        """
        ...

    @abstractmethod
    async def retrieve(self, file_id: str, agent_uuid: str) -> bytes:
        """Retrieve raw file bytes by file_id.
        
        Raises FileNotFoundError if the file doesn't exist.
        """
        ...

    @abstractmethod
    async def delete(self, file_id: str, agent_uuid: str) -> bool:
        """Delete a file's bytes from storage. Return True if deleted."""
        ...

    @abstractmethod
    async def exists(self, file_id: str, agent_uuid: str) -> bool:
        """Check whether a file exists in this backend."""
        ...

    # ─── URL Generation (backend-specific) ───

    @abstractmethod
    async def get_url(
        self,
        file_id: str,
        agent_uuid: str,
        expires_in: int = 3600,
    ) -> str:
        """Generate a consumer-ready URL for this file.
        
        For local backends: returns an API-served path like /files/{file_id}
        For S3: returns a presigned URL valid for `expires_in` seconds
        For GCS: returns a signed URL
        
        This is the one method that turns the abstract file_id into
        something the frontend or another service can actually fetch.
        """
        ...

    # ─── Lifecycle Helpers (optional overrides) ───

    async def cleanup_expired(self, agent_uuid: str) -> int:
        """Delete files past their expiry. Return count deleted.
        Default: no-op (override for backends that support TTL).
        """
        return 0

    async def cleanup_agent(self, agent_uuid: str) -> int:
        """Delete ALL files for an agent session. Return count deleted.
        Called when an agent session is terminated/archived.
        """
        return 0

Two design decisions here are worth calling out explicitly.

First, get_url() lives on the backend, not on a separate "URL service." The reason is that URL shape is inextricably tied to storage type. A local backend's URL goes through your own API server. An S3 URL is a presigned Amazon URL. A GCS URL is a Google signed URL. No amount of abstraction can make these identical — the backend is the authority on how to generate its own URLs. Trying to put URL generation elsewhere just creates a second class that needs to know which backend is active, duplicating the dispatch logic.

Second, agent_uuid appears on every method. Files are namespaced to agent sessions. This is critical for cleanup (when a session ends, you can delete all its files), for security (an agent can't access another agent's files), and for storage layout (each agent's files can live in a separate prefix/directory).

Chapter 4 — Implementations
Concrete Backends

Local Disk and S3 — Same Interface, Different Worlds

Implementation A: Local Filesystem

The local backend is the simplest. It's what you use in development and single-server deployments. Files go into a directory, organized by agent UUID.

storage/local.py Local Backend 
import hashlib, os, aiofiles

class LocalFileBackend(FileStorageBackend):
    """Stores files on the local filesystem.
    
    Layout: {base_dir}/{agent_uuid}/{file_id}/{filename}
    This nested layout makes per-agent cleanup a simple rmtree.
    """

    def __init__(self, base_dir: str, serve_url_prefix: str = "/api/files"):
        self.base_dir = base_dir
        self.serve_url_prefix = serve_url_prefix

    def _path(self, agent_uuid: str, file_id: str, filename: str = "") -> str:
        return os.path.join(self.base_dir, agent_uuid, file_id, filename)

    async def store(self, file_id, filename, content, agent_uuid, mime_type="") -> FileMetadata:
        dir_path = self._path(agent_uuid, file_id)
        os.makedirs(dir_path, exist_ok=True)

        file_path = os.path.join(dir_path, filename)
        async with aiofiles.open(file_path, "wb") as f:
            await f.write(content)

        return FileMetadata(
            file_id=file_id,
            filename=filename,
            mime_type=mime_type,
            size_bytes=len(content),
            checksum=hashlib.sha256(content).hexdigest(),
            agent_uuid=agent_uuid,
            storage_backend="local",
            storage_key=file_path,  # Full filesystem path
        )

    async def retrieve(self, file_id, agent_uuid) -> bytes:
        # Find the file in the file_id directory
        dir_path = self._path(agent_uuid, file_id)
        if not os.path.isdir(dir_path):
            raise FileNotFoundError(f"No file with id {file_id}")
        entries = os.listdir(dir_path)
        if not entries:
            raise FileNotFoundError(f"Empty directory for {file_id}")
        async with aiofiles.open(os.path.join(dir_path, entries[0]), "rb") as f:
            return await f.read()

    async def get_url(self, file_id, agent_uuid, expires_in=3600) -> str:
        # Local: route through the API server's file-serving endpoint
        return f"{self.serve_url_prefix}/{agent_uuid}/{file_id}"

    async def delete(self, file_id, agent_uuid) -> bool:
        import shutil
        dir_path = self._path(agent_uuid, file_id)
        if os.path.isdir(dir_path):
            shutil.rmtree(dir_path)
            return True
        return False

    async def exists(self, file_id, agent_uuid) -> bool:
        return os.path.isdir(self._path(agent_uuid, file_id))

    async def cleanup_agent(self, agent_uuid) -> int:
        import shutil
        agent_dir = os.path.join(self.base_dir, agent_uuid)
        if os.path.isdir(agent_dir):
            count = len(os.listdir(agent_dir))
            shutil.rmtree(agent_dir)
            return count
        return 0

Implementation B: Amazon S3

The S3 backend is what you use in production and multi-replica deployments. Files are objects in a bucket, and URLs are presigned for time-limited access.

storage/s3.py S3 Backend 
import hashlib
from aiobotocore.session import get_session

class S3FileBackend(FileStorageBackend):
    """Stores files in an S3 bucket.
    
    Object key layout: {prefix}/{agent_uuid}/{file_id}/{filename}
    This mirrors the local layout for consistency.
    """

    def __init__(
        self,
        bucket: str,
        prefix: str = "agent-files",
        region: str = "us-east-1",
    ):
        self.bucket = bucket
        self.prefix = prefix
        self.region = region
        self._session = get_session()

    def _key(self, agent_uuid: str, file_id: str, filename: str) -> str:
        return f"{self.prefix}/{agent_uuid}/{file_id}/{filename}"

    async def store(self, file_id, filename, content, agent_uuid, mime_type="") -> FileMetadata:
        key = self._key(agent_uuid, file_id, filename)
        
        async with self._session.create_client("s3", region_name=self.region) as s3:
            await s3.put_object(
                Bucket=self.bucket,
                Key=key,
                Body=content,
                ContentType=mime_type or "application/octet-stream",
            )

        return FileMetadata(
            file_id=file_id,
            filename=filename,
            mime_type=mime_type,
            size_bytes=len(content),
            checksum=hashlib.sha256(content).hexdigest(),
            agent_uuid=agent_uuid,
            storage_backend="s3",
            storage_key=key,          # S3 object key
            storage_bucket=self.bucket,
        )

    async def retrieve(self, file_id, agent_uuid) -> bytes:
        # Look up the storage_key from metadata (or reconstruct from listing)
        async with self._session.create_client("s3", region_name=self.region) as s3:
            # List objects with the file_id prefix to find the filename
            prefix = f"{self.prefix}/{agent_uuid}/{file_id}/"
            resp = await s3.list_objects_v2(Bucket=self.bucket, Prefix=prefix, MaxKeys=1)
            contents = resp.get("Contents", [])
            if not contents:
                raise FileNotFoundError(f"No S3 object for {file_id}")
            obj = await s3.get_object(Bucket=self.bucket, Key=contents[0]["Key"])
            return await obj["Body"].read()

    async def get_url(self, file_id, agent_uuid, expires_in=3600) -> str:
        # S3: generate a presigned URL — no auth needed by the consumer
        prefix = f"{self.prefix}/{agent_uuid}/{file_id}/"
        async with self._session.create_client("s3", region_name=self.region) as s3:
            resp = await s3.list_objects_v2(Bucket=self.bucket, Prefix=prefix, MaxKeys=1)
            key = resp["Contents"][0]["Key"]
            url = await s3.generate_presigned_url(
                "get_object",
                Params={"Bucket": self.bucket, "Key": key},
                ExpiresIn=expires_in,
            )
        return url

    # ... delete, exists, cleanup_agent follow the same pattern ...

Chapter 5 — The Resolver
The Bridge Between Identities and Consumers

MediaResolver — Translating file_id Into What Each Consumer Needs

We now have a storage backend that can persist and retrieve bytes, and a metadata model that gives files a stable identity. The missing piece is the resolver — the service that translates a file_id into whatever format a specific consumer needs. The LLM provider adapter needs base64. The frontend needs a URL. The tool sandbox needs a local path. The conversation log needs just the file_id plus some display metadata.

Developer's inner voice "This is the Adapter pattern applied to media references. Each consumer has a different 'plug shape' they expect, and the resolver is the travel adapter that converts the universal file_id into the right shape. But it's a thin layer — it delegates to the storage backend for the actual work. It's less of a service and more of a convenience facade."
media_resolver.py The Adapter Layer 
import base64

class MediaResolver:
    """Resolves file_id references into consumer-specific formats.
    
    This is the single service that all consumers call when they need
    a file in a specific representation. It knows about the storage
    backend and the metadata store, and it adapts between them.
    
    It is NOT a storage system. It doesn't own bytes. It translates.
    """

    def __init__(
        self,
        storage: FileStorageBackend,
        metadata_store: FileMetadataStore,    # Your DB repository for FileMetadata
    ):
        self.storage = storage
        self.metadata_store = metadata_store

    # ── For the LLM provider adapter (Anthropic, OpenAI, etc.) ──

    async def to_base64(self, file_id: str, agent_uuid: str) -> dict:
        """Resolve file_id → base64 payload for LLM API requests.
        
        Returns a dict ready to be embedded in an API content block:
          {"data": "iVBORw0K...", "media_type": "image/png"}
        """
        meta = await self.metadata_store.get(file_id)
        raw_bytes = await self.storage.retrieve(file_id, agent_uuid)
        return {
            "data": base64.standard_b64encode(raw_bytes).decode("utf-8"),
            "media_type": meta.mime_type,
        }

    # ── For the frontend / conversation history ──

    async def to_url(self, file_id: str, agent_uuid: str, expires_in: int = 3600) -> str:
        """Resolve file_id → consumer-ready URL.
        
        Delegates to the storage backend's get_url() which knows whether
        to return a presigned S3 URL or a local API path.
        """
        return await self.storage.get_url(file_id, agent_uuid, expires_in)

    # ── For the conversation log (lightweight reference) ──

    async def to_reference(self, file_id: str) -> dict:
        """Resolve file_id → display-safe reference for the conversation log.
        
        Returns metadata sufficient for the UI to render a preview card
        without embedding the actual file bytes.
        """
        meta = await self.metadata_store.get(file_id)
        return {
            "file_id": meta.file_id,
            "filename": meta.filename,
            "mime_type": meta.mime_type,
            "size_bytes": meta.size_bytes,
            "url": await self.storage.get_url(file_id, meta.agent_uuid),
        }

    # ── Batch resolution (for messages with multiple media blocks) ──

    async def resolve_content_blocks(
        self,
        blocks: list,       # List of canonical ContentBlock instances
        agent_uuid: str,
        target: str = "base64",  # "base64" for LLM, "url" for frontend
    ) -> list:
        """Walk a list of ContentBlocks and resolve all file_id references.
        
        This is the workhorse method that the LLM provider adapter calls
        before building an API request, and that the conversation history
        serializer calls before sending data to the frontend.
        """
        resolved = []
        for block in blocks:
            if hasattr(block, "source_type") and block.source_type == SourceType.FILE_ID:
                if target == "base64":
                    payload = await self.to_base64(block.data, agent_uuid)
                    # Create a new block with resolved data
                    new_block = type(block)(
                        source_type=SourceType.BASE64,
                        media_type=payload["media_type"],
                        data=payload["data"],
                    )
                    resolved.append(new_block)
                elif target == "url":
                    url = await self.to_url(block.data, agent_uuid)
                    new_block = type(block)(
                        source_type=SourceType.URL,
                        media_type=block.media_type,
                        data=url,
                    )
                    resolved.append(new_block)
            else:
                resolved.append(block)  # Pass through non-media blocks
        return resolved

The key method is resolve_content_blocks(). It walks a list of canonical content blocks and resolves any FILE_ID references to the target format. The LLM provider adapter calls it with target="base64" right before building the API request. The conversation history serializer calls it with target="url" right before sending data to the frontend. The canonical content blocks themselves stay clean — they store file_id references, and resolution happens at the boundary.

Fig 3 — Files are ingested once, stored once, referenced by file_id, resolved at the boundary
💡 Why Resolution Happens at the Boundary

A common mistake is resolving file references eagerly — converting file_id to base64 the moment you create the content block. But that couples the canonical model to a specific consumer's format. If you resolve lazily at the boundary, the same ImageContent(source_type=FILE_ID) block can be projected into base64 for the LLM, a URL for the frontend, or a lightweight metadata dict for the conversation log, depending on who's consuming it and when. This is the same dual-projection insight from the tool result envelope pattern, applied to media references.

Chapter 6 — Composition
The Composition Root

Wiring It All Together at Startup

As with every pattern-based system, there is one place that knows about the concrete implementations and wires them together. This is the composition root — your main.py or dependency injection configuration.

main.py Composition Root 
from config import Settings

def create_file_system(settings: Settings):
    """Factory: build the entire file subsystem from config."""

    # ── Pick the storage backend from config ──
    match settings.file_storage_backend:
        case "local":
            storage = LocalFileBackend(
                base_dir=settings.local_storage_path,
                serve_url_prefix=settings.file_serve_prefix,
            )
        case "s3":
            storage = S3FileBackend(
                bucket=settings.s3_bucket,
                prefix=settings.s3_prefix,
                region=settings.s3_region,
            )
        case _:
            raise ValueError(f"Unknown backend: {settings.file_storage_backend}")

    # ── Build the metadata store (always a DB, regardless of byte storage) ──
    metadata_store = PostgresFileMetadataStore(settings.database_url)

    # ── Compose the resolver ──
    resolver = MediaResolver(storage=storage, metadata_store=metadata_store)

    return storage, metadata_store, resolver


# ── In your .env or settings: ──
# FILE_STORAGE_BACKEND=local         # Development
# FILE_STORAGE_BACKEND=s3            # Production
# S3_BUCKET=my-client-bucket-eu      # Enterprise client's own bucket

The enterprise compliance requirement from the Monday breakage? It's now a one-line config change. The client provides their S3 bucket name, you point the config at it, and their files never touch your infrastructure. The code doesn't change. The resolver doesn't change. The canonical model doesn't change. Only the bytes flow to a different location.

Chapter 7 — Testing
The Payoff

In-Memory Backend for Testing

The pattern's real payoff is testability. You don't need Docker, S3 emulators, or temp directories. You write a MemoryFileBackend that stores bytes in a dictionary and run your entire test suite in milliseconds.

storage/memory.py Test Backend 
class MemoryFileBackend(FileStorageBackend):
    """In-memory backend for unit tests. No I/O, no cleanup."""

    def __init__(self):
        self._store: dict[str, bytes] = {}         # file_id → bytes
        self._meta: dict[str, FileMetadata] = {}   # file_id → metadata

    async def store(self, file_id, filename, content, agent_uuid, mime_type="") -> FileMetadata:
        self._store[file_id] = content
        meta = FileMetadata(
            file_id=file_id, filename=filename, mime_type=mime_type,
            size_bytes=len(content), agent_uuid=agent_uuid,
            storage_backend="memory", storage_key=file_id,
        )
        self._meta[file_id] = meta
        return meta

    async def retrieve(self, file_id, agent_uuid) -> bytes:
        if file_id not in self._store:
            raise FileNotFoundError(file_id)
        return self._store[file_id]

    async def get_url(self, file_id, agent_uuid, expires_in=3600) -> str:
        return f"memory://{file_id}"   # Fake URL for assertions

    async def exists(self, file_id, agent_uuid) -> bool:
        return file_id in self._store

    async def delete(self, file_id, agent_uuid) -> bool:
        return self._store.pop(file_id, None) is not None

@pytest.mark.asyncio
async def test_store_and_resolve_to_base64():

✅ The Litmus Test

Three questions to verify the factoring is right. "If I add a new storage backend (GCS), how many files do I touch?" One — the new backend implementation. "If I add a new consumer format (e.g., a thumbnail URL), how many files do I touch?" One — add a method to MediaResolver. "If I change the file identity scheme (e.g., content-addressed hashing), how many files do I touch?" One — the FileMetadata model and the ID generation logic. Each axis of change is isolated.

Naming the Patterns We Discovered

As before, we didn't start with patterns — we started with three breakages in one week. The patterns emerged from resolving the specific tensions. Let's name them.

Pattern Where What It Resolves
Repository FileStorageBackend Abstracts where bytes live. Local disk, S3, GCS, memory — the rest of the system doesn't know or care.
Identity Map FileMetadata + file_id Separates a file's stable identity from its physical location. The same file_id works regardless of which backend stores the bytes.
Adapter / Resolver MediaResolver Translates the universal file_id reference into consumer-specific formats (base64, URL, metadata dict). Each consumer gets its own "plug shape."
Lazy Resolution source_type=FILE_ID in content blocks Content blocks store abstract references, not concrete data. Resolution to base64/URL happens at the boundary when a specific consumer needs it, not at creation time.
Composition Root create_file_system() One function that wires the concrete backend choice from config. Change storage for an entire deployment with one environment variable.
🔑 The Unifying Insight

Everything in this system flows from one principle: separate identity from location from representation. The file_id is the identity — immutable, universal, stored everywhere. The storage backend manages the location — where bytes physically sit. The resolver manages the representation — how each consumer sees the file. These three concerns change for different reasons, at different times, triggered by different events. Braiding them together (as the naive version did) means every change affects everything. Separating them means each change is local.

The best file systems are invisible.
Every consumer says "give me this file" and gets exactly what it needs,
without knowing where the bytes came from or how they got there.

The Incident Report

Monday Morning: Two Problems Wearing the Same Disguise

Your media backend from Part I works beautifully for profile pictures, small tool screenshots, and short documents. Then a user uploads a 340MB engineering schematic PDF, and a separate user's agent tries to grep through a 50MB log file stored on S3. Both operations fail, but for completely different reasons that expose two latent weaknesses in the architecture.

The PDF upload crashes the API server. Your store() method accepts bytes, which means the entire 340MB file must sit in Python process memory at once. The ASGI server reads the upload body into RAM, your endpoint reads it again into a bytes object, and you pass that to the storage backend which reads it again to compute the SHA-256 checksum. For a brief, catastrophic moment, you have three copies of a 340MB file in memory — over a gigabyte of RAM consumed by a single upload. The OOM killer intervenes.

The grep operation doesn't crash — it just takes 12 seconds. The file_read tool calls storage.retrieve(file_id), which downloads the entire 50MB from S3 into memory, writes it to a temp file, then grep reads that file. Next tool call: file_read on the same file — another 12-second download. Developer's inner voice "These are two separate problems. The upload crash is about memory pressure — I'm holding entire files in memory as bytes objects, which doesn't scale. The grep latency is about I/O locality — tools need files to be locally available, but my storage backend might be in another datacenter. The first problem needs streaming. The second needs caching. Let me tackle them separately."
🚨 The Two Problems

Problem 1 (Memory): The store(content: bytes) signature forces the entire file into process memory. For large files, this means multiple copies in RAM simultaneously, leading to OOM crashes. We need a streaming interface that never holds the full file in memory. Problem 2 (Latency): Tools that operate on files (read, write, grep, patch) need filesystem-level access. When storage is remote (S3), every tool invocation pays the full download cost, even for files already downloaded seconds ago. We need a local cache that materializes remote files on first access and serves them from disk on subsequent accesses.

Chapter 1 — The Bytes Problem
Reasoning About Memory

Why bytes Works Until It Doesn't

Before we jump to solutions, let's understand exactly what Python gives us to work with. The standard library has a hierarchy of file-like abstractions, and knowing the trade-offs between them is what separates a fix that works from a fix that's appropriate.

bytes is the simplest — an immutable blob in memory. It's perfect when the entire content fits comfortably in RAM (say, under 10MB). Your tool result screenshots, small CSVs, and agent-generated code files are all fine as bytes. The problem only appears when a single file's size becomes a meaningful fraction of your process memory.

BinaryIO is Python's protocol for anything that looks like an open file — it has .read(), .write(), .seek(). It doesn't promise where the bytes live. The backing store could be RAM (via BytesIO), a real file on disk, a network socket, or a temporary file. When you accept BinaryIO instead of bytes, you gain the ability to process data in chunks without holding it all in memory at once.

SpooledTemporaryFile is Python's built-in hybrid. It starts as an in-memory buffer (like BytesIO) and automatically "spills" to a real temp file on disk once it exceeds a configured size threshold. This is clever because small files get the speed of RAM, and large files get the safety of disk, with the crossover happening transparently.

Developer's inner voice "SpooledTemporaryFile sounds perfect, but I need to think about what I'm actually optimizing for. My concern isn't just 'don't crash on large files.' It's that the upload data arrives as a stream from the HTTP body, gets stored into a backend that might also want a stream (S3's put_object can accept streaming bodies), and gets checksummed along the way. The ideal architecture is a pipeline where bytes flow from source to destination, being checksummed as they pass through, with nothing ever holding the complete file. SpooledTemporaryFile is a buffer, not a pipeline. It's useful, but it's a building block, not the whole solution."
Fig 1 — Memory profiles for three approaches to a 340MB upload
💡 The Insight: Buffer vs Pipeline

SpooledTemporaryFile is a buffer — it accumulates the entire file (in RAM or on disk) before you can do anything with it. A streaming pipeline is a flow — bytes pass through from source to destination, being processed (checksummed, compressed, encrypted) as they move, with only a small chunk in memory at any instant. For the storage backend, we want the pipeline. For the local tool cache (Problem 2), the SpooledTemporaryFile actually turns out to be the perfect building block because we do want the complete file on disk for tools to read from.

Chapter 2 — Streaming Storage
Evolving the Contract

Updating FileStorageBackend for Streaming I/O

The key design decision is that we don't replace the bytes-based interface — we augment it. Small files (tool screenshots, generated code) should still be passable as bytes for convenience. But large files should be passable as streams. The interface accepts a union type and the backend handles both cases internally.

Developer's inner voice "I need to be careful about the type hierarchy here. In async Python, I can't just use BinaryIO because file reads are blocking. I need an async iteration protocol — something I can `async for chunk in source`. The ASGI request body is already an async iterator of chunks. S3's put_object can accept an async readable body in some clients, or I can use multipart upload with chunked writes. The abstraction I need is: 'something I can read chunks from asynchronously.'"
storage/types.py Type Definitions 
from typing import AsyncIterator, Protocol, Union, runtime_checkable

# The type that represents "something we can read bytes from in chunks"
# This matches what ASGI gives us, what aiofiles gives us, and what
# httpx/aiohttp response bodies give us.

@runtime_checkable
class AsyncByteStream(Protocol):
    """Anything that yields bytes chunks asynchronously."""
    def __aiter__(self) -> AsyncIterator[bytes]: ...
    async def __anext__(self) -> bytes: ...

# What store() accepts: either complete bytes or an async stream of chunks
FileContent = Union[bytes, AsyncByteStream]

# Default chunk size for streaming operations
DEFAULT_CHUNK_SIZE = 64 * 1024  # 64KB — sweet spot for I/O throughput vs memory

Now we update the abstract backend to accept FileContent instead of just bytes, and add a streaming retrieval method alongside the existing one.

storage/base.py Updated Contract 
class FileStorageBackend(ABC):

    # ─── Store: now accepts bytes OR async stream ───

    @abstractmethod
    async def store(
        self,
        file_id: str,
        filename: str,
        content: FileContent,          # ← was: bytes
        agent_uuid: str,
        mime_type: str = "",
        size_hint: int | None = None,  # ← optional: total size if known (for S3 content-length)
    ) -> FileMetadata:
        """Store file content. Accepts bytes (small files) or async stream (large files).
        
        When content is bytes, the backend can process it directly.
        When content is an AsyncByteStream, the backend must consume it
        chunk by chunk, never loading the full content into memory.
        
        size_hint is optional — some backends (S3) can use it to set
        Content-Length upfront. If not provided, the backend figures it out
        (e.g., by using multipart upload for S3).
        """
        ...

    # ─── Retrieve: full bytes (small files) ───

    @abstractmethod
    async def retrieve(self, file_id: str, agent_uuid: str) -> bytes:
        """Retrieve full file content. Fine for files < 50MB.
        For larger files, use retrieve_stream().
        """
        ...

    # ─── Retrieve stream: chunked async iteration (large files) ───

    @abstractmethod
    async def retrieve_stream(
        self, file_id: str, agent_uuid: str, chunk_size: int = DEFAULT_CHUNK_SIZE,
    ) -> AsyncIterator[bytes]:
        """Retrieve file content as an async stream of chunks.
        
        The caller iterates: `async for chunk in backend.retrieve_stream(...):`
        Each chunk is at most chunk_size bytes. Memory usage is O(chunk_size).
        """
        ...

    # ... get_url, delete, exists, cleanup — unchanged ...

The implementation in the local backend shows how the streaming store works in practice. The critical insight is the tee pattern — as chunks flow from the source to the destination file, they're simultaneously fed to the SHA-256 hasher, so the checksum is computed in the same pass without ever accumulating the full file in memory.

storage/local.py Streaming Store 
class LocalFileBackend(FileStorageBackend):

    async def store(self, file_id, filename, content, agent_uuid, mime_type="", size_hint=None):
        dir_path = self._path(agent_uuid, file_id)
        os.makedirs(dir_path, exist_ok=True)
        file_path = os.path.join(dir_path, filename)

        hasher = hashlib.sha256()
        total_bytes = 0

        if isinstance(content, bytes):
            # Small file: write in one shot (the common case for tool outputs)
            hasher.update(content)
            total_bytes = len(content)
            async with aiofiles.open(file_path, "wb") as f:
                await f.write(content)
        else:
            # Large file: stream chunk by chunk. The tee pattern —
            # each chunk is written to disk AND fed to the hasher
            # in the same pass. Peak memory: one chunk (~64KB).
            async with aiofiles.open(file_path, "wb") as f:
                async for chunk in content:
                    hasher.update(chunk)
                    total_bytes += len(chunk)
                    await f.write(chunk)

        return FileMetadata(
            file_id=file_id, filename=filename, mime_type=mime_type,
            size_bytes=total_bytes, checksum=hasher.hexdigest(),
            agent_uuid=agent_uuid,
            storage_backend="local", storage_key=file_path,
        )

    async def retrieve_stream(self, file_id, agent_uuid, chunk_size=DEFAULT_CHUNK_SIZE):
        # yield chunks from disk — never loads the whole file into memory
        dir_path = self._path(agent_uuid, file_id)
        entries = os.listdir(dir_path)
        if not entries:
            raise FileNotFoundError(file_id)
        async with aiofiles.open(os.path.join(dir_path, entries[0]), "rb") as f:
            while True:
                chunk = await f.read(chunk_size)
                if not chunk:
                    break
                yield chunk
🧩 The Tee Pattern

The name comes from the Unix tee command, which reads stdin and writes it to both stdout and a file simultaneously. Here, each chunk of the upload stream is simultaneously written to disk and fed to the SHA-256 hasher. This means that by the time the last byte is written, the checksum is already computed — no second pass over the data required. This same pattern applies to the S3 backend, where chunks can be fed into a multipart upload while being hashed, and to any future backend that supports streaming writes.

Chapter 3 — End-to-End Streaming
Connecting the Pipeline

From HTTP Body to Storage Backend Without Buffering

The streaming backend is only useful if the data actually flows through as a stream from end to end. If your FastAPI endpoint reads the entire request body into memory first, you've defeated the purpose. Here's how to wire the ASGI request body directly into the streaming store.

routes/upload.py FastAPI Endpoint 
from starlette.requests import Request

@router.post("/agent/{agent_uuid}/files")
async def upload_file(agent_uuid: str, request: Request):
    """Stream-through upload: ASGI body → storage backend.
    
    We do NOT use UploadFile here because it buffers the entire body
    via SpooledTemporaryFile internally. Instead, we read the raw ASGI
    request body as an async stream and pipe it directly to the backend.
    """
    content_type = request.headers.get("content-type", "")
    filename = request.headers.get("x-filename", "upload")
    file_id = f"file_{uuid.uuid4().hex[:16]}"

    # The request body IS an async iterator of bytes chunks.
    # We can pass it directly to the streaming store.
    async def body_stream():
        async for chunk in request.stream():
            yield chunk

    # Pass the async stream directly to storage — no buffering in this process
    meta = await storage.store(
        file_id=file_id,
        filename=filename,
        content=body_stream(),  # ← async generator, NOT bytes
        agent_uuid=agent_uuid,
        mime_type=content_type,
        size_hint=int(request.headers.get("content-length", 0)) or None,
    )

    # Save metadata to DB
    await metadata_store.save(meta)

    return {"file_id": meta.file_id, "size": meta.size_bytes}


# ── For small files from tool results, the simple bytes path still works: ──

async def store_tool_output(file_bytes: bytes, filename: str, agent_uuid: str):
    """Store a small file from a tool result. Uses the bytes path."""
    file_id = f"file_{uuid.uuid4().hex[:16]}"
    meta = await storage.store(
        file_id=file_id, filename=filename,
        content=file_bytes,  # ← plain bytes, the backend handles it
        agent_uuid=agent_uuid,
    )
    await metadata_store.save(meta)
    return meta

A subtle but important point: we're not using FastAPI's UploadFile parameter here. FastAPI's UploadFile internally uses SpooledTemporaryFile, which means the framework buffers the entire request body before your handler even runs. By dropping down to the raw Starlette Request, we get direct access to the ASGI body stream and can pipe it straight through to the storage backend. For a 340MB file, this is the difference between using ~64KB of RAM and using ~340MB of RAM.

A caveat worth noting "There's a trade-off here. Using the raw request stream means I can't easily retry the upload if the storage backend fails partway through — the bytes have already been consumed from the network socket and they're gone. For mission-critical uploads, I might want a SpooledTemporaryFile as a local staging buffer after all, especially if the destination is S3 and network errors are likely. The right answer depends on whether I'm optimizing for memory safety (raw stream) or reliability (staged buffer). For most agentic workloads where files are under 100MB, SpooledTemporaryFile with a 10MB threshold is the pragmatic middle ground. For genuinely large files (1GB+), the raw stream with retry-at-the-application-level is the only option."
Chapter 4 — The Local Cache
Problem 2: Tool Latency

LocalFileCache — Materializing Remote Files for Tool Access

Now let's tackle the second problem. Tools like file_read, file_write, grep, and apply_patch need files to exist as real files on the local filesystem. They can't work with S3 object keys or async streams — they need a /path/to/file that they can open, seek, and modify. When your storage backend is remote, you need a caching layer that downloads files on first access and serves them from the local disk on subsequent accesses.

Developer's inner voice "This is conceptually a read-through cache. The first time a tool asks for a file, the cache downloads it from the storage backend and writes it to a local directory. Every subsequent access within the same agent session returns the local path instantly. The tricky parts are: what happens when a tool modifies a file (write-back to the backend), what happens when the agent session ends (cleanup), and how to handle concurrent access from tools that might be running in parallel."

The cache is not a storage backend. It's a separate layer that sits between tools and the storage backend. Tools interact with the cache to get local filesystem paths. The cache interacts with the storage backend to fetch and sync bytes. This separation is important because the cache has a completely different lifecycle (ephemeral, per-session) and different semantics (read-through, write-back) from the storage backend (durable, shared).

Fig 2 — Tools talk to the cache; the cache talks to the storage backend
cache/local_file_cache.py The Cache Layer 
import asyncio, os, hashlib, shutil, time
from pathlib import Path
from dataclasses import dataclass, field

@dataclass
class CacheEntry:
    """Tracks state for a single cached file."""
    file_id: str
    local_path: Path
    is_dirty: bool = False      # True if the file was modified locally since download
    last_accessed: float = 0.0  # For LRU eviction if cache grows too large
    size_bytes: int = 0


class LocalFileCache:
    """Read-through, write-back cache that materializes remote files locally.
    
    Lifecycle:
      1. Tool calls get_path(file_id) → cache downloads file on first access
      2. Tool reads/modifies the file at the returned path
      3. Tool calls mark_dirty(file_id) if it modified the file
      4. Agent loop calls flush() to write dirty files back to storage
      5. Agent session end calls cleanup() to delete the cache directory
    
    Concurrency: Uses asyncio.Lock per file_id to prevent duplicate downloads
    when multiple tools request the same file simultaneously.
    """

    def __init__(
        self,
        storage: FileStorageBackend,
        metadata_store: FileMetadataStore,
        cache_dir: str = "/tmp/agent-cache",
        max_cache_bytes: int = 2 * 1024**3,  # 2GB default max cache size
    ):
        self.storage = storage
        self.metadata_store = metadata_store
        self.cache_dir = Path(cache_dir)
        self.max_cache_bytes = max_cache_bytes

        # Internal state
        self._entries: dict[str, CacheEntry] = {}
        self._locks: dict[str, asyncio.Lock] = {}
        self._total_bytes: int = 0

    # ─── The primary interface for tools ───

    async def get_path(self, file_id: str, agent_uuid: str) -> Path:
        """Get a local filesystem path for a file.
        
        If the file is already cached, returns the path immediately.
        If not, downloads it from the storage backend first.
        
        This is the method tools call. It returns a real, readable path.
        """
        # Fast path: already cached
        if file_id in self._entries:
            entry = self._entries[file_id]
            entry.last_accessed = time.monotonic()
            return entry.local_path

        # Slow path: need to download. Lock per file_id prevents
        # duplicate downloads when two tools request the same file.
        lock = self._locks.setdefault(file_id, asyncio.Lock())
        async with lock:
            # Double-check after acquiring lock (another task may have downloaded it)
            if file_id in self._entries:
                return self._entries[file_id].local_path

            return await self._download(file_id, agent_uuid)

    # ─── Internal: download from backend to local cache ───

    async def _download(self, file_id: str, agent_uuid: str) -> Path:
        """Download a file from the storage backend into the cache directory."""
        meta = await self.metadata_store.get(file_id)
        
        # Evict old entries if we're approaching the cache size limit
        if meta.size_bytes + self._total_bytes > self.max_cache_bytes:
            await self._evict(meta.size_bytes)
        
        # Create the local directory structure
        local_dir = self.cache_dir / agent_uuid / file_id
        local_dir.mkdir(parents=True, exist_ok=True)
        local_path = local_dir / meta.filename

        # Stream from backend to local file — uses retrieve_stream
        # so we never hold the full file in memory
        async with aiofiles.open(local_path, "wb") as f:
            async for chunk in self.storage.retrieve_stream(file_id, agent_uuid):
                await f.write(chunk)

        # Register in cache
        entry = CacheEntry(
            file_id=file_id,
            local_path=local_path,
            last_accessed=time.monotonic(),
            size_bytes=meta.size_bytes,
        )
        self._entries[file_id] = entry
        self._total_bytes += meta.size_bytes

        return local_path

    # ─── Write-back: sync modified files to the backend ───

    def mark_dirty(self, file_id: str):
        """Mark a cached file as modified. Called by file_write, apply_patch, etc."""
        if file_id in self._entries:
            self._entries[file_id].is_dirty = True

    async def flush(self, agent_uuid: str):
        """Write all dirty files back to the storage backend.
        
        Called by the agentic loop after tool execution, or at turn boundaries.
        """
        for entry in self._entries.values():
            if not entry.is_dirty:
                continue
            
            # Stream the modified local file back to the storage backend
            async def file_stream():
                async with aiofiles.open(entry.local_path, "rb") as f:
                    while chunk := await f.read(DEFAULT_CHUNK_SIZE):
                        yield chunk

            await self.storage.store(
                file_id=entry.file_id,
                filename=entry.local_path.name,
                content=file_stream(),
                agent_uuid=agent_uuid,
            )
            entry.is_dirty = False

    # ─── Eviction: LRU when cache is full ───

    async def _evict(self, needed_bytes: int):
        """Evict least-recently-accessed non-dirty entries to free space."""
        # Sort by last_accessed, skip dirty entries (they must be flushed first)
        candidates = sorted(
            [(fid, e) for fid, e in self._entries.items() if not e.is_dirty],
            key=lambda x: x[1].last_accessed,
        )
        freed = 0
        for fid, entry in candidates:
            if freed >= needed_bytes:
                break
            if entry.local_path.exists():
                shutil.rmtree(entry.local_path.parent)
            freed += entry.size_bytes
            self._total_bytes -= entry.size_bytes
            del self._entries[fid]

    # ─── Session cleanup: delete everything ───

    async def cleanup(self, agent_uuid: str):
        """Flush dirty files, then delete the entire cache for this session."""
        await self.flush(agent_uuid)
        agent_cache = self.cache_dir / agent_uuid
        if agent_cache.exists():
            shutil.rmtree(agent_cache)
        # Clear entries for this agent
        to_remove = [fid for fid, e in self._entries.items()
                     if str(agent_uuid) in str(e.local_path)]
        for fid in to_remove:
            self._total_bytes -= self._entries[fid].size_bytes
            del self._entries[fid]
🧩 Read-Through + Write-Back: The Two-Phase Cache

The cache has two complementary behaviors. Read-through: when a tool calls get_path() for a file that isn't cached, the cache transparently downloads it from the storage backend, saves it locally, and returns the local path. The tool never knows a download happened. Write-back: when a tool modifies a file, it calls mark_dirty(). The agentic loop calls flush() at turn boundaries to sync modified files back to the storage backend. The tool never explicitly uploads — the cache handles it. This means tools can be written as if every file is local, even when the storage backend is S3 in another region.

Chapter 5 — Tool Integration
Connecting to the Tool System

How Tools Use the Cache

The beauty of the cache abstraction is that tools become extremely simple. They don't think about storage backends, downloads, or uploads. They ask for a path, do their work, and mark the file dirty if they changed it. Here's how the file_read and file_write tools look in practice.

tools/file_tools.py Tool Implementations 
class FileReadTool(ConfigurableToolBase):
    """Read file contents. The tool sees local paths; the cache handles remote files."""
    
    def __init__(self, cache: LocalFileCache, agent_uuid: str, **kwargs):
        super().__init__(**kwargs)
        self.cache = cache
        self.agent_uuid = agent_uuid

    def get_tool(self):
        cache = self.cache
        agent_uuid = self.agent_uuid

        async def file_read(file_id: str, offset: int = 0, limit: int | None = None) -> str:
            """Read the contents of a file.

            Args:
                file_id: The file to read.
                offset: Line offset to start reading from.
                limit: Maximum number of lines to return.
            """
            # This call is instant if the file is already cached,
            # or transparently downloads it from S3/GCS on first access.
            path = await cache.get_path(file_id, agent_uuid)

            # Now we just read from a normal local file. Fast, seekable, simple.
            async with aiofiles.open(path, "r") as f:
                lines = await f.readlines()

            if limit:
                lines = lines[offset:offset + limit]
            else:
                lines = lines[offset:]

            return "".join(lines)

        return self._apply_schema(file_read)


class FileWriteTool(ConfigurableToolBase):

    def __init__(self, cache: LocalFileCache, agent_uuid: str, **kwargs):
        super().__init__(**kwargs)
        self.cache = cache
        self.agent_uuid = agent_uuid

    def get_tool(self):
        cache = self.cache
        agent_uuid = self.agent_uuid

        async def file_write(file_id: str, content: str) -> str:
            """Write content to a file.

            Args:
                file_id: The file to write to.
                content: The full content to write.
            """
            path = await cache.get_path(file_id, agent_uuid)

            async with aiofiles.open(path, "w") as f:
                await f.write(content)
            
            # Tell the cache this file was modified
            # The agentic loop will call cache.flush() to sync to backend
            cache.mark_dirty(file_id)

            return f"Wrote {len(content)} chars to {file_id}"

        return self._apply_schema(file_write)

And in the agentic loop, the cache is flushed at the right points to maintain consistency between local modifications and the durable storage backend.

agent_loop.py Integration Points 
class AgentLoop:
    async def run(self):
        try:
            while not self.should_stop():
                # 1. Call the LLM
                response = await self.call_llm()

                # 2. Process tool calls
                for tool_call in response.tool_calls:
                    envelope = await self.execute_tool(tool_call)
                    await self.handle_tool_result(tool_call, envelope)

                # 3. After tool execution: flush any modified files back to storage
                #    This ensures the backend always has the latest version
                #    before the next LLM turn (in case context refers to file state)
                await self.file_cache.flush(self.agent_uuid)

        finally:
            # 4. Session end: final flush + cleanup local cache directory
            await self.file_cache.cleanup(self.agent_uuid)
Chapter 6 — The Complete Picture
Reflection

How Both Solutions Compose

Let's step back and see the complete system. The two solutions — streaming I/O and the local file cache — address different problems but compose cleanly because they both build on the same abstraction: the FileStorageBackend interface with its streaming methods.

Fig 3 — Streaming ingest, pluggable storage, dual consumers (resolver + cache)

Naming the Patterns

Pattern Where What Problem It Resolves
Streaming Pipeline (Tee) store(stream) with inline checksumming Large file uploads without holding bytes in memory. Source → checksum → destination in a single pass with O(chunk_size) memory.
Read-Through Cache LocalFileCache.get_path() Tools get instant local file access. First call downloads transparently; subsequent calls return cached path. Tools never know about remote storage.
Write-Back Cache mark_dirty() + flush() Tools modify files locally; changes are synced to durable storage at turn boundaries. Tools never explicitly upload.
Double-Checked Locking get_path() lock-per-file_id Prevents duplicate downloads when concurrent tools request the same file. Fast path is lock-free; slow path acquires a per-file lock.
LRU Eviction _evict() in the cache Prevents the cache from consuming unbounded disk space. Evicts least-recently-accessed non-dirty files first.
Dual Interface (bytes | stream) FileContent = bytes | AsyncByteStream Small files keep the convenience of bytes. Large files get the safety of streaming. The backend handles both transparently.
🔑 The Unifying Thread

Both problems — memory pressure and tool latency — are really about the distance between bytes and their consumer. The streaming pipeline minimizes the distance between the HTTP body and the storage backend by eliminating intermediate copies. The local cache minimizes the distance between the storage backend and the tool by materializing a local copy. In both cases, the solution is to put bytes as close as possible to the thing that needs them, without letting them accumulate in places they shouldn't be (process memory for large uploads, remote storage for tool-accessed files). The FileStorageBackend interface needed exactly two additions to support both solutions — FileContent union on store and retrieve_stream() on retrieve — which tells us the original abstraction was well-factored. The new concerns didn't break the boundary; they extended it.

Bytes should be close to the thing that needs them,
and nowhere else. Every copy in memory is a liability.
Every round-trip to remote storage is a tax on your agent's reasoning speed.

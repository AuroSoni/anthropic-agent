Designing Abort & Steer
for an Agentic AI System

Developer Diary — Interrupting a Running Agent Without Breaking the Contract

The agent runs in a loop: stream a response, execute tools, repeat. What happens when the user says "stop" in the middle of all that?

The Problem

You Can't Just Kill an Agent Mid-Turn

A user clicks "Stop." It sounds simple. But the agent is mid-stream: Claude is generating text, tool calls are half-formed, the JSON for a tool input hasn't finished arriving. Or maybe the agent finished streaming and is now executing three tools in parallel — two have returned, one is still running. Or the agent has paused entirely, waiting for a browser-side tool to report back.

In all three cases, the user wants to stop. And in all three cases, simply killing the process leaves behind a corrupted message chain that makes it impossible to resume the conversation.

Developer's inner voice "The problem isn't cancellation. I know how to cancel an asyncio task. The problem is what the conversation looks like after I cancel. Claude's API has strict rules about message ordering. Every tool_use block must have a matching tool_result. Messages must alternate user/assistant. Thinking blocks have cryptographic signatures. If I cancel mid-stream and naively save whatever partial state I have, the next API call will return a 400 and the conversation is dead."

And then there's "steer" — a user wants to not just stop, but redirect. "Stop what you're doing and do this instead." That's abort plus immediately continuing with a new instruction. The message chain has to be valid at the abort point and ready to accept a new user message.

🚨 The Core Challenge

The agent loop was designed to run to completion. Every step assumes the previous step finished cleanly. There is no concept of "mid-step state." Introducing abort means introducing the idea that any step can be interrupted at any point, and the system must produce a valid message chain from whatever partial state exists. This is a fundamental change to the agent's execution model.

Chapter 1 — The Rules We Cannot Break
The Contract

Claude's Message Chain: The Six Commandments

Before designing any abort logic, we need to internalize exactly what Claude's API enforces. The SDK does zero client-side validation — everything is validated server-side and violations come back as HTTP 400. Here are the non-negotiable rules:

Rule 1: Role Alternation

Messages must have role "user" or "assistant". The first message must be "user". The system prompt goes in a separate top-level parameter, not in the message chain.

Rule 2: tool_use → tool_result Pairing

This is the one that makes abort hard. Every tool_use block in an assistant message must have a corresponding tool_result block in the immediately following user message. The tool_use_id must match exactly. If an assistant emits three tool_use blocks, the next user message must contain three tool_result blocks — no exceptions.

Rule 3: Content Block Ordering

In user messages, tool_result blocks must come before text blocks. In assistant messages, the API ordering (thinking → text → tool_use) must be preserved.

Rule 4: Extended Thinking Integrity

Thinking blocks from the current tool-use loop must be preserved. The signature field is cryptographic — modifying a single character invalidates the signature and the API rejects the message. redacted_thinking blocks must be passed through untouched.

Rule 5: Incomplete Responses

If a stream is aborted mid-way, you may have partial content blocks. A tool_use block whose input_json_delta events never completed will have malformed JSON. Such blocks must be removed — they cannot be sent back to the API.

Rule 6: stop_reason Semantics

stop_reason Response State What Must Happen
end_turn Complete Nothing special
tool_use Complete, expecting results Must provide tool_result for every tool_use
max_tokens Truncated, possibly mid-block May have incomplete tool_use
pause_turn Server tool loop limit May have pending server_tool_use
None (aborted) Stream never finished Any block could be incomplete
🔑 The Pairing Rule Is Everything

Almost all abort complexity flows from Rule 2. If the assistant message has tool_use blocks, we must provide matching tool_result blocks, even if the tools never ran. This means abort cannot simply discard the assistant message — if any tool_use blocks completed their streaming, they're in the accumulated message, and we're contractually obligated to answer them. The only escape hatch is to remove incomplete tool_use blocks before appending the assistant message, and synthesize tool_result blocks for any that remain.

Chapter 2 — The Three Abort Scenarios
Scenario Analysis

Where the Agent Can Be When the User Says "Stop"

The agent loop has three distinct phases where it blocks on I/O. Abort can happen during any of them, and each requires different cleanup logic.

Fig 1 — Each scenario requires different cleanup logic to produce a valid chain

Scenario A: Abort During API Streaming

This is the hardest case. The stream from self.provider.generate_stream() (which internally calls anthropic_stream_with_backoff()) is open. The stream processor (_process_stream_events()) has received some events but not all. Content blocks may be in various states of completion.

The key data structure is already in the stream processor: _process_stream_events() in providers/anthropic/retry.py tracks content blocks via block_types and tool_buffers. When we cancel the stream, we know which blocks received a content_block_stop event (they were popped from the buffers) and which are still partial (they remain in the buffers).

providers/anthropic/anthropic_agent.py — _resume_loop() No Abort Path

What must happen on abort: Stop consuming stream events. Determine which content blocks completed (received content_block_stop). Remove any incomplete tool_use blocks from the assistant message. Keep completed text and thinking blocks. If any completed tool_use blocks remain, synthesize tool_result blocks with an abort indicator. Append the sanitized assistant message and synthetic tool results to the chain.

Scenario B: Abort During Tool Execution

The assistant message is already committed to self.agent_config.context_messages. The tool execution is running via self.tool_registry.execute_tools(). Some tasks may have completed; others are in-flight.

            tool_calls, ...  # parallel up to max_parallel_tool_calls

What must happen on abort: Cancel in-flight tool tasks. Collect results from completed tools. Synthesize tool_result blocks for cancelled tools (with an abort message). Combine real and synthetic results. Append them as the user message. The assistant message is already in the chain, so we must provide all matching tool results.

Scenario C: Abort During Relay Wait

The agent has returned with stop_reason="relay". Backend results are saved in self.agent_config.pending_relay.completed_results. Frontend results haven't arrived. The agent is not running — it's in a paused state persisted to the database via the PendingToolRelay dataclass.

What must happen on abort: Synthesize tool_result blocks for the pending frontend/confirmation tools in pending_relay.frontend_calls and pending_relay.confirmation_calls. Combine with existing completed_results. Append to context_messages. Clear agent_config.pending_relay from both memory and persisted state. This is the simplest scenario because the agent isn't running — we just need to fix up the state.

Chapter 3 — The Message Sanitizer
New Module

One Place That Knows How to Fix a Broken Chain

The central insight is that all three abort scenarios converge on the same need: take a potentially invalid message chain and make it valid. Rather than sprinkling fix-up logic across the streaming layer, the tool executor, and the frontend-wait handler, we extract it into a single module.

Developer's inner voice "I need a module that's the single source of truth for 'what makes a message chain valid.' It should be pure logic — no async, no I/O, no framework dependencies. Just functions that take message data in, return cleaned message data out. This makes it trivially testable and keeps the async abort orchestration separate from the data cleanup."
core/message_sanitizer.py New Module 
"""Message chain sanitization for abort/steer scenarios.

Pure functions that take potentially invalid message chain state
and produce valid state that satisfies Claude's API contract.
No async, no I/O — just data transformation.
"""

from __future__ import annotations


def sanitize_partial_assistant_message(
    content_blocks: list[dict],
    completed_block_indices: set[int],
) -> tuple[list[dict], list[str]]:
    """Remove incomplete blocks from a partial assistant message.

    Returns:
        (clean_blocks, orphaned_tool_use_ids)
        clean_blocks: Only blocks that fully completed streaming.
        orphaned_tool_use_ids: IDs of tool_use blocks that completed
            but whose tools never ran (need synthetic results).
    """
    clean_blocks: list[dict] = []
    orphaned_tool_use_ids: list[str] = []

    for i, block in enumerate(content_blocks):
        if i not in completed_block_indices:
            continue  # Drop incomplete blocks entirely

        clean_blocks.append(block)

        if block.get("type") == "tool_use":
            orphaned_tool_use_ids.append(block["id"])

    return clean_blocks, orphaned_tool_use_ids


def synthesize_abort_tool_results(
    tool_use_ids: list[str],
    reason: str = "Tool execution was cancelled by user.",
) -> list[dict]:
    """Create tool_result blocks for tool_use blocks that need answers.

    These are "synthetic" results — the tools never ran (or were
    cancelled), but the API contract demands a tool_result for every
    tool_use. We provide one with is_error=True.
    """
    return [
        {
            "type": "tool_result",
            "tool_use_id": tool_id,
            "content": reason,
            "is_error": True,
        }
        for tool_id in tool_use_ids
    ]


def ensure_chain_validity(
    messages: list[dict],
) -> list[dict]:
    """Walk the entire chain and fix structural violations.

    Fixes:
    - Trailing assistant messages with tool_use but no following
      tool_result → synthesize results
    - tool_result blocks after text blocks in user messages
      → reorder content
    - Orphaned tool_result blocks with no matching tool_use
      → remove
    """
    result = []
    for i, msg in enumerate(messages):
        if msg["role"] == "assistant":
            tool_use_ids = {
                b["id"] for b in msg.get("content", [])
                if isinstance(b, dict) and b.get("type") == "tool_use"
            }
            if tool_use_ids:
                # Check if next message has matching results
                next_msg = messages[i + 1] if i + 1 < len(messages) else None
                if next_msg is None or next_msg["role"] != "user":
                    # Trailing assistant with tool_use and no results
                    result.append(msg)
                    result.append({
                        "role": "user",
                        "content": synthesize_abort_tool_results(
                            list(tool_use_ids)
                        ),
                    })
                    continue
        result.append(msg)
    return result
🧩 Why Pure Functions Matter Here

The sanitizer is stateless. It takes data in, returns data out. It doesn't know about asyncio, queues, streams, or databases. This means: (1) you can unit test every edge case with simple dicts, (2) the abort orchestration layer stays focused on when to sanitize, not how, and (3) the sanitizer can be reused anywhere — in the main agent, in sub-agents, in a standalone recovery tool.

Chapter 4 — Threading the Cancellation Signal
The Mechanism

An asyncio.Event Threaded Through Every Blocking Call

The agent needs a way to signal "stop" that every blocking operation can check cooperatively. The right primitive is asyncio.Event: it's set from outside the loop, and each layer checks it between iterations.

Developer's inner voice "I considered cancelling the asyncio Task directly, but that's too blunt. A cancelled task raises CancelledError at the nearest await point, which could be inside the Anthropic SDK, inside aiohttp, inside a database write — anywhere. I'd have no control over where execution stops. An Event is cooperative: each layer checks it at a safe boundary and exits cleanly."

It emits canonical StreamDelta objects (TextDelta, ToolCallDelta, etc.) to the queue via the StreamFormatter.

🎯 The completed_blocks Set Is the Key

The formatter already tracks every block by index. We just need one additional set[int] to record which blocks received content_block_stop. On abort, this set tells us exactly which blocks are safe to keep and which must be discarded. No heuristics, no guessing — the streaming protocol gives us a definitive answer.

Chapter 5 — Agent Phase Tracking
New Concept

The Agent Needs to Know Where It Is

When abort() is called, it needs to know which cleanup path to take. Is the agent streaming? Executing tools? Waiting for frontend? The agent doesn't currently track this. We introduce a simple enum.

providers/anthropic/anthropic_agent.py — phase tracking New 
from enum import Enum


class AgentPhase(Enum):
    """Where the agent is in its execution lifecycle."""
    IDLE = "idle"                          # Not running
    STREAMING = "streaming"                # Waiting for API response
    EXECUTING_TOOLS = "executing_tools"    # Running backend tools
    AWAITING_RELAY = "awaiting_relay"      # Waiting for frontend/confirmation tools


# In AnthropicAgent._resume_loop():

    else:
        response_message = await self.provider.generate(...)

    # ... append to context_messages, conversation_history ...

        if needs_relay:
            self._phase = AgentPhase.AWAITING_RELAY
            # ... set pending_relay, persist, return AgentResult("relay") ...
        else:
            self._phase = AgentPhase.EXECUTING_TOOLS
            tool_results = await self.tool_registry.execute_tools(...)
            # ...

self._phase = AgentPhase.IDLE

The phase is set at the boundaries between operations. It's read by abort() to determine which cleanup path to take. For the AWAITING_RELAY phase, it's also persisted to the database so abort can work even if the agent process has restarted.

Chapter 6 — The abort() Method
Core Implementation

Wiring It All Together

The abort() method is the public API. It reads the current phase, signals cancellation, waits for the current operation to acknowledge, and then uses the sanitizer to fix the chain.

providers/anthropic/anthropic_agent.py — abort() New Method 
async def abort(self) -> AgentResult:
    """Cancel the current agent turn and produce a valid message chain.

    Safe to call from any context (another task, a signal handler, an
    HTTP endpoint). The agent loop detects the cancellation cooperatively
    and cleans up.

    Returns:
        AgentResult with stop_reason="aborted"
    """
    # 1. Signal cancellation
    self._cancellation_event.set()

    phase = self._phase

    if phase == AgentPhase.IDLE:
        # Nothing running — just return current state
        return self._build_aborted_result()

    if phase == AgentPhase.STREAMING:
        # _process_stream_events() will break out of its loop.
        # _resume_loop() will receive a StreamResult with was_cancelled=True.
        # It will use the sanitizer to clean up the partial message.
        # We wait for the loop to finish its cleanup.
        await self._abort_completion.wait()

    elif phase == AgentPhase.EXECUTING_TOOLS:
        # Tool tasks will check the event and bail out.
        # The main loop collects partial results and synthesizes
        # tool_result blocks for cancelled tools.
        await self._abort_completion.wait()

    elif phase == AgentPhase.AWAITING_RELAY:
        # Agent is paused (not running). We fix up the chain directly.
        # PendingToolRelay holds frontend_calls, confirmation_calls,
        # and completed_results from backend tools.
        from .message_sanitizer import synthesize_abort_tool_results

        relay = self.agent_config.pending_relay
        pending_ids = [
            tc.tool_id for tc in relay.frontend_calls + relay.confirmation_calls
        ]
        synthetic_results = synthesize_abort_tool_results(pending_ids)
        all_results = relay.completed_results + synthetic_results

        tool_result_message = self._build_tool_result_message(all_results)
        self.agent_config.context_messages.append(tool_result_message)

        # Clear pending relay state
        self.agent_config.pending_relay = None
        self._phase = AgentPhase.IDLE

        await self._persist_state()

    return self._build_aborted_result()

Notice the two different patterns: for STREAMING and EXECUTING_TOOLS, abort sets the event and waits for the main loop to handle cleanup (via _abort_completion). For AWAITING_RELAY, abort does the cleanup directly because the main loop isn't running.

Developer's inner voice "The abort_completion event is crucial. Without it, the caller of abort() would return before the message chain is actually fixed. The main loop sets abort_completion after it has sanitized the chain and saved state. This gives abort() a clean synchronization point: when it returns, the chain is guaranteed valid."
Chapter 7 — Cancelling In-Flight Tools
Parallel Execution

From asyncio.gather to asyncio.wait

The current tool_registry.execute_tools() uses parallel execution (up to max_parallel_tool_calls). This waits for all tasks and provides no way to bail out early. We need to switch to asyncio.wait() which gives us fine-grained control over which tasks completed and which to cancel.

tools/registry.py — execute_tools with cancellation Modified 
async def execute_tools(
    self,
    tool_calls: list[ToolCallInfo],
    cancellation_event: Optional[asyncio.Event] = None,
) -> list[ToolResultContent]:
    """Execute tools with cancellation support."""
    tasks: dict[asyncio.Task, str] = {}  # task → tool_use_id

    for tc in tool_calls:
        task = asyncio.create_task(
            self._execute_single_tool(tc)
        )
        tasks[task] = tc.id

    results: dict[str, dict] = {}  # tool_use_id → result
    pending = set(tasks.keys())

    while pending:
        # Wait for either a task to complete or cancellation
        cancel_task = asyncio.create_task(
            cancellation_event.wait()
        ) if cancellation_event else None
        done, pending = await asyncio.wait(
            pending | {cancel_task},
            return_when=asyncio.FIRST_COMPLETED,
        )

        for task in done:
            if task == cancel_task:
                # Cancellation signalled — cancel remaining tools
                for p in pending:
                    p.cancel()
                # Synthesize results for cancelled tools
                for p in pending:
                    tool_id = tasks.get(p)
                    if tool_id and tool_id not in results:
                        results[tool_id] = {
                            "type": "tool_result",
                            "tool_use_id": tool_id,
                            "content": "Cancelled by user.",
                            "is_error": True,
                        }
                pending = set()
                break
            else:
                # Normal tool completion
                tool_id = tasks[task]
                results[tool_id] = task.result()
                pending.discard(cancel_task)

        cancel_task.cancel()  # Clean up if not triggered

    return [results[tc.id] for tc in tool_calls]
⚔️ Race Condition: Tool Completes During Abort

A tool might complete in the same event loop tick that cancellation is signalled. The asyncio.wait() pattern handles this correctly: tasks in the done set already have their results, regardless of the cancellation event. We process completed tasks first, then cancel the rest. No results are lost.

Chapter 8 — Steer: Abort + Redirect
Composition

Steer Is Abort Followed by a New User Message

With abort producing a valid message chain, steer is almost trivial. Abort, then append a new user message with the steering instruction, then re-enter the agent loop.

providers/anthropic/anthropic_agent.py — steer() New Method 
async def steer(
    self,
    new_instruction: str,
    queue: Optional[asyncio.Queue] = None,
    stream_formatter: Optional[str | StreamFormatter] = "json",
) -> AgentResult:
    """Abort the current turn and redirect with a new instruction.

    This is the "steer" operation: stop what you're doing, here's
    what I want instead.

    Args:
        new_instruction: The user's new direction.
        queue: SSE queue for streaming output.
        stream_formatter: Output formatter.

    Returns:
        AgentResult from the redirected run.
    """
    # Step 1: Abort cleanly (produces valid chain)
    await self.abort()

    # Step 2: Build a Message with the new instruction
    from ..core.messages import Message
    from ..core.types import TextContent, Role

    steer_message = Message(
        role=Role.USER,
        content=[TextContent(text=new_instruction)],
    )
    self.agent_config.context_messages.append(steer_message)
    self.agent_config.conversation_history.append(steer_message)

    # Step 3: Reset cancellation for the new run
    self._cancellation_event = asyncio.Event()
    self._abort_completion = asyncio.Event()

    # Step 4: Resume the agent loop
    return await self._resume_loop(
        queue=queue, stream_formatter=stream_formatter,
    )
Developer's inner voice "This is why the sanitizer matters so much. The entire steer method is just four steps because abort() guarantees the chain is valid when it returns. If I had sprinkled cleanup logic throughout the codebase, steer would need to know about every abort scenario. Instead, it just calls abort() and trusts the result."
Chapter 9 — The Modified Agent Loop
Integration

How the Main Loop Handles Cancellation

The main loop is modified to check the StreamResult for cancellation after each API call, and to handle tool execution cancellation. Here's the full modified loop with the new abort paths highlighted:

providers/anthropic/anthropic_agent.py — modified _resume_loop() Modified 
async def _resume_loop(self, queue=None, stream_formatter=None):
    # ... existing initialization ...

    self._cancellation_event = self._cancellation_event or asyncio.Event()
    self._abort_completion = asyncio.Event()

    while self.agent_config.current_step < self.max_steps:

        # Compaction
        did_compact, compacted_messages = await self.compactor.apply_compaction(...)

        self._phase = AgentPhase.STREAMING
        if queue:
            stream_result = await self.provider.generate_stream(
                ...,
                cancellation_event=self._cancellation_event,
            )
        else:
            stream_result = await self.provider.generate(...)

        # ---- NEW: Handle stream cancellation ----
        if stream_result.was_cancelled:
            from .message_sanitizer import (
                sanitize_partial_assistant_message,
                synthesize_abort_tool_results,
            )
            raw_blocks = stream_result.message.content
            clean_blocks, orphaned_ids = sanitize_partial_assistant_message(
                raw_blocks,
                stream_result.completed_blocks,
            )

            if clean_blocks:
                sanitized_msg = Message(
                    role=Role.ASSISTANT, content=clean_blocks,
                )
                self.agent_config.context_messages.append(sanitized_msg)
                if orphaned_ids:
                    abort_results = synthesize_abort_tool_results(orphaned_ids)
                    result_msg = self._build_tool_result_message(abort_results)
                    self.agent_config.context_messages.append(result_msg)

            self._phase = AgentPhase.IDLE
            self._abort_completion.set()  # Signal abort() caller
            return self._build_aborted_result()

        # ---- Normal path (unchanged) ----
        response_message = stream_result.message
        self.agent_config.current_step += 1
        self._accumulate_usage(response_message.usage)
        self.agent_config.context_messages.append(response_message)
        self.agent_config.conversation_history.append(response_message)

        stop_reason = response_message.stop_reason

        if stop_reason == "tool_use":
            tool_calls = self._extract_tool_calls(response_message)

            if needs_relay:
                self._phase = AgentPhase.AWAITING_RELAY
                # ... set pending_relay, persist, return ...
            else:
                self._phase = AgentPhase.EXECUTING_TOOLS
                tool_results = await self.tool_registry.execute_tools(...)

                # tool_results now contains real + synthetic abort results
                tool_result_message = self._build_tool_result_message(tool_results)
                self.agent_config.context_messages.append(tool_result_message)
                self.agent_config.conversation_history.append(tool_result_message)

                # Check if we were cancelled during tool execution
                if self._cancellation_event.is_set():
                    self._phase = AgentPhase.IDLE
                    self._abort_completion.set()
                    return self._build_aborted_result()

                continue

        # ... rest of loop (end_turn, pause_turn handling, etc.) ...

Chapter 12 — The API-Layer Problem
Critical Flaw

asyncio.Event Doesn't Cross HTTP Request Boundaries

Everything described so far assumes you have a reference to the live agent object. But in the NOVA backend — and any production deployment — agents are reconstructed from database state per-request. The SSE streaming handler creates the agent, runs it in a background task, and holds the only reference. When a separate /abort HTTP request arrives, it creates a new agent instance. That new instance has no connection to the running one — its asyncio.Event is a different object in a different request context.

Developer's inner voice "The asyncio.Event pattern works beautifully within a single process where both abort() and run() share the same object. But in NOVA, the SSE stream handler (router.py:354-376) creates agent_task = asyncio.create_task(run_agent_and_signal()). That task reference is a local variable inside the generator. A separate HTTP request for /abort has absolutely no way to reach it. Setting an event on a freshly-constructed agent does nothing."

🚨 The Fundamental Problem

The asyncio.Event lives on a Python object. Python objects live in a single process. HTTP requests are isolated contexts. There is no cross-request mechanism to reach a running agent's in-memory state. We need an intermediary that both the streaming request and the abort request can access.

Solution: A Process-Level Task Registry

The fix is a shared, process-level registry that maps agent UUIDs to their running tasks and cancellation events. Both the SSE handler and the abort endpoint access this registry.

excel_agent/agent_registry.py New Module 
"""Process-level registry of running agent tasks.

Maps agent_uuid to the live task + cancellation event so that
a separate HTTP request (abort, steer) can reach a running agent.

Thread-safe within asyncio (single event loop). For multi-worker
deployments, this must be backed by Redis or similar.
"""
from __future__ import annotations

import asyncio
from dataclasses import dataclass, field
from typing import Optional


@dataclass
class RunningAgent:
    """Handle to a running agent task."""
    agent_uuid: str
    task: asyncio.Task
    cancellation_event: asyncio.Event
    queue: asyncio.Queue
    created_at: float = field(default_factory=lambda: asyncio.get_event_loop().time())


# Module-level singleton — shared across all requests in the same process
_registry: dict[str, RunningAgent] = {}


def register(
    agent_uuid: str,
    task: asyncio.Task,
    cancellation_event: asyncio.Event,
    queue: asyncio.Queue,
) -> None:
    """Register a running agent. Called by the SSE stream handler."""
    _registry[agent_uuid] = RunningAgent(
        agent_uuid=agent_uuid,
        task=task,
        cancellation_event=cancellation_event,
        queue=queue,
    )


def unregister(agent_uuid: str) -> None:
    """Remove a finished agent. Called in finally block of stream handler."""
    _registry.pop(agent_uuid, None)


def get(agent_uuid: str) -> Optional[RunningAgent]:
    """Look up a running agent by UUID."""
    return _registry.get(agent_uuid)


def is_running(agent_uuid: str) -> bool:
    """Check if an agent is currently active."""
    entry = _registry.get(agent_uuid)
    return entry is not None and not entry.task.done()

The SSE stream handler registers the agent when it starts and unregisters when it ends:

excel_agent/router.py — with registry Modified 
from .agent_registry import register, unregister

async def stream_agent_response(...):
    agent = _create_agent(config, agent_uuid, ...)
    queue: asyncio.Queue = asyncio.Queue()
    cancellation_event = asyncio.Event()

    async def run_agent_and_signal():
        try:
            result = await agent.run_stream(
                prompt, queue,
                cancellation_event=cancellation_event,
            )
            return result
        finally:
            await queue.put(None)

    agent_task = asyncio.create_task(run_agent_and_signal())

    # Register so /abort can reach this task
    register(agent.agent_uuid, agent_task, cancellation_event, queue)

    try:
        while True:
            chunk = await queue.get()
            if chunk is None: break
            yield f"data: {chunk}\n\n"
    except asyncio.CancelledError:
        agent_task.cancel()
        raise
    finally:
        unregister(agent.agent_uuid)  # Always clean up

    await agent_task
    result = agent_task.result()
    await _deduct_credits_from_result(result, ...)

excel_agent/router.py — abort endpoint New Endpoint 
from .agent_registry import get as get_running_agent

@router.post("/{agent_uuid}/abort")
async def abort_agent(agent_uuid: str):
    """Abort a running agent turn."""
    running = get_running_agent(agent_uuid)

    if running is None:
        # Not running in this process — check DB for AWAITING_RELAY
        return await _abort_from_db_state(agent_uuid)

    # Signal the running agent's cancellation event
    running.cancellation_event.set()

    # Wait for the task to finish its cleanup (with timeout)
    try:
        await asyncio.wait_for(running.task, timeout=10.0)
    except asyncio.TimeoutError:
        running.task.cancel()  # Force-cancel if cleanup is stuck

    return {"status": "aborted", "agent_uuid": agent_uuid}

Where This Breaks: Multi-Worker Deployments

The process-level dict works for a single-worker Uvicorn deployment. But if you scale to multiple workers (multiple processes), the abort request might land on a different worker than the one running the agent. Two options:

Approach How It Works Tradeoff
Sticky sessions Route all requests for the same agent_uuid to the same worker (via load balancer hash) Simple, but limits scaling flexibility
Redis pub/sub Abort writes a cancel:{agent_uuid} message to Redis. Each worker subscribes and checks against its local registry. Works across workers, adds Redis dependency
Database polling Abort sets agent_config.abort_requested = true in PostgreSQL. The agent loop polls the DB every N seconds. Slowest (polling delay), but no new infrastructure. Already have the DB.
Developer's inner voice "For NOVA today, we run a single Uvicorn worker per container, so the process-level registry works. But I should design the agent_base layer to accept the cancellation_event as a parameter from the outside — not create it internally. That way the framework doesn't care where the event comes from. The API layer provides it, whether it's a local Event or a wrapper around Redis pub/sub."
🎯 Design Principle: Cancellation Event as Injected Dependency

The cancellation_event should be a parameter to run(), not an internal member of the agent. The agent doesn't create it — the caller does. This means: (1) in a simple script, you create a local asyncio.Event(); (2) in a single-worker API, the registry holds events; (3) in a multi-worker deployment, the event is a wrapper that checks Redis. The agent code is identical in all three cases. The framework provides the mechanism; the deployment provides the signal.

✅ The Agent Base Stays Clean

The agent_base framework doesn't need to know about registries, Redis, or databases. It accepts a cancellation_event parameter and checks it at safe boundaries. How that event gets set — locally, via registry, via pub/sub — is the deployment's concern. The framework provides one additional helper: an abort_from_persisted_state() class method that reconstructs the agent from storage (via the three-adapter pattern: AgentConfigAdapter, ConversationAdapter, AgentRunAdapter), sanitizes the message chain by clearing pending_relay and synthesizing tool results, and saves it. This covers the AWAITING_RELAY case where no live agent exists.

Chapter 13 — Edge Cases & Gotchas
Watch Out

The Traps That Will Bite You

Thinking Block Signatures

Extended thinking blocks have a signature field that is cryptographically signed by Anthropic. Modifying a single character of the thinking text invalidates the signature and the API will reject the message with a 400. On abort, you must keep thinking blocks exactly as received or discard them entirely. Never truncate, edit, or reconstruct a thinking block.

🚨 Critical Rule

If an abort happens mid-thinking-block (the thinking text is only partially received), you must discard the entire thinking block. A partial thinking block with no valid signature will be rejected by the API. The completed_blocks set tells you whether the block received its content_block_stop event — only keep it if it did.

Race Between Abort and Tool Completion

A tool completes in the same event loop tick that cancellation is signalled. The asyncio.wait() pattern handles this: the completed task appears in the done set regardless of cancellation. Process completed tasks first; only synthesize abort results for tasks still in pending.

Compaction After Abort

After an abort, the message chain may contain synthetic tool_result blocks with is_error: True. The compactor must not strip these, or the chain becomes invalid. The current compactor only truncates tool result content (not entire blocks), so this should be safe — but needs a test.

Multiple Aborts in Sequence

User aborts, then steers, then aborts again. Each abort must independently produce a valid chain. The sanitizer must be idempotent: calling ensure_chain_validity() on an already-valid chain should return it unchanged.

Queue Draining on Abort

After abort, the SSE queue may contain stale events from the cancelled turn (partial text deltas, incomplete tool results). Options: (a) drain the queue before emitting new events, or (b) emit an "aborted" envelope and let the client ignore subsequent events from the old turn. Option (b) is simpler and more resilient.

Frontend Tool Abort and DB Consistency

When aborting during AWAITING_RELAY, we must clear agent_config.pending_relay from both the in-memory state and the persisted AgentConfig. Otherwise, a stale resume_with_relay_results() call could arrive from the browser and operate on invalid state.

Abort During Compaction

Compaction modifies self.agent_config.context_messages. If abort is signalled mid-compaction, the list could be inconsistent. Solution: compaction should be atomic (build a new list, then swap) or abort should defer until compaction finishes. Compaction is fast (pure data manipulation), so deferring is acceptable.

Epilogue
Patterns At Work

The Design Patterns Behind the Architecture

Pattern Where Why
Cooperative Cancellation asyncio.Event threaded through layers Each layer checks at safe boundaries and exits cleanly, rather than being killed mid-operation. This guarantees the system reaches a known state.
Data Repair / Sanitizer message_sanitizer.py A single module that knows the API contract and can fix any violation. Keeps repair logic out of the async orchestration code.
Phase State Machine AgentPhase enum The agent is always in exactly one phase. Abort reads the phase to determine cleanup strategy. No ambiguity about what's running.
Completion Signal _abort_completion event The abort() caller waits for the main loop to finish cleanup. Guarantees the chain is valid when abort() returns.
Composition steer() = abort() + message + resume() Complex behavior built from simpler, independently tested pieces.
🔑 The Unifying Insight

The entire abort/steer architecture flows from one principle: the message chain must be valid after every operation, including interrupted ones. This is the same invariant that databases maintain with transactions — either the operation completes fully, or it's as if it never happened. The sanitizer is the "rollback" mechanism: it takes whatever partial state exists and produces a consistent snapshot. Every other piece — the cancellation event, the phase tracking, the completion signal — exists to get the system to a point where the sanitizer can do its job.

The hardest part of cancellation isn't stopping.
It's leaving behind a world that still makes sense.
Abort is a transaction boundary for agent state.
